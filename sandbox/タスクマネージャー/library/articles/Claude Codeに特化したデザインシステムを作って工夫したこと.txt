Claude Codeが生成するUIの品質が安定しない課題
最初の頃は、何を作っても「いかにもAIが作りました」感のあるデザインで生成されてしまうストレスがあって、例えば、コンポーネントに強めのシャドウをかけたり、カードUIの上部に強めのカラーバーを付けたりしてイマイチだった。
「いい感じに」とか「シンプルに」と指示してもダメで、AIには「いい感じ」の基準がなく、 基準がないから、学習データの中で「よくあるデザイン」を確率的に再現してしまう。
その結果、同じプロダクトなのに、画面ごとに微妙にボタンの高さが違ったり色のトーンがズレる、スペーシングがバラつくし、AIはやたらと絵文字を多用するのが課題だったので、Claude Code開発に適したデザインシステムを作ってみることに。
AI最適化したデザインシステムとは
サイト: https://melta.tsubotax.com
GitHub: https://github.com/tsubotax/melta-ui
工夫した事は「2つの読み手」のためのアーキテクチャで、人間と AI の両方が読めるように設計したこと。ファイルは大きく「人間向け」「AI 向け」「両方」の3層に分かれている。
デザインコンセプトはコンテンツファーストで細かい所に手が行き届くような、Notion、Stripe、Vercelのような、ミニマルで高速なSaaSに共通する「静かだけど確かに伝わる」感覚を目指した。
フォルダ構成
melta-ui/
├── CLAUDE.md                 ← AI のエントリーポイント
├── foundations/              ← 設計原則・トークン定義（13 md）
├── components/               ← コンポーネント仕様（28 md）
├── patterns/                 ← レイアウト・フォーム等のパターン（5 md）
├── tokens/tokens.json        ← 106 トークンの SSOT
├── metadata/components.json  ← 28 コンポーネントの機械可読メタデータ
├── src/                      ← MCP サーバー（TypeScript）
├── docs/                     ← ショーケース（index.html）・アイコン一覧
├── examples/                 ← 12 種のサンプルサイト
├── scripts/                  ← テーマ CSS・JS
├── assets/icons/             ← Charcoal + Lucide（222 アイコン）
├── tools/                    ← トークン→CSS/Tailwind 生成スクリプト
├── skills/                   ← Claude Code スキル（design-review 等）
└── .cursor/rules/            ← Cursor 向け DS ルール
CLAUDE.mdに書いていること
実際のAIの挙動としては、CLAUDE.mdが会話開始時に自動でコンテキストに入り、そこに書かれたクイックリファレンスだけで大半のUI生成が完結する。 CLAUDE.mdが主役で、tokens.jsonはその背後にあるSSOT（唯一の真実の源）として値の一貫性を保証している 構造になっている。
デザインシステムでイメージ通りにUIが作れるようになった
まだ育てている途中だけど、DS導入前と後でAI生成UIのレビュー体験が明確に変わった感覚。
Before:
Claude Codeに画面を作らせる → レビューで「shadow-lgはやめて」「この色はトークン使って」と指摘 → 修正 → また別の箇所で同じミス
毎回プロンプトに「グラデーション禁止」「装飾控えめに」と書く → 書き忘れると元に戻る
After:
CLAUDE.mdを置くだけで、Claude Codeが起動時に自動でDSを読み込む
禁止パターンを構造化データで渡しているので、text-black や shadow-lg がほぼ出なくなった
色・スペーシング・コンポーネントの仕様がJSONで定義されているので、画面間のバラつきが消えた
DSが品質の下限を担保してくれるので、人間はより上流の議論に時間を使えるようになった。 色やクラスの指摘ではなく、「この導線は正しいか」「この情報の優先順位は合ってるか」に集中できる。
そのための設計で特に意識したこと
CLAUDE.md 1ファイルで基本的なUIが生成できるようにした
今回のデザインシステムでは CLAUDE.md 1ファイルを読むだけで、基本的なUIが生成できる ようにした。ボタンのクラス、カードの構成、テーブルの書き方など、よく使うものはクイックリファレンスとしてCLAUDE.mdに全部入れた。
CLAUDE.mdの中身はこんな感じで、デザイントークンやコンポーネントのクイックリファレンスが1ファイルにまとまっている。
htmlbars
## クイックリファレンス — ボタン
| Variant | Class |
|---------|-------|
| Primary | `bg-primary-500 text-white hover:bg-primary-600` |
| Secondary | `bg-white border border-slate-300 text-slate-700` |
| Size SM | `h-8 px-3 text-sm` |
| Size MD | `h-10 px-4 text-base` |
さらに「タスクベース読み込みガイド」を書いて、「フォーム画面を作る」なら patterns/form.md → textfield / select / checkbox / button を読めと明示化。タスクに応じて必要なファイルだけ読ませる設計にした。
CLAUDE.mdの書き方一つでAIの出力精度は劇的に変わる。
AIがドキュメントを「解釈」するのではなく、JSONを「参照」することで曖昧さを減らした
「primary-500は鮮やかな青です」と記載しても、AIは #3B82F6 を使うかもしれないし #2563EB を使うかもしれない。自然言語の曖昧さが、色のブレとして出てくる。
このDSでは106個のデザイントークンをJSONで定義した。色、スペーシング、書体、影、角丸、アニメーションのすべてが tokens.json に集約されている。
{
  "color": {
    "primary-500": "#2b70ef",
    "primary-600": "#1d5fd4"
  }
}
ただし苦い教訓もある。当初primary-500を #2250df にしていたが、人間の目で見て明るさが均一に感じられるカラースケール（OKLCH）で再生成した結果 #2b70ef に変わった。古い値がCSS変数、サンプルコード、JSなどあらゆる場所に散らばっていて、全部洗い出すのにClaudeで何往復も指示を出して再チェックした。
tokens.jsonを参照するフローが最初から徹底されていれば1箇所変えるだけで済んだはず。SSOTの重要性を身をもって学んだ。
禁止パターンを明示的に定義した
調整していく中で最も効果が大きかったのは禁止パターン。AIは「やるな」が苦手で、人間のデザイナーが体に染み込ませている「やりすぎ」の感覚を持っていない。なので76項目の禁止パターンを明示的に定義した。
text-black 禁止 → text-slate-900 を使う
shadow-lg 禁止 → shadow-sm で十分
border-l-4 のカラーバー禁止 → border rounded-lg で全周ボーダー
bg-indigo-* や bg-blue-* のハードコード禁止 → primary-* を使う
特に力を入れたのが「AI生成パターンの排除」セクション。
Claude CodeでUIを作っていて「なんかAIっぽいな」と感じるたびに、その場で禁止ルールに追加していった。開発中に /ban-pattern というスキルを作って、ワンコマンドで禁止パターンを登録できるようにした。
AIはなぜかカラーバー付きのカードを好むようで、気を抜くとカラーバーが入るので禁止パターンに入れている。こういう、禁止ルールを追加してリストを育てることで、使えば使うほどDS準拠の出力が安定していく仕組みにした。
MCPサーバーでAIがDSをAPIとして叩けるようにした
Claude CodeやCursorから get_component("table") と呼べば、テーブルの仕様・HTMLサンプル・禁止パターンが返ってくる。check_rule("text-black shadow-lg") と呼べば、2つの禁止パターン違反が検出される。
AIがドキュメントを「読む」のではなく、APIとして「叩ける」。将来的にはCI/CDに組み込んで、PRレビュー時にDS準拠を自動チェックする使い方も見えている。
MCPサーバーはクラウドサービスではなく、リポジトリに同梱されたローカルツール。git clone して npm run build すれば、AIはすぐにこのデザインシステムをAPIとして叩ける。
AIの「良い感じにしようとする」癖との戦い
Claude Codeに「このUI、なんかAIっぽくない？」と聞いても「いい感じだと思います」と返ってくる。この「AIっぽさ」を判断できるのは、まだ人間だけだ。
なのでレビューの仕組みも作った。
リポジトリに /design-review というスキルを同梱していて、クローン後すぐに使えるようにした。
/design-review examples/ec-home.html
→ DS準拠チェック → 違反検出・重大度分類 → 修正提案を出力
生成したHTMLをこのスキルに通すと、禁止パターン違反やトークン未使用の箇所を自動で洗い出してくれる。人間がレビューする前の一次チェックとして使っている。
AIは「実装者」としては極めて優秀だが、「判断者」としてはまだ人間が必要。 人間の目利きとAIの実行速度の組み合わせが、今は一番早く実装できそう。
Showcaseとサンプルサイト
Showcaseで全28コンポーネントのプレビューとコード例、禁止パターンのDo/Don't比較が確認できる。
リポジトリの公開
melta UIはMITライセンスで公開した。Claude Code向けに作ったが、考え方自体はCursorの .cursor/rules/ など他ツールでも応用できるし、コンポーネントライブラリではなく設計仕様書なのでshadcn/uiやChakra UIなど既存ライブラリとも共存できる。
一番手軽な方法（5分）
melta UIリポジトリからCLAUDE.mdだけコピーして、自分のプロジェクトルートに置く。CLAUDE.md単体でもボタン・カード・テーブルなど基本的なコンポーネントの仕様が入っているので、これだけでClaude Codeの出力が変わる。
もう少しちゃんと使うなら 
リポジトリを git clone して、foundations / componentsのmdとtokens.json、prohibited.mdを自分のプロジェクトにコピー。JSONトークンを参照するようになるので、色やスペーシングの精度がさらに上がる。
フルで使うなら
MCPサーバーを起動して、AIエージェントからAPI経由でアクセス。Cursorにも .cursor/rules/ を同梱しているのでそのまま使える。
フレームワーク非依存（React / Vue / Svelte何でもOK）で、Tailwind CSSが入っていれば動く。npmインストール不要、git clone してプロジェクトに置くだけ。
まだv1.1.0で育てている途中だけど、AIでUIを作るワークフローが変わるので、使ってみて何か発見があったら教えてほしい。Issue / PRも歓迎です。
