Claude Codeに「バズってるX記事を集めて」と言ったら5分後に545件の記事がレポートになって出てきた。

コスト約25円。

いいね数、RT数、ブックマーク数、閲覧数、フォロワー数、記事の全文テキスト。全部入り。しかも日本語の記事だけ自動で分類されてる。

「いや、さすがにそれは盛ってるでしょ」って思うよね。僕も最初は無理だと思ってた。Xで検索しても記事ってうまく見つからないし、Web検索しても個別のバズ記事は引っかからない。正直、無理だと思ってた。

でもClaude Codeと試行錯誤をする中で格安で自動リサーチできる抜け道を見つけた。

今では、リサーチツールをSkillに登録してるから日本語で「X記事をリサーチして」と打つだけで全部動く。

X記事リサーチしたいけどうまくいかないって悩んでる人多いと思うから今回は、バズってるX記事のリサーチをClaude Codeで全自動化したやり方を全部書いていく。

# なんでX記事を自動リサーチしたかったのか

X記事、今めちゃくちゃ伸びてる。

Xのアルゴリズムに優遇されてて、フォロワー少なくてもおすすめに載る。2026年1月にPremiumユーザーに開放されてから、記事を書く人が一気に増えた。

僕はAIを使った情報発信やマネタイズについて発信してるから、「AI系でどんな記事がバズってるのか」を知りたかった。タイトルの付け方、見出しの数、文字数、冒頭の書き出し。伸びてる記事の共通点をデータで掴みたかった。

ただ、「なんとなく伸びてる記事」じゃ意味がない。いいね1000以上のガチで伸びてる記事だけ、しかも日本語の記事だけを抜き出したい。

で、これを手動でやるのは無理。X記事にはGoogleみたいな検索機能がないし、他人のプロフィールを1人ずつ開いて記事タブを見に行くしかない。

だからClaude Codeで全自動化しようとした。ここから試行錯誤が始まる。

# Claude Codeでリサーチ開始 → 壁にぶつかる

まずClaude CodeにX記事をプログラムで取得する方法を調べさせたんだけど、思いのほか障壁が多かった。

## 壁①: X API v2 → 記事を絞る検索コマンドが存在しない

Claude Codeに「X APIで記事だけを取得する方法を調べて」と指示した。

X API v2の公式ドキュメント、開発者コミュニティ、検索演算子の一覧を全部調べさせた。

結果、こうだった。

- filter:articles → 存在しない。
- has:articles → 存在しない。
- is:article → 存在しない。

X API v2の検索演算子にはhas:links、has:media、has:images、is:retweet、is:replyみたいなフィルタはある。でもArticles用の演算子は一切定義されてない。

しかもX APIは従量課金で、ポストの読み取りだけで1件あたり$0.005（約0.75円）かかる。記事かどうかもわからないポストを片っ端から取得して、その中から記事を探す。10,000件取得したら$50（約7,500円）。コスパが悪すぎる。

## 壁②: xAIのGrok API → これも記事フィルタなし

次にClaude Codeに xAIのGrok APIの x_search を調べさせた。

キーワード検索、ユーザー検索（allowed_x_handlesで最大10アカウント指定）、スレッド取得はできる。日付範囲の指定もできる。

でもArticlesを独立したコンテンツタイプとしてフィルタする機能は、やっぱりない。

## 壁③: Google検索 → X記事はインデックスされにくい

site:x.com inurl:article とか "x.com" "/articles/" で検索しても、Xの記事機能のヘルプページが出てくるだけ。個別のバズ記事は引っかからない。

ここまでの調査結果をClaude Codeにまとめさせた結果がこれ。

「X ArticlesをAPIベースで取得する公式な方法は、2026年3月時点で存在しない。」

# 転機: SocialData APIの発見

でもClaude Codeに「X APIの代替になるサードパーティのAPIを探して」と指示したら、SocialData APIというサービスが出てきた。

SocialData APIはX（Twitter）のデータを取得できるサードパーティのAPI。X API v2とは別物で、独自のエンドポイントを持ってる。

何がでかいかっていうと、まず価格。

1件あたり$0.0002（約0.03円）。

X APIの$0.005（約0.75円）と比べると25分の1。固定費もなくて、使った分だけの完全従量課金。

で、もう1つでかいのが、記事の詳細を取得できる専用APIを持ってること。/twitter/article/{tweetId} というエンドポイントで、タイトル、本文（フォーマット付き）、いいね、RT、ブックマーク数、閲覧数まで全部きれいに返ってくる。

# url:x.com/i/article ： 記事だけを抜き出すトリック

ここがこの記事の一番のポイント。

url: 演算子自体はX API v2にもSocialData APIにもある。ポスト内のURLパターンで検索できる演算子。

で、APIコストが安いから色んなパターンでの取得と抽出をClaude Codeに試してもらっていく中で、X記事のポストには内部的に x.com/i/article というURLが埋め込まれてることを発見した。

つまり、こういう検索クエリを投げると1,000いいね以上の記事だけが返ってくる。

- url:x.com/i/article → 記事だけに絞る（歩留まり100%、ノイズ0）
- min_faves:1000 → いいね1000以上のバズ記事のみ
- -filter:replies → リプライを除外

# 完成したツールの全体フロー

① ユーザーの指示（いいね数、期間など）を受け取る
② SocialData API で検索（url:x.com/i/article min_faves:〇〇 -filter:replies）
③ プロフィールの日本語判定で ja / other に一次分類（コスト0）
④ 記事詳細API（/twitter/article/{tweetId}）で1件ずつ本文取得
⑤ 重複除去（タイトル+著者でユニーク化、いいね最大のものを残す）
⑥ エンゲージメント数降順でソート
⑦ レポート出力

## 実際の実行結果

- 検索ヒット数: 545件
- 日本語記事: 269件 / その他言語: 276件
- APIコール数: 573回
- 合計コスト: 約25円（$0.17）

## 検索クエリの応用パターン

url:x.com/i/article min_faves:500 -filter:replies   → より広く拾う
url:x.com/i/article min_faves:5000 -filter:replies  → トップ記事だけ
url:x.com/i/article min_faves:1000 min_retweets:100 -filter:replies
url:x.com/i/article min_faves:1000 -filter:replies since:2026-03-01 until:2026-03-24
