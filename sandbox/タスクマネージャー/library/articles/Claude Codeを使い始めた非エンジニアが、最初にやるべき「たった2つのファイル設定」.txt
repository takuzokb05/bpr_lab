settings.local.json — AIに「鍵」をかける
まず1つ目。settings.local.json です。
このファイルは、AIの行動を「仕組みとして」コントロールします。日常に例えるなら、ドアに取り付ける「鍵」のようなものです。AIがルールを忘れても関係ない。鍵がかかっていれば、物理的に開かないのと同じです。
私が実際に使っているファイルの中身がこちらです。
{
  "permissions": {
    "allow": [
      "Bash(mkdir:*)",
      "Bash(move:*)",
      "Bash(cp:*)",
      "Bash(node --version:*)",
      "Bash(npm --version)",
      "Bash(npm init -y)",
      "Bash(npm start)",
      "WebFetch(domain:www.pexels.com)",
      "WebFetch(domain:pixabay.com)",
      "WebFetch(domain:code.claude.com)",
      "WebFetch(domain:raw.githubusercontent.com)",
      "WebFetch(domain:agentnativedev.medium.com)"
    ],
    "deny": [
      "Bash(rm:*)",
      "Bash(rm -rf:*)",
      "Bash(del:*)",
      "Bash(rmdir:*)",
      "Bash(Remove-Item:*)"
    ]
  }
}
パッと見ると英語だらけで怖く見えるかもしれません。 でも構造はシンプルです。大きく2つのリストがあるだけ。
✅️allow（許可リスト）— 確認なしでやっていいこと
ここに書いた操作は、AIが「やっていいですか？」と聞かずに、自分の判断で実行できます。安全だと確認できたものだけを入れています。
■ファイル操作
mkdir — フォルダを新しく作る。新しい箱を用意するだけなので安全
move — ファイルを別の場所に移す。引っ越しのようなもの
cp — ファイルをコピーする。バックアップを取るときにも使える
■バージョンの確認
node --version — Node.jsが入っているか、どのバージョンかを見るだけ
npm --version — npmが入っているか確認するだけ
■プロジェクト操作
npm init -y — プロジェクトの初期設定ファイルを作る。ノートの表紙を作るようなもの
npm start — プロジェクトを起動する。アプリの「再生ボタン」を押す感覚
■Webアクセス←これは削除してもOK、私が個人的に追加
Pexels、Pixabay — 無料で使える写真・画像サイト。素材探しに使う
code.claude.com — Claudeの公式ドキュメント。使い方を調べにいける
raw.githubusercontent.com — GitHubにある公開ファイルを取得するため
agentnativedev.medium.com — 参考にしているブログ記事を読むため
ここに載っていない操作はどうなるかというと、AIが毎回「これ実行していいですか？」と確認を求めてきます。たとえばソフトのインストールは、あえてこのリストに入れていません。毎回説明してもらって、自分で判断してからOKを出したいからです。
⛔️deny（禁止リスト）— 絶対にやってはダメなこと
ここが一番大事です。本当に重要です。
rm — ファイルを消す（Mac/Linux用）
rm -rf — フォルダごと中身も全部強制削除する。最も危険な操作
del — ファイルを消す（Windows用）
rmdir — フォルダを消す（Windows用）
Remove-Item — ファイルやフォルダを消す（Windowsの別のやり方）
名前は5つありますが、やっていることは全部同じ。「消す」操作です。Windows、Mac、Linuxそれぞれの方言があるので、全部まとめてブロックしています。
ここに入れた操作は、AIがどんなに「必要です」と判断しても、システムが自動で止めます。お願いではなく、物理的にできない状態です。
CLAUDE.md — AIに「ルール表」を見せる
2つ目のファイルが CLAUDE.md です。
settings.local.json が「鍵」なら、こちらは「壁に貼ったルール表」です。AIがこのファイルを読んで、自分がどう振る舞うべきかを理解します。
私が使っている実際の内容はこちらです。最初に紹介した3つのポストを参考に非エンジニア向けとClaude Codeの能力を引き上げる内容としてブラッシュアップしています。長いですが、全文載せます。
## ワークフローのオーケストレーション（日本語版・統合）

### 0. 運用ルール（あなたが入力した内容に対する実行指示）
- あなたが入力した内容は、次のやり方で処理する。
- 作業は「バックグラウンドで動く担当（バックグラウンド担当）」が進める想定にする。
- 一定間隔で状況を確認し、やっていることを短く要約して伝える。
- あなたは技術者ではない前提で、簡潔で分かりやすく説明する。
- 仕組みの説明は「探索担当（解説担当）」を使って、バックグラウンド担当が何をしているか／どう動くかを学べる形で説明する。
- バックグラウンド担当がエラーに遭遇したら、必ずあなたに伝える。
  - その際、「どう直せばいいか」を分かる言葉で案内する。
- すべて日本語で回答する。
- 技術用語は使わない（使わざるを得ない場合は、必ず言い換えを先に置く）。

### 1. Plan ノード（デフォルト）
- どんな「手順が多い作業」でも plan モードに入る（目安：3ステップ以上、または大事な判断が必要なもの）
- 何かがおかしくなったら、すぐに STOP して再計画する（押し切らない）
- plan モードは「作る」だけでなく「確かめる手順」にも使う
- 曖昧さを減らすため、最初に詳細な仕様を書く

### 2. サブエージェント戦略（役割分担）
- メインの作業メモを散らかさないため、役割分担を積極的に使う
- 調査・探索・並列作業は、別の担当に任せる
- 複雑な問題は、担当を増やして同時に進める
- 1担当につき1タスク（同時に抱えない）

### 3. 自己改善ループ
- あなたから修正が入ったら必ず：その学びを tasks/lessons.md に記録する
- 同じミスを防ぐための「自分向けルール」を書く
- ミスが減るまで繰り返し改善する
- セッション開始時に、関係する学びを見直す

### 4. 完了前の検証
- 動くことを証明するまで、完了扱いにしない
- 必要に応じて、変更前と変更後の動きの違いを確認する
- 自問する：「経験豊富な人が見てもOKと言えるか？」
- 確認作業・記録確認・正しさの実演を行う

### 5. エレガンス要求（バランス）
- 手順が多い変更では立ち止まり、「もっとスッキリしたやり方はあるか？」と問う
- 修正が無理やり感あるなら：「今の知識が最初からあるなら、自然な解決にする」
- 単純で明らかな修正はスキップ（やりすぎない）
- 提出前に、自分で厳しく見直す

### 6. 自律的なバグ修正
- バグ報告を受けたら：そのまま直しに行く（手取り足取りを求めない）
- 「どこが変／何が起きているか」を示し、その上で解決する
- あなたに余計な切り替え作業を要求しない
- 指示されなくても、失敗している自動チェックを直しに行く

## タスク管理
1. まず計画：チェックできる項目で plan を tasks/todo.md に書く
2. 計画の確認：実装を始める前にチェックインする
3. 進捗の追跡：進めながら項目を完了にしていく
4. 変更の説明：各ステップで高レベルの要約を書く
5. 結果の文書化：tasks/todo.md にレビューセクションを追加する
6. 学びの記録：修正が入った後に tasks/lessons.md を更新する

## コア原則
- シンプル最優先：あらゆる変更を可能な限りシンプルに。最小限の手数で最大効果。
- 怠けない：根本原因を見つける。一時しのぎは禁止。きちんと直す。
- 最小影響：必要な箇所だけを触る。新しい問題を持ち込まない。

## 安全ルール（ファイル・コマンド操作の保護）

### 7. 既存ファイルの上書き禁止（確認必須）
- 既存ファイルを編集・上書きする前に、必ず「〇〇を上書きしますが、よろしいですか？」と確認する
- 確認なしに既存ファイルの内容を変更しない
- 可能であれば変更前のバックアップを作成する（例：filename.bak）

### 8. 削除コマンドの実行禁止
- rm、del、rmdir などの削除系コマンドは原則として実行しない
- どうしても必要な場合は、対象ファイル名と理由を明示して承認を得てから実行する
- rm -rf のような再帰的強制削除は、いかなる場合も実行しない

### 9. パッケージ追加は事前説明と承認が必要
- npm install、pip install、brew install などのパッケージ追加コマンドは、実行前に以下を説明する：
  - 何をインストールするか（パッケージ名）
  - なぜ必要か（目的・用途）
  - 影響範囲（グローバルかローカルか）
- 説明後、承認を得てから実行する

### 10. 不明なコマンドは実行前に日本語で説明
- 私がエンジニアではないことを常に意識する
- 実行しようとするコマンドが技術的・専門的な場合は、実行前に日本語で以下を説明する：
  - このコマンドは何をするか（平易な言葉で）
  - 実行するとどうなるか（結果・影響）
  - リスクがある場合はその内容
- 説明後、「実行してよいですか？」と確認する
長いですが、やっていることはシンプルです。順に見ていきます。
運用ルール（0番）— 「どう振る舞ってほしいか」の基本姿勢
ここで伝えているのは、AIの態度や話し方の基本設定です。
私はエンジニアではないので、「すべて日本語で」「専門用語は使わない」「分かりやすく説明する」と明記しています。これを書いておかないと、AIは英語やコマンドを交えた説明をしてきます。
作業スタイル（1〜6番）— 「どう仕事を進めるか」のルール
AIがどんなふうに仕事を進めるかのルールです。
特に大事なのは「1. Plan ノード」の部分。複雑な作業をいきなり始めるのではなく、まず計画を立ててから動くようにしています。人間のアシスタントに「まず段取りを見せて」と言うのと同じです。
「3. 自己改善ループ」も面白いところで、AIが自分のミスを記録して、同じ失敗を繰り返さないようにする仕組みです。
コア原則 — AIの「仕事のポリシー」
3つだけ。シンプルにやれ、手を抜くな、余計なことはするな。
これを書いておくと、AIが「ついでにこれも直しておきました」と勝手に範囲を広げるのを防げます。
安全ルール（7〜10番）— ここが今回のメイン
ここが settings.local.json と連動する部分です。
7番：ファイルを書き換える前に「いいですか？」と聞く
8番：ファイルを消す操作は原則やらない
9番：ソフトを入れる前に「何を・なぜ・どこに」を説明する
10番：よくわからないコマンドは実行前に説明する
settings.local.json は「仕組みとして止める」ものでした。CLAUDE.md の安全ルールは「なぜ止めるべきかをAIに理解させる」ためのものです。
なぜ2つとも必要なのか
ここまで読んで「片方だけでよくない？」と思った方もいるかもしれません。
実は、私も最初はそう思いました。
CLAUDE.md にルールを書いておけばAIは守ってくれるはず。そう考えるのは自然です。でも、AIは長い作業をしているうちにルールを見落とすことがあります。人間だって、壁に貼ったルール表を毎回ちゃんと読むかというと、ちょっと怪しい。
逆に、settings.local.json だけではどうか。こちらは仕組みとして確実にブロックできます。でも「やっていい・ダメ」しか伝えられない。理由は書けません。

まとめ — 二重ロックで、安心してAIと仕事する
Claude Codeは非エンジニアにとっても強力なアシスタントです。 でも、最初に「やっていいこと」と「ダメなこと」を教えておかないと、思わぬところでヒヤッとする場面が出てきます。
やることは2つだけです。
.claude/settings.local.json を作って、許可と禁止のリストを書く
.claude/CLAUDE.md を作って、AIへのルールと理由を文章で書く
settings.local.json は「鍵」、CLAUDE.md は「ルール表」。 片方だけではなく、両方を組み合わせた「二重ロック」がおすすめです。
非エンジニアだからこそ、この設定をやっておく価値があります。
エンジニアの方は、何かあっても自分で元に戻せる場合もある。でも、私たちはそうはいかない。だからこそ、先に守りを固めておく。そうすれば安心してAIに任せられるし、もっと大胆に使えるようになります。
ちなみにsettings.local.json 、CLAUDE.mdは現時点での設定です。日々有益な設定が流れてきたり、公式がセキュリティ面を強化して指示が不要になるかもしれません。「今回設定したから完璧だ！」と安心しきらず、更新し続けていきましょう！
