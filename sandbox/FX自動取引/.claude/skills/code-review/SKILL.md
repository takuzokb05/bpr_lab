# Code Review - 多角的コードレビュー

## 概要

コードを堅牢性・効率性・セキュリティ・保守性の4つの観点から体系的にレビューし、重大度分類付きの改善提案を行うスキル。

**対象**: あらゆる言語・フレームワークのコードレビュー
**キーワード**: コードレビュー, 品質改善, セキュリティ, リファクタリング

## コアプロセス

### 1. スコープ確認

レビュー対象を特定し、文脈を把握する:
- 対象ファイル・関数・変更差分の範囲
- プロジェクトの技術スタック・アーキテクチャ
- 変更の目的（新機能 / バグ修正 / リファクタリング）
- プロジェクト固有の規約（claude.md / コーディング規約）

### 2. 4観点チェックリスト走査

#### 観点A: 堅牢性（Robustness）
- [ ] エラーハンドリングは適切か（例外の握りつぶし、不十分なcatch）
- [ ] エッジケースへの対応（null/undefined、空配列、境界値）
- [ ] 入力バリデーション（型チェック、範囲チェック、サニタイズ）
- [ ] リソース管理（ファイル/DB接続のクローズ、メモリリーク）
- [ ] 並行処理の安全性（競合状態、デッドロック）

#### 観点B: 効率性（Efficiency）
- [ ] アルゴリズムの計算量（O(n²)の改善余地、不要なループ）
- [ ] データベースクエリ（N+1問題、インデックス不足、不要なフルスキャン）
- [ ] メモリ使用（大量データの一括読み込み、不要なコピー）
- [ ] I/O操作（不要なファイルアクセス、API呼び出しの最適化）
- [ ] キャッシュ活用の余地

#### 観点C: セキュリティ（Security）
- [ ] インジェクション対策（SQL、XSS、コマンド、パストラバーサル）
- [ ] 認証・認可の適切性
- [ ] 機密情報の扱い（ハードコード、ログ出力、エラーメッセージ漏洩）
- [ ] 依存パッケージの脆弱性
- [ ] CORS / CSRFの設定

#### 観点D: 保守性（Maintainability）
- [ ] 命名の明確さ（変数名、関数名、クラス名が意図を表現）
- [ ] 関数・クラスの責務分離（単一責任原則）
- [ ] コードの重複（DRY原則違反）
- [ ] 適切な抽象化レベル（過度な抽象化 / 抽象化不足）
- [ ] テスタビリティ（依存関係の注入、モック可能性）

### 3. 重大度分類

各指摘を以下の重大度で分類する:

| 重大度 | 基準 | 対応 |
|--------|------|------|
| **Critical** | セキュリティ脆弱性、データ損失リスク、本番障害の可能性 | 即座に修正必須 |
| **High** | バグの温床、パフォーマンス劣化、重要な設計問題 | リリース前に修正 |
| **Medium** | 可読性低下、軽微な非効率、改善推奨の設計 | 次回以降で対応 |
| **Low** | スタイル、命名の好み、微細な改善提案 | 任意で対応 |

### 4. 改善提案

各指摘に対して以下を含める:
- **問題の説明**: 何が問題で、なぜ問題なのか
- **修正前コード**: 問題箇所の引用
- **修正後コード**: 具体的な修正例
- **補足**: 修正の根拠・参考情報

### 5. 総合評価

レビュー全体のサマリーを出力:
- 観点別の評価（A〜D各5段階）
- Critical/High件数のサマリー
- 特に良い実装の言及（ポジティブフィードバック）
- 優先的に対応すべき項目トップ3

## ルール

1. **指摘には必ず修正例を添える** - 「ここがダメ」だけでなく「こうすれば良い」を示す
2. **良い実装も評価する** - 問題点だけでなく、優れた設計や工夫にも言及する
3. **プロジェクト文脈を考慮** - プロジェクトのclaude.md・規約があればそれに準拠して評価
4. **過度な指摘を避ける** - Lowレベルの指摘を大量に出すより、Critical/Highに集中する
5. **建設的なトーン** - 批判ではなく改善提案として伝える

## 出力フォーマット

```markdown
# コードレビュー結果

## レビュー対象
- ファイル: (対象ファイル一覧)
- 変更目的: (新機能 / バグ修正 / リファクタリング)

## 指摘事項

### [Critical] (タイトル)
- **観点**: セキュリティ
- **場所**: `file.py:42`
- **問題**: (問題の説明)
- **修正前**:
  ```python
  (問題コード)
  ```
- **修正後**:
  ```python
  (修正コード)
  ```

### [High] (タイトル)
...

## 良い実装
- (ポジティブフィードバック)

## 総合評価
| 観点 | 評価 |
|------|------|
| 堅牢性 | ★★★★☆ |
| 効率性 | ★★★☆☆ |
| セキュリティ | ★★★★★ |
| 保守性 | ★★★★☆ |

## 優先対応事項
1. (最優先の修正)
2. (次に重要な修正)
3. (その次)
```

## 使用例

### ケース1: PR レビュー
```
ユーザー: このPRをレビューして
→ 変更差分を読み取り、4観点でチェック
→ 重大度分類付きの指摘リストを出力
→ 良い実装へのポジティブフィードバックも含める
```

### ケース2: 既存コードの品質監査
```
ユーザー: src/ 以下のコード品質を確認したい
→ 主要ファイルを走査
→ 観点別にクリティカルな問題を洗い出し
→ 優先度付きの改善ロードマップを提案
```

## 注意事項

- 大規模なコードベース全体のレビューは、ディレクトリ単位で分割して実施する
- 言語固有のベストプラクティスはプロジェクトの技術スタックに合わせる
- 自動フォーマッター・リンターで対処可能な指摘（インデント、セミコロン等）はLowに分類し、ツール導入を提案する
