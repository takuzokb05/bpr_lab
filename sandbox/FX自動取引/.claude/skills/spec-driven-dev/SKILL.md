# Spec-Driven Dev - 仕様書駆動開発

## 概要

「作りたいもの」を自然言語から構造化された仕様書に変換し、仕様書に基づいて段階的に実装を進めるワークフロースキル。要件の抜け漏れを防ぎ、実装の順序と完了基準を明確にする。

**対象**: 新機能開発、ツール開発、プロトタイプ構築
**キーワード**: 仕様書, 段階的実装, 要件定義, ワークフロー, プロジェクト管理

## コアプロセス

### Phase 1: 要件構造化

ユーザーの自然言語の意図を構造化する:

1. **目的の明確化**: 何を、誰のために、なぜ作るのか
2. **機能の洗い出し**: 必要な機能をリストアップ
3. **優先度付け**: Must / Should / Could / Won't に分類
4. **制約の特定**: 技術スタック、外部依存、期限、パフォーマンス要件
5. **確認**: ユーザーに構造化した要件を提示し、認識齟齬がないか確認

### Phase 2: 仕様書作成

以下のテンプレートに沿って仕様書を作成する:

```markdown
# [プロジェクト名] 仕様書

## 1. 概要
- 目的:
- 対象ユーザー:
- 主要な価値:

## 2. 機能一覧
| ID | 機能名 | 説明 | 優先度 | 依存 |
|----|--------|------|--------|------|
| F1 | ... | ... | Must | - |
| F2 | ... | ... | Must | F1 |
| F3 | ... | ... | Should | F2 |

## 3. データ構造
- 入力データ:
- 出力データ:
- 永続化（DB/ファイル）:

## 4. 外部依存
- API:
- ライブラリ:
- サービス:

## 5. 実装順序
F1 → F2 → F3 → ...（依存関係に基づく）

## 6. 各機能の詳細仕様
### F1: [機能名]
- 入力:
- 処理:
- 出力:
- エラーケース:
- 完了基準:
```

### Phase 3: 実装計画

仕様書から具体的な実装計画を策定する:

1. **ディレクトリ構造の設計**: プロジェクト構造を定義
2. **機能→ファイルのマッピング**: 各機能がどのファイルに実装されるか
3. **実装順序の確定**: 依存関係グラフに基づくF1→F2→...の順序
4. **各機能の作業ステップ**: ファイル作成→実装→テスト→確認のサイクル

### Phase 4: 段階的実装

各機能を以下のサイクルで実装する:

```
┌──────────────────────────┐
│ 1. 機能の説明            │
│   （何を作るか確認）      │
└──────┬───────────────────┘
       │
       ▼
┌──────────────────────────┐
│ 2. テストデータ / 期待値  │
│   （先に完了基準を明確化） │
└──────┬───────────────────┘
       │
       ▼
┌──────────────────────────┐
│ 3. 実装                  │
│   （仕様書に沿って記述）  │
└──────┬───────────────────┘
       │
       ▼
┌──────────────────────────┐
│ 4. 動作確認              │
│   （テスト実行 / 手動確認）│
└──────┬───────────────────┘
       │
       ▼
┌──────────────────────────┐
│ 5. ユーザー確認          │
│   （OK → 次の機能へ）     │
└──────────────────────────┘
```

**重要ルール**: 1機能ずつ実装し、ユーザー確認を得てから次へ進む。一度に全機能を実装しない。

### Phase 5: 完了検証

全機能の実装完了後:

1. **仕様書との突合**: 各機能の完了基準をチェック
2. **統合テスト**: 機能間の連携が正しく動作するか
3. **仕様書の更新**: 実装中に変更された仕様を反映
4. **残課題の整理**: Should/Could機能の今後の計画

## ルール

1. **仕様書ファーストで実装しない** - Phase 2の仕様書作成をスキップしない。ユーザーが急いでいても「まず仕様書を作りましょう」と提案する
2. **F1→F2→F3の順序を守る** - 依存関係のある機能を飛ばして実装しない
3. **各機能で必ずユーザー確認** - 自動的に次の機能に進まない
4. **仕様変更は仕様書に反映** - 実装中に仕様が変わったら仕様書を先に更新する
5. **完了基準を先に定義** - 実装の前に「何ができたら完了か」を明確にする

## 出力フォーマット

### 要件構造化の出力
```markdown
## 要件整理

### 目的
(ユーザーの意図を構造化)

### 機能一覧
| ID | 機能名 | 優先度 |
|----|--------|--------|
| F1 | ... | Must |

### 確認事項
1. (認識齟齬がないか確認したい点)
2. ...
```

### 進捗管理の出力
```markdown
## 実装進捗

- [x] F1: データ構造定義 ✓
- [x] F2: 基本機能 ✓
- [ ] **F3: スコアリング** ← 現在
- [ ] F4: レポート生成

### 次のステップ
F3の実装を開始します。まず...
```

## 使用例

### ケース1: 新ツール開発
```
ユーザー: SlackのメッセージをAIで要約するボットを作りたい

→ Phase 1: 目的・機能・制約をヒアリング
→ Phase 2: 仕様書を作成（F1:Slack接続, F2:メッセージ取得, F3:AI要約, F4:投稿）
→ Phase 3: ディレクトリ構造・ファイルマッピング
→ Phase 4: F1から順に実装（各機能でユーザー確認）
→ Phase 5: 全体テスト・仕様書更新
```

### ケース2: 既存機能の拡張
```
ユーザー: 既存のダッシュボードにエクスポート機能を追加したい

→ Phase 1: 既存コードの理解 + 新要件の整理
→ Phase 2: 追加機能の仕様書（既存仕様との整合性確認）
→ Phase 3-5: 段階的実装
```

## 注意事項

- 小規模な修正（バグ修正、1関数追加など）にはこのワークフローは過剰。適切な粒度で使うこと
- 仕様書はMarkdownファイル（`SPEC.md` や `docs/spec.md`）としてプロジェクト内に保存し、Git管理する
- ユーザーが「全部一気に作って」と言っても、Phase 4の段階的サイクルは崩さない。ただし確認の粒度は調整可能（F1+F2をまとめて確認など）
