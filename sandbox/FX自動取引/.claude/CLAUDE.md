# FX自動取引システム

## プロジェクト概要

- **目的**: AIを活用した安全で効率的なFX自動取引システムの調査・設計・実装
- **対象ユーザー**: 個人投資家（開発者本人）
- **主要な価値**: リスクを最小化しながら安定的なリターンを実現する自動取引の仕組み
- **メインテーマ**: いかに安全に効率よく取引できるか

## 技術スタック

- **言語**: Python 3.9+
- **フレームワーク**: 調査後に決定（候補: Streamlit / FastAPI）
- **データベース**: SQLite（取引ログ・バックテスト結果）
- **外部サービス**: FXブローカーAPI（調査後に選定）、AI/ML関連API
- **パッケージ管理**: pip + requirements.txt

## 開発ルール

### コーディング規約
- コメントは日本語で書く
- 変数名・関数名は意味が明確な英語を使用
- 型ヒントを使用する（`def func(name: str) -> dict:`）
- 1関数は1つの責務に絞る
- マジックナンバーは `config.py` に定数として定義

### Git規約
- コミットメッセージ: `<type>(<scope>): <subject>`
  - type: feat / fix / docs / refactor / test
  - 日本語OK
- ブランチ: `feature/xxx`, `fix/xxx`
- コミット前に動作確認を実施すること

### API連携ルール

#### 環境変数管理
- APIキー・シークレットは `.env` に格納し、`.gitignore` で除外
- ハードコードされた認証情報は絶対に禁止
- `.env.example` にキー名のみ（値は空）を記載

#### リトライ・タイムアウト
- 全API呼び出しに `timeout` を設定する（デフォルト: 30秒）
- 一時的なエラー（5xx, Timeout）は指数バックオフでリトライ（最大3回）
- レート制限エラー（429）はRetry-Afterヘッダに従う

### エラーハンドリング
- 例外を握りつぶさない（空のexcept禁止）
- 外部API呼び出しにはタイムアウトとリトライを設定
- ユーザーに見せるエラーメッセージは日本語で分かりやすく
- ログには `logging` モジュールを使用（print禁止）

## プロジェクト固有のルール

### フェーズ構成

**Phase A: 調査**（完了）
- FX自動取引の現状、リスク管理、AI手法、技術設計を調査
- 調査ドキュメント4本を作成（ファクトチェック済・反論屋v3承認済）

**Phase B: 設計・実装**（現在 — Phase 1: 基盤構築）
- 調査結果に基づいてアーキテクチャ設計
- spec-driven-devスキルに従い段階的に実装
- 仕様書: `docs/SPEC.md`

### 調査の4つの問い

1. **FX自動取引の現状はどうなっているか** — プラットフォーム、API、規制、既存ソリューション
2. **安全に取引するためのリスク管理とは** — 損切り、ポジションサイジング、ドローダウン制御
3. **AIはFX取引のどこに有効か** — 予測モデル、センチメント分析、強化学習の実用性
4. **どう作るのが現実的か** — アーキテクチャ、バックテスト、ペーパートレード、段階的移行

### 調査ルール
- 主張にはソース（URL・出典）を必ず付ける
- 「絶対儲かる」系の情報を排除し、リスクを正直に記述する
- バックテスト結果と実取引のギャップ（スリッページ、約定力等）を常に意識する
- 日本の法規制（金融商品取引法等）を考慮する
- 具体的な数字・データを優先する

### 安全性に関する原則（最重要）
- **資金管理が最優先**: どんなに優秀な戦略でも、資金管理が甘ければ破綻する
- **最大損失を先に決める**: 1取引あたり、1日あたり、月あたりの損失上限を設定
- **ペーパートレードで十分に検証してからリアル取引**: 最低3か月
- **自動停止機構は必須**: 想定外の動きを検知したら即停止する仕組み
