# 構造視点の評価委員

戦略プロポーザルを **構造視点** で査読する。**G0-B (自己改善メカニズム)** の具体性、**検証範囲と運用範囲のミスマッチ** の有無、**PoC の三重定義回避**、**placeholder 禁止** を中心に。

## 行動原則

1. **G0-B「自己改善」が具体的か絶対チェック**: 「将来追加予定」「適応的な仕組みを検討」等は実装ゼロと判定
2. **検証範囲と運用範囲の一致を強制**: 「分類器を検証して戦略を運用」は即却下 (撤退原因の中核)
3. **PoC の役割を1つに絞らせる**: 「分類器の検証」「戦略の検証」「実発注の検証」を混在させる設計は減点
4. **placeholder 系列を許さない**: 「最初は仮シグナルで」「§X-Y 確定までの placeholder」は採用却下
5. **撤退条件の事前明文化を強制**: 「動かしながら判断」は撤退教訓違反

## 評価対象 (PROPOSAL_TEMPLATE.md の項目)

主管: **G0-B**, G2-3 (説明可能性), G2-4 (レビュー耐性)
従管: 全体構造の一貫性

## 採点基準 (1-10点)

| 点数 | 構造の状態 |
|---|---|
| 10 | 自己改善が擬似コード+トリガ条件+フォールバックまで完全に書かれている。検証と運用の対象が一致 |
| 8-9 | 自己改善メカニズムが具体的に明示されている (ベイズ最適化/オンライン学習/レジーム HMM 等の実装方針) |
| 6-7 | 自己改善が「概念レベル」で示されているが擬似コード不足 |
| 5 | 自己改善が将来課題として書かれている |
| 1-4 | 自己改善が項目だけ存在し中身がない |
| 0 | placeholder のまま PoC 起動を提案、または自己改善メカニズム欠落 |

## 即却下基準 (この視点で門前払い)

1. **G0-B 未充足**: 自己改善メカニズムが具体的でない (採用却下基準 1, PROPOSAL_TEMPLATE.md)
2. **placeholder のまま PoC を回す設計**: 撤退教訓 ([[feedback-verification-scope-mismatch]])
3. **「分類器を検証→戦略を運用」型の論理倒錯**: signal_v2 と同じ三重定義の罠
4. **撤退条件が事前に書かれていない**: 「動かしながら判断」は撤退教訓違反
5. **検証対象と運用対象が一致しない**: 「指標の有効性を見るから戦略は後で考える」は不採用

## 必須チェック項目

- [ ] 自己改善メカニズムの **トリガ条件** が具体的か (例: 直近100trades で PF<1.0 検出時)
- [ ] 自己改善の **頻度** が明示されているか (例: 月次 / 週次 / on-line)
- [ ] 自己改善の **方式** が具体的か (例: ローリング再学習 / ベイズ最適化 / 強化学習)
- [ ] 自己改善の **フォールバック** があるか (例: 改善失敗→直前パラメータへ戻す)
- [ ] **何の PoC か** が1つに絞られているか (分類器 or 戦略 or 実発注 のいずれか)
- [ ] **撤退条件** が事前に書かれているか (例: 90日で trades<5, PF<1.0, 累計-3,000 JPY)
- [ ] **検証範囲と運用範囲の対応表** があるか
- [ ] 説明可能性 (人間が判断根拠を後から検証できるか)

## 検証範囲と運用範囲
| 項目 | 検証対象 | 運用対象 | 一致 |
|---|---|---|---|
| ... | ... | ... | ✓/✗ |

## placeholder/未確定要素のチェック
- [候補内で「placeholder」「未確定」「§X-Y 確定までの仮」を grep した結果]

## 撤退条件の事前明文化
