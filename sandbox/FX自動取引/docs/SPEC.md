## 3. データ構造

### 入力データ
- OANDA REST API v20からのOHLCV価格データ（4時間足メイン）
- 口座情報（残高、証拠金、保有ポジション）
- 環境変数（`.env`）: APIキー、口座ID

### 出力データ
- 取引シグナル（BUY / SELL / HOLD）
- 注文（成行/指値 + SL/TP）
- 取引ログ

### 永続化（SQLite）
- `prices` テーブル: OHLCV価格データ
- `trades` テーブル: 取引ログ（エントリー/イグジット/損益）
- `backtest_results` テーブル: バックテスト結果・メトリクス
- `kill_switch_log` テーブル: キルスイッチ発動記録

## 6. 各機能の詳細仕様

### F1: 設定管理（`src/config.py`）

**入力**: `.env`ファイル、定数値

**処理**:
- `python-dotenv` で環境変数を読み込み
- リスクパラメータを定数として定義
- 設定値のバリデーション（必須キーの存在確認、数値範囲チェック）

**出力**: 他モジュールからインポート可能な定数群

**定義する定数**（doc 04セクション4.1準拠）:

```python
# === 環境変数（.envから読み込み） ===
OANDA_API_KEY: str          # OANDAのAPIトークン
OANDA_ACCOUNT_ID: str       # OANDA口座ID
OANDA_ENVIRONMENT: str      # "practice"（デモ）or "live"

# === 1トレードあたりのリスク ===
MAX_RISK_PER_TRADE = 0.01       # 口座資金の1%
MAX_RISK_PER_TRADE_HARD = 0.02  # 絶対上限2%

# === レバレッジ ===
MAX_LEVERAGE = 10               # 自主制限（法的上限25倍の40%）

# === ドローダウン制御（5段階） ===
DRAWDOWN_LEVELS = {
    0.05: "WARNING",       # 5%: 警告通知
    0.10: "REDUCE",        # 10%: ポジションサイズ半減
    0.15: "MINIMUM",       # 15%: 最小ロットのみ
    0.20: "STOP",          # 20%: 全停止（ハードリミット）
    0.25: "EMERGENCY",     # 25%: 全ポジション強制決済
}

# === 日次・月次損失上限 ===
MAX_DAILY_LOSS = 0.02           # 日次: 口座の2%
MAX_WEEKLY_LOSS = 0.05          # 週次: 口座の5%
MAX_MONTHLY_LOSS = 0.10         # 月次: 口座の10%

# === 連続負け制限 ===
MAX_CONSECUTIVE_LOSSES = 5      # 5連敗で一時停止

# === ポジション制限 ===
MAX_OPEN_POSITIONS = 3          # 最大同時ポジション数
MAX_CORRELATION_EXPOSURE = 2    # 相関通貨ペアの最大同時保有数

# === 戦略パラメータ ===
MA_SHORT_PERIOD = 20            # 短期MA期間
MA_LONG_PERIOD = 50             # 長期MA期間
RSI_PERIOD = 14                 # RSI計算期間
RSI_OVERBOUGHT = 70             # RSI買われすぎ閾値
RSI_OVERSOLD = 30               # RSI売られすぎ閾値
ATR_PERIOD = 14                 # ATR計算期間
ATR_MULTIPLIER = 2.0            # ATR損切り乗数
MIN_RISK_REWARD = 2.0           # 最小リスクリワード比

# === タイムフレーム ===
MAIN_TIMEFRAME = "H4"           # メインタイムフレーム（4時間足）

# === API設定 ===
API_TIMEOUT = 30                # APIタイムアウト（秒）
API_MAX_RETRIES = 3             # 最大リトライ回数

# === キルスイッチ閾値 ===
KILL_ATR_MULTIPLIER = 3.0       # ATR通常の3倍でボラティリティキル
KILL_SPREAD_MULTIPLIER = 5.0    # スプレッド通常の5倍でスプレッドキル
KILL_API_DISCONNECT_SEC = 30    # API切断30秒で全ポジション決済
```

**エラーケース**:
- `.env` に必須キー（`OANDA_API_KEY`, `OANDA_ACCOUNT_ID`）が未設定 → 起動時にエラーメッセージ表示して停止
- 数値パラメータが範囲外（例: `MAX_RISK_PER_TRADE` が0以下 or 1以上）→ バリデーションエラー

**完了基準**:
- [x] 全定数が定義済み
- [x] `.env` から環境変数が正しく読み込める
- [x] 必須キー欠損時にわかりやすいエラーメッセージが出る
- [x] `pytest` で設定バリデーションのテスト通過

### F3: OANDA API接続（`src/oanda_client.py`）

**入力**: `config.py` の環境変数、APIリクエストパラメータ

**処理**:
- `oandapyV20` を使って OANDA REST API v20 と通信
- `BrokerClient` インターフェースの6メソッドを実装
- 全API呼び出しに `timeout=30秒` を設定
- 5xx/Timeout は指数バックオフでリトライ（最大3回）
- 429はRetry-Afterに従う

**出力**: 各メソッドの戻り値（DataFrame、dict、list）

**エラーケース**:
- API認証失敗 → `AuthenticationError` を送出、日本語メッセージでログ
- タイムアウト → リトライ後も失敗なら `ConnectionError` を送出
- レート制限 → Retry-After秒待って再試行
- 不正なinstrument → `ValueError` を送出

**完了基準**:
- [x] デモ口座で価格データ取得できる
- [x] デモ口座で成行注文 → 決済ができる
- [x] タイムアウト・リトライが動作する
- [x] `pytest` でモック使用のユニットテスト通過
- [x] デモ口座での統合テスト通過

### F4: リスク管理コア（`src/risk_manager.py`）

**入力**: 口座情報（残高、ポジション）、価格データ（ATR算出用）、取引履歴

**処理**:
1. **ポジションサイジング**: 1-2%ルールに基づくロット数計算
2. **ドローダウン制御**: 5段階の閾値判定 → 対応アクション
3. **損失上限チェック**: 日次/週次/月次の損失集計 → 上限到達で停止
4. **連続負けカウンター**: 連敗数追跡 → 5連敗で24時間停止
5. **レバレッジ制限**: 実効レバレッジが10倍以下か検証
6. **キルスイッチ**: 6種類の発動条件を監視

**出力**:
- ポジションサイズ（ロット数）
- 取引許可/拒否の判定結果
- キルスイッチ発動状態
- ドローダウンレベル

**キルスイッチ6種類**（doc 04セクション4.2準拠）:

| 種類 | 発動条件 | アクション |
|------|---------|-----------|
| 日次損失 | 日次損失 > 2% | 当日取引停止 |
| 連続負け | 5連敗 | 24時間停止 |
| ボラティリティ | ATR > 通常の3倍 | 新規エントリー停止 |
| スプレッド | スプレッド > 通常の5倍 | 新規エントリー停止 |
| API接続断 | 接続断 > 30秒 | 全ポジション決済 |
| 手動 | 管理者が発動 | 即時停止 |

**エラーケース**:
- 口座情報取得失敗 → 安全側に倒す（新規取引を拒否）
- ATR計算に必要なデータ不足 → 最大リスクの半分で計算
- キルスイッチ発動のログ記録失敗 → stderr + 最低限のファイル出力

**完了基準**:
- [x] ユニットテスト5項目全合格（doc 04セクション6.3）:
  1. ポジションサイジング: 口座サイズ変動時に1-2%ルール遵守
  2. ドローダウン制御: 5段階の閾値で正しいアクション発動
  3. 損失上限チェック: 日次/週次/月次の上限±0.1%以内で発動
  4. 連続負けカウンター: 正確にカウント・停止
  5. レバレッジ制限: 10倍超過する注文を拒否

### F7: RSIフィルター付きMA戦略（`src/strategy/ma_crossover.py`）

**入力**: OHLCV価格データ（`pd.DataFrame`）

**処理**（doc 04セクション5.1準拠）:
1. MA(20)とMA(50)のクロスを検出
2. RSI(14)でフィルタリング:
   - 買いシグナル: MA(20)がMA(50)上抜け **かつ** RSI(14) < 70
   - 売りシグナル: MA(20)がMA(50)下抜け **かつ** RSI(14) > 30
3. ATR(14) × 2.0 のトレーリングストップ算出
4. リスクリワード比 1:2 以上のみエントリー許可

**出力**: `Signal`（BUY / SELL / HOLD）、SL価格、TP価格

**エラーケース**:
- データ行数が50未満（長期MA計算不能） → `HOLD` を返す
- ATR計算不能 → `HOLD` を返す（エントリーしない = 安全側）

**完了基準**:
- [x] バックテストでWFE > 50%
- [x] バックテストでシャープレシオ > 1.0（年率、doc 04セクション11.0定義）
- [x] バックテストで最大ドローダウン < 20%
- [x] `pytest` でシグナル生成ロジックのテスト通過
