### B-2. 過剰実装かどうかの判定

前回の review でこれは「P1 中、影響しない (デモ + lot 0.01)」と判定していた。M1 で「30分の WARN ログ」のレベルを大幅超過して **完全配線** が実装された。pragmatist として 30秒 BT 精神でこれを見ると:

**評価**: **過剰だが許容**。理由:
- 実装サイズは `update_spread` 30行 + `check_spread_anomaly` 12行 + ループ内呼び出し 40行 = 約 80 行で局所化されている
- EMA (alpha=0.1) は 1 ループで状態追従、テストも 2 件で完結
- baseline 確立期間 (起動後 数時間) の挙動が PHASE_2A_PLAN.md §2.1 で「この期間は異常検知が発火しない誤発火防止設計」と明記されており、起動直後の混乱を回避
- 異常値で baseline 自体を汚染させない設計 (`update_spread` で 3 倍超を更新スキップ) は副作用の連鎖を断っており、これは「YAGNI 違反」ではなく「実際に問題化しうる連鎖の遮断」

**ただし注意**: Ultra/Karen の要求レベル (デモ口座 + lot 0.01) に対しては明らかに過剰投資。Phase 2'A スコープなら「WARN ログだけ」で十分だったはず。今回の 80 行 + 2 テストは Phase 2'C 本番 + ATR 可変 lot を見据えた前倒し投資として正当化される。**最小コスト原則からは逸脱しているが、本番までに必要な機能ではあるので無駄ではない**。

### B-3. baseline 確立期間中の挙動の妥当性

`check_spread_anomaly` は `baseline is None or baseline <= 0` で False を返す (誤発火防止)。これは:
- ✅ 起動直後にスプレッドがそもそも測れない期間に誤ブロックを出さない
- ✅ 初回 `update_spread` で baseline=current_pips に設定 → 2 回目以降は EMA 動作
- ⚠️ 初回サンプルが既に異常値だった場合 (例: 起動直後に NFP 直撃)、baseline 自体が高い値で初期化されるため、その後の正常値が異常検知されないリスク

**評価**: ⚠️ は理論上ありうるが、Phase 2'A 起動を「経済指標発表 30 分前」以外から始める運用ルールで回避可能 (実害なし)。**起動可否は左右しない**。

**判定**: **P0-2 完全解消**。要求超だが過剰投資は許容範囲。
