### B-2. 閾値の妥当性 [部分的に疑問]

**90日 trades<5 (条件1)**:
- 期待: USD_JPY ~8-12 trades/月、GBP_JPY ~20-25 trades/月 → 90日で USD 24-36 / GBP 60-75
- 撤退閾値「90日<5」は**極めて緩い**。期待値の 10% 以下に落ちて初めて発動。
- これは「Phase 2'A 30日で目標 trades 達成」ゲートとの**不整合**を生む。Phase 2'A 30日で USD<8 → ゲート FAIL → 60日延長で<5 → 撤退、というワークフローになるが、SPEC_V3 §7.1 と §4.5 がこの繋がりを明示していない。

**累計 -3,000 JPY (条件3)**:
- lot 0.01 で 1 pip = 10 JPY → **300 pips の累計損失**で発動
- USD_JPY ATR ≈ 60 pips、SL ≈ 90 pips (ATR×1.5) → **3-4 連敗で発動**
- これは厳しすぎる可能性。Phase 0' BT で勝率 44.3% なので、4 連敗の発生確率は (0.557)^4 ≈ 9.6% で月 1 回程度発生し得る
- **対策**: Phase 2'A 起動前に「条件3 が早期発動する確率」を Phase 0' データから推定し、想定通りか確認

**LLM コスト >5,000円 (条件4)**:
- §8.3 で「Phase 0' 実コスト: USD_JPY ¥0.62/件、GBP_JPY ¥0.62/件 → 月間想定 ~¥30-50」
- 5,000円閾値は **100倍のマージン**を持つ。早期検知には機能するが過剰防衛的に近い。許容。

### B-3. 緊急停止 (人手判断) のトリガ [部分的合格]

PHASE_2A_PLAN.md §6.4 で:
- Black Swan で 1日 -10% 超
- LLM CONFIRM 率 90%+ 連続
- 反論屋3体の中で 2体以上が「停止」判定

**評価**: トリガは具体的だが、**「CONFIRM 率 90%+ 連続」の検知が実装にない**。Slack 日次サマリで人が気づくレベル。自動検知の実装が薄い。

- GBP_JPY ≥0.60: PF 1.294 (n=469) ✅ 一致

### C-2. しかし IS-OOS 整合性の主張に虚飾あり [重大]

IMPROVEMENT_META_ANALYSIS.md L196-200 / SPEC_V3.md L23 で:
> 「IS PF 1.335 → OOS PF 1.376 (n=336/336)、**OOS で +0.041 改善**」

しかし `data/_cycle2_meta_analysis_temporal_v2.json` の `I_is_oos_split` を直接読むと、**標準的な「年単位半々分割」(IS 2024-05 〜 2025-06、OOS 2025-06 〜 2026-05) では**:

| 構成 | IS PF | IS n | OOS PF | OOS n | diff |
|---|---|---|---|---|---|
| USD_JPY ≥0.65 | 1.523 | 106 | **1.623** | 97 | **+0.100 (改善)** |
| GBP_JPY ≥0.60 | 1.329 | 269 | **1.236** | 200 | **-0.092 (劣化)** |
| **Proposal 3 (USD≥0.65 + GBP≥0.60、合算)** | **1.367** | 375 | **1.329** | 297 | **-0.038 (劣化)** |

**つまり**:
- SPEC_V3.md の「IS 1.335 → OOS 1.376 で +0.041 改善 (オーバーフィットなし)」は、**前期/後期非対称分割 (IS=2024-05〜2025-01 8ヶ月、OOS=2025-01〜2026-05 16ヶ月)** の場合の数字。これは「**n=336 で揃える分割**」を採用した結果、有利な数字になっただけ
- **標準的な期間半々分割**だと Proposal 3 は IS-OOS で **-0.038 劣化**
- 個別ペアでは GBP_JPY が **OOS -0.092 劣化** — Phase 2'B ゲートの PF≥1.30 を OOS で**ぎりぎり** (1.329 ≈ 1.30) しかクリアしていない

**虚飾の構造**: 「IS<OOS」を主張するために**特別な分割方法を採用し、その分割方法でこそ成立する数字**を SPEC v3 §0 TL;DR と §10.1 karen 反論応答に書いている。標準分割の数字を見れば「**OOS で性能維持** (RETREAT 撤退条件回避の根拠)」は **USD_JPY 単独でしか成立しない**。

### C-3. 直近 2025Q3+ PF 1.338 の信頼性 [一部疑問]

n=296 という総数は十分に見えるが、その内訳:
- USD_JPY ≥0.65 の各四半期 n: 2025Q3=3, 2025Q4=14, 2026Q1=62, 2026Q2=17 = 計 96件
- GBP_JPY ≥0.60 の各四半期 n: 2025Q3=38, 2025Q4=56, 2026Q1=87, 2026Q2=19 = 計 200件

つまり **直近 PF 1.338 の 67% が GBP_JPY**。USD_JPY は 2025Q3 で n=3 (PF 4.566)、2026Q2 で n=17 (PF 4.621) と**極めてスパース**で高変動。これらの高 PF が混入することで全体平均を持ち上げている。

| 四半期 | USD n | GBP n | Combined n | Combined cum_pips |
|---|---|---|---|---|
| 2024Q2 | 15 | 30 | 45 | +74.5 |

| 2024Q4 | 36 | 86 | 122 | +210.4 |

**所見**:
- **2025Q1 で USD_JPY n=0** — 1 四半期まるごとシグナルがゼロ。「8-12件/月」期待が**運用半年で 0 件**になる期間がある

これは「IS-OOS 安定性 ✅」の主張とは別に、**「シグナル枯渇リスク」が現実的**であることを示す。撤退条件1 (90日 trades<5) で USD_JPY が早期撤退する可能性は十分にある。

### C-4. 多重検定オーバーフィットの誠実な評価 [部分的に虚飾]

`_cycle2_meta_analysis_result.json` の D_optimal_confidence セクションから、**11 個の閾値 (0.4, 0.45, 0.5, 0.55, 0.6, 0.65, 0.7, 0.75, 0.8, 0.85, 0.9)** を試して 0.65/0.60 を選択している。

**しかし C-2 で示した通り、標準 IS/OOS 分割では Proposal 3 は OOS 劣化** しているため、緩和策の判定基準を Proposal 3 自身が満たしていない可能性がある。

## D. SPEC v2 撤退教訓の構造的継承

### D-1. PoC 三重定義回避 [合格]

RETREAT バグ④「分類器+戦略+実発注の同居」は、SPEC v3 でも「signal_v2 (戦略) + LLM (補完層) + confidence 閾値 (取捨)」の 3 層が demo_loop.py に同居しているが、**それぞれの役割が明確分離されている**:
- signal_v2: シグナル生成 (改変禁止、完成済戦略)
- LLM: 質判定 (再現性テストで安定性確認予定)
- 閾値: 取捨選択 (Proposal 3 で確定値)

各層の検証範囲は Phase 0' 5,059 件で個別に測定済 (CONFIRM only / 各閾値 PF など)。SPEC v2 の「分類器の検証 ★★★★★ を戦略の検証と取り違え」は**構造的に回避**されている。✅

ただし C-2 で指摘した通り「**LLM 補完層の OOS 検証**」が Proposal 3 では十分でない (GBP_JPY OOS -0.092 劣化)。三重定義は回避できているが、3 層目 (閾値) の OOS 妥当性が**部分的にしか保証されていない**。

### D-2. 経済性 Gate の構造的不在 [部分的合格]

RETREAT バグ③「経済性 Gate 構造的欠落」は、SPEC v3 §7.2 で Phase 2'B ゲート (PF≥1.3, Sharpe≥0.8, MaxDD≤15%, OOS trades 達成, LLM コスト≤5,000円) として**形式的に組み込み済**。✅

しかし以下の懸念:
1. **Phase 2'B ゲートの実装が存在しない**。30 日後に人が手動集計して判定する設計。`src/spec_v3/` に gate_evaluator.py のような自動判定モジュールがない。
2. **ゲートの数値 (PF≥1.3) と OOS 実測値 (1.329) がぎりぎり**。スプレッド・スリッページで -0.05 でも劣化したら FAIL する。
3. **ゲート FAIL 時のワークフローが「60日延長 → 撤退条件 #2 統合判定」と曖昧**。具体的に何を見て撤退/継続判断するか不明確。

### D-3. 撤退済 GBP_JPY 再採用の自己一貫性 [重大]

RETREAT バグ②「撤退済 GBP_JPY を 3 日後に再採用 (自己一貫性崩壊)」が、SPEC v3 で**形を変えて再発している可能性**:

**事実**:
- RETREAT_2026-05-26.md L21: **2026-05-07 GBP_JPY 撤退決定** (旧 MTFPullback で本番 -2,004 JPY)
- RETREAT_2026-05-26.md L41 (memory に展開): 亡き者本番 GBP_JPY 37件 / 勝率 35.1% / **PF 0.868** / -1,239 JPY / **年換算 -45,000 JPY**
- SPEC_V3.md §1.3: **GBP_JPY (副) として再採用**、Phase 0' BT で **PF 1.294 (n=469)**

**疑問**:
- 亡き者本番 PF 0.868 と Phase 0' BT PF 1.294 の乖離は **+0.426** — なぜか?
- 「戦略が違う (MTFPullback → signal_v2 + LLM) から本番実績は無関係」というロジックが SPEC v3 で**明示されていない**
- SPEC_V3.md §1.4 (除外ペア) で EUR_USD の除外理由は書かれているが、GBP_JPY 採用理由として「亡き者本番マイナスをどう扱うか」の言及がない

**反論**: 「signal_v2 + LLM フィルタは新規戦略なので、旧 MTFPullback の実績はゼロから出直しで評価すべき」は**部分的に正しい**。だが karen として:
- 同じ通貨ペアで本番マイナス実績がある事実
- スプレッド/スリッページ/Broker特性などペア固有の要因
- LLM が GBP_JPY のレジームをどう理解しているかの再現性

これらが**Phase 2'A 開始前に検証されていない**。SPEC v2 撤退の決定打となった「自己一貫性崩壊」のパターンと同じ可能性がある。

## karen 視点の結論

### 起動可否判定: **設計修正必要 (起動延期、最低 1-2 日)**

**根拠**:

前回 SPEC v2 撤退の決定打となった karen 指摘 (撤退済 GBP_JPY 再採用、★★★★★ 虚飾、経済性 Gate 不在) のうち:
- **撤退済 GBP_JPY 再採用** → SPEC v3 でも形を変えて再発 (D-3)、根拠の明示が不足
- **★★★★★ 虚飾** → SPEC v3 の「OOS で +0.041 改善」は標準分割では -0.038 劣化、虚飾の構造同じ (C-2)
- **経済性 Gate 不在** → 形式的には改善されたが、ゲート数値 (PF≥1.3) が OOS 実測値 (1.329) と**ぎりぎり** (D-2)

つまり**前回撤退の決定打 3 件のうち 2.5 件が形を変えて残存**している。これは「形式的継承」ではあるが「実質的継承」とは言えない。

ただし SPEC v3 自体は SPEC v2 と比較して**構造的改善は確実にある**:
- ✅ 撤退条件の事前明文化 (運用後追加禁止が定数化で機能)
- ✅ PoC 三重定義の役割分離 (signal_v2 改変禁止、LLM フィルタ独立、閾値層独立)
- ✅ 24 時間最大保持で亡き者挙動データを反映
- ✅ Phase 2'B → 2'C 段階移行で本番投入を慎重化

これらは継承の確実な成果。

### 「設計修正必要」と判定した理由

**1-2 日の追加作業で重大度の高い 3 件 (虚飾の正直化、スプレッドキルスイッチ配線、GBP_JPY 採用根拠整理) を解消できる**。これを Phase 2'A 30 日連続稼働の前に解消しておくのは合理的コスト。
逆に**修正せずに起動すると、30 日後に「あの時 karen が指摘した点がやはり問題」というパターンになるリスクが高い**。SPEC v2 撤退の effort heuristic 罠 (= 「ここまで作ったから進む」) を反復しないため、**直前修正は許容範囲**と判断。
