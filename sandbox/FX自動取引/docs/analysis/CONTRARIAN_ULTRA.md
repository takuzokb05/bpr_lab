**作成日**: 2026-05-25
**立場**: 第3視点・反論屋 (A/C の結論を踏襲せず批判する役)
**問い**: A/C は「閾値妥当・観察期間延長」で結論一致した — これは独立検証か、それとも盲点の重複か
**結論**: **盲点の重複である**。A/C は同じ「ボラ分布」レイヤーだけを見ており、その下にある **PoC 設計そのものの論理倒錯** と **亡き者データの不継承** を見ていない。

## 1. 発見した構造バグ (5 件)

### バグ A: PoC の「目的」が二重定義され、検証単位が定まっていない

**事実**:
- `src/spec_v2/signal_v2.py` L1: 「§ 3-1 確定までのプレースホルダ」
- `src/spec_v2/signal_v2.py` L21-22: 「PoC の目的は SeasonalDetector が現実の相場で意味を持つかの検証であり、シグナル自体の最適性は本フェーズの問いではない」
- `src/spec_v2/poc_loop.py` L1-13: 「SeasonalDetector で季節判定 → 新規エントリー → SL/TP 設定 → DB に記録」
- `STATUS.md` L33-35: 「戦略: SeasonalDetector (M15 YZ_vol > 30%ile + H1 YZ_vol > 0.00175 二層判定)」「最大保持 4 時間 / 1 ポジション制限」

**断定**: これは **設計バグである**。「シグナルロジックは仮値」+「経済性は問わない」と宣言しているが、`poc_loop.py` の実装は「SeasonalDetector が VOLATILE と判定したら placeholder シグナルで実発注する」になっている。つまり:
- SeasonalDetector の発火頻度を観察する (= 機能 PoC)
- placeholder シグナルで「経済性が分かる」と読み手に誤読される (= 経済性 PoC)
- 0 発火が「閾値が高すぎる」ことの証拠と読み手に誤読される (= 閾値感度 PoC)

の **3 つの PoC が混在している**。Agent A は (3) の問いに答え、Agent C は (1) を擁護して (2) を保留にしているが、**どの PoC を主としているかが誰も明示できていない**。

**証拠**: `data/fx_spec_v2.db` (本ローカル checkout) は seasonal_judgments テーブルに **0 件** (今回独自に SQLite 直接 query で確認、表示クエリは末尾 § 6 に明記)。Agent A の参照する 7,834 件は VPS DB (`C:\bpr_lab_spec_v2\sandbox\FX自動取引` worktree) に存在。ローカルとリモートで「PoC の DB」が物理分離している。**ローカル開発者が「PoC が何を発火させたか」を確認するには VPS にしか正解がない**。

### バグ C: PoC の 4 時間損切りは亡き者 GBP_JPY の挙動と構造的にミスマッチ

**事実 (data/fx_trading_prod_snapshot.db 実測、N=37)**:

| 保持時間 | 件数 | 比率 |
|---|---|---|
| ≤ 30 分 | 0 | 0.0% |
| 30-120 分 | 0 | 0.0% |
| 120-240 分 | 22 | 59.5% |
| 4-8 時間 | 14 | 37.8% |
| > 8 時間 | 1 | 2.7% |
| **中央値** | **234.4 分 (3.9 時間)** | — |
| **平均** | **257.6 分 (4.3 時間)** | — |
| **最大** | **555.2 分 (9.3 時間)** | — |

**断定**: `src/spec_v2/poc_loop.py` L61 `POC_MAX_HOLDING_MINUTES = 240` は **設計バグである**。
- 亡き者の GBP_JPY 実取引の **中央値 234.4 分** は 240 分とほぼ重なる
- **40.5% (15 件 / 37 件)** が 240 分を超えており、PoC 設計下なら時間損切りで強制決済される
- 平均 257.6 分 > 240 分なので、**平均的取引すら完走しない**

これは「亡き者の挙動データを SPEC v2 PoC に反映していない」(継承漏れ) という証拠の二つ目。`POC_MAX_HOLDING_MINUTES` の根拠は `poc_loop.py` L1-13 のドキュメントにも、`SPEC_v2.md` のどこにも書かれていない。M15 16 本 = 4 時間 という「キリのいい数字」が選ばれているように見える。

**示唆**: 仮に PoC が発火したとしても、亡き者と同じ取引時間分布を持つなら **約 4 割が時間損切りで切られる** → 経済性検証として歪んだ結果になる。これは A/C の議論 (閾値が発火するかしないか) よりはるかに **下流の致命的バグ**。

### バグ D: SPEC v2 ロードマップに経済性検証ステップが存在しない

**事実 (STATUS.md L82-103 のテーブル)**:

```
  本番投入 🔴 禁止
```

**断定**: これは **設計バグであり、PREMISE.md L62「数字を使わない」の歪曲適用である**。
- 「期待 PnL」「Sharpe」「機会費用比較」「最大ドローダウン期待値」のステップが SPEC v2 ロードマップに存在しない
- 「実用性検証 WFA」 (Step 4) は閾値 ×{1.0, 1.5, 2.0, 2.5} で usable fold 数を測ったが、これは **閾値選定の頑健性検証** であり **経済性検証ではない**
- `STEP_C_NEW_P0_VERIFIED_SUMMARY.md` を grep してもキーワード「sharpe」「機会費用」「expected return」「profit factor」は SPEC_v2.md L69 で「PF 置換は Bailey-López de Prado deflate 処理が **未実施**」と明記され、残課題扱い
- 結果: 「統計的妥当性 (Permutation + WFA) を獲得した瞬間、次のフェーズ = ペーパートレード = PoC = 本番投入準備」と直列化されており、**統計的に有意な戦略が経済的にも採算 ペイ するかは検証されないまま PoC に降りる**

**最も恐ろしい盲点**: もし PoC で SeasonalDetector が発火し、placeholder シグナルが取引を発生させた場合、その結果が「採算 ペイ するか」「機会費用に勝つか」を判定するメカニズム自体がない。「PnL が +/-」だけが指標になる。

## 3. 亡き者の GBP_JPY 実取引 vs SPEC v2 PoC 採用の整合性検証

**断定**: SPEC v2 § 2-1 確定 (2026-05-10) は、**わずか 3 日前 (2026-05-07) に亡き者の世界で「撤退決定」した同じ通貨ペアを「主候補 ★★★★★」として再採用** している。これは:

- (a) **亡き者の挙動データを新しい SPEC v2 検証に反映していない** (継承漏れ) → 部分的に成立 (PR レベルの撤退判断は反映されていない)
- (b) **「亡き者」と「再構築」の論理的境界が機能していない** → 完全成立 (戦略 / パラメータ / ペア構成は別物だが、ペア選定の論理だけは亡き者から逆方向に再採用されている)

**両方が成立している**。

### 弁護論への反論

「亡き者は **MTFPullback** で負けただけで、**SeasonalDetector + breakout** なら勝てる可能性がある」という弁護は成立するか:
- 成立条件: GBP_JPY の通貨ペアとしての特性 (流動性 / スプレッド / ボラの分布) が SeasonalDetector に合致することを別途検証する必要がある
- 現状: SPEC v2 で行われているのは Permutation Test、WFA、Spearman ρ などの **指標 - リターン関係** の統計検証であって、**通貨ペアとしての交易環境** (スプレッド、スリッページ、約定力) の検証ではない
- 亡き者の 37 件は通貨ペアとしての交易環境を実証する **唯一のサンプル** であり、そこで PF=0.868 を記録した

→ **「戦略が違うから別の話」と切り捨てるのは、亡き者から学ぶべきメタな教訓 (ペア固有のスリッページ実測など) を捨てることになる**。これは PREMISE.md L50「スリッページ実測の重要性」継承条項に反する。

## 7. 結論

A/C は **「ボラ分布レイヤー」では正しい仕事をした**。しかし「現状の閾値 0.00175 は妥当」「観察期間延長」という結論は、**そのレイヤーの外にある 5 つの構造バグを温存する効果を持つ**。

最大の盲点は次の 1 つに集約される:

> **亡き者の世界で 11 日前に撤退決定した通貨ペア (GBP_JPY, PF=0.868, -1,239 JPY) を、その挙動データを参照しないまま、SPEC v2 で「主候補 ★★★★★」に再採用している**

これは「過去は捨てた、教訓だけ持つ」(PREMISE.md L105) の **歪曲適用** である。捨てるべきは戦略・パラメータ・物語であって、**実取引データ (数値) は反証材料として保持すべき**だった。

A/C の「観察期間延長」結論は、この構造バグを **時間で溶かす** 提案になっている。観察を続けても、亡き者の 37 取引が SPEC v2 採用判定に組み込まれることはないし、4 時間時間損切りが亡き者 40.5% を切ることも変わらない。

**第 3 視点としての本ドキュメントの主張**: **観察を続ける前に、PoC 設計とロードマップを修正せよ**。具体的には § 5 の P1 〜 P3 を 1 週間以内、P4 〜 P6 を 1 か月以内に実施。それまで PoC は judgment_only モードで動かし、placeholder シグナルでの実発注は **凍結** すべきである。
