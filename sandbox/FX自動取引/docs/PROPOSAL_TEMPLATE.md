# 戦略プロポーザル評価フレーム

> SPEC v2 撤退 (2026-05-26) を受け、**手法選定が先・フレーム整備は後** で再起動する。各候補手法はこのテンプレートで提出され、Gate 0 を通過したものだけが Phase 2 (精査 BT) へ進む。

---

## 採点ルール

### Gate 0 — 絶対必須 (未充足は門前払い)

**G0-A. PF 0.95 を上回る成果を出せるか**

PoC 撤退の直接原因は signal_v2 (ATR breakout) で過去2年 BT が PF = 0.95 だったこと。**いかなる手法も、まず過去5年 (またはそれ相当) で PF > 0.95 を示すか、示せる理論的根拠を持つこと**。曖昧な期待値表明は不可。

提出時に必答:
- [ ] 簡易 BT で過去5年累計 PF を提示 (スプレッド差し引き後)
- [ ] PF > 0.95 を示せる場合: BT スクリプトと結果テーブルを添付
- [ ] PF > 0.95 を示せない場合: 「**なぜそれでも採用に値するか**」を別途論証 (例: シャープレシオは高い、相関の薄い分散源として有用 等)

**G0-B. 放置してても自己改善ができるか**

「庭師 × 生命体」(STORY.md) の北極星に従い、戦略は **人手介入なしで市場変化に追従できる仕組み** を持つこと。

提出時に必答:
- [ ] 市場ドリフト検出の仕組み (例: ローリング統計の閾値超過、KL ダイバージェンス、レジーム HMM)
- [ ] パラメータ自動再最適化の方式 (例: ローリング再学習、ベイズ最適化、強化学習)
- [ ] 自己改善の **トリガ条件** と **頻度** (例: 月次 / 直近100トレードで PF<1.0 検出時)
- [ ] 改善失敗時のフォールバック (例: 直前安全パラメータへ戻す、停止)

未充足の場合: 採用却下。**「最初に作って終わり」の戦略は今後採用しない** (SPEC v2 PoC で同じ罠を踏んだため)。

### Gate 1 — 重要 (各10点、合計70点で評価)

| 項目 | 配点 | 評価軸 |
|---|---|---|
| **G1-1. 想定エッジ源** | 10 | なぜ儲かるか — 構造的優位 (microstructure / 行動経済 / 統計的非効率) が明示されているか。「過去データで効いた」だけは弱い |
| **G1-2. データ要件** | 10 | 必要データの可用性 (MT5 / yfinance / 外部 API)、コスト、ラグ |
| **G1-3. 実装複雑度** | 10 | 工数見積もり、依存ライブラリ、外部 API 依存度。低複雑度を加点 |
| **G1-4. ロバスト性** | 10 | パラメータ感度 (±20% 変動で PF が大崩れしないか)、市場レジーム変化耐性 |
| **G1-5. リスクプロファイル** | 10 | 想定 MaxDD、損切り設計、テールリスク (Black Swan 耐性) |
| **G1-6. 機会費用比較** | 10 | 米国債 4% / 全世界株 8% / 銀行預金 0.05% との比較。年間期待 PnL を機会費用換算 |
| **G1-7. WFA / OOS 再現性** | 10 | Walk-forward、In-sample/Out-of-sample で PF が大きく落ちないか。Deflated Sharpe |

### Gate 2 — 差別化要因 (各5点、合計30点で評価)

| 項目 | 配点 | 評価軸 |
|---|---|---|
| **G2-1. スプレッド/スリッページ耐性** | 5 | コスト 2x で PF が 1.0 を割らないか |
| **G2-2. 他戦略との相関** | 5 | 既存戦略 (もしあれば) と低相関で分散源になるか |
| **G2-3. 説明可能性** | 5 | 取引判断の根拠を人間が後から検証できるか (LSTM ブラックボックスは減点) |

| **G2-5. 拡張性** | 5 | ペア追加・タイムフレーム変更・規模拡大が容易か |
| **G2-6. 過去挙動データとの整合** | 5 | 亡き者の世界の実取引データ (`data/fx_trading_prod_snapshot.db`) と矛盾しないか (例: 亡き者で完全に負けたパターンを再現していないか) |

## プロポーザル提出フォーマット

各候補は以下のセクションを埋めて1つの Markdown ファイルとして提出:

```markdown
# プロポーザル: <戦略名>

## 1. 戦略仮説 (1段落)
何を、なぜ、どのように。

## 2. 想定エッジ源 [G1-1]
構造的優位の所在。

## 3. シグナル定義 (擬似コード)
入力 → 判定 → 出力。

## 4. データ要件 [G1-2]
必要データ・取得元・コスト・ラグ。

## 5. リスクモデル [G1-5]
SL/TP 設計、ポジションサイジング、想定 MaxDD。

## 6. 自己改善メカニズム [G0-B] — 必須
ドリフト検出、再学習、フォールバック。

## 7. 過去 BT 結果 [G0-A] — 必須
- 期間、ペア、簡易 BT スクリプト
- 累計 PF、シャープ、MaxDD、Trade件数
- スプレッド・スリッページ込みの数字
- PF > 0.95 を超えるか or 超えない場合の論証

## 8. WFA / OOS [G1-7]
ロバスト性検証。

## 9. 実装複雑度 [G1-3]
工数、依存、外部 API。

## 10. 機会費用比較 [G1-6]
米国債/株式/預金との対比。

## 11. リスク・既知の弱点
正直に列挙。

## 12. 採点自己評価
G0/G1/G2 各項目の自己採点。
```

## 進行ルール

- 各プロポーザルは1ファイル: `docs/proposals/<short_name>.md`
- 評価は反論屋3エージェント (karen / ultrathink / pragmatist) を再活用して並列レビュー
- **同じデータの別角度ではなく、別データの別観点で評価** (Ultra 教訓、CONTRARIAN_ULTRA.md 参照)
- 採用判断は人間 (= ユーザー) が最終決定。AI は採点と推薦まで

---

## 不採用基準 (即時却下)

以下のいずれかに該当する候補は Gate 0 PASS でも採用却下:

1. **placeholder で PoC を回す設計** (SPEC v2 の二の舞)
2. **過去の挙動データを反証材料として継承しない設計**
3. **撤退条件が事前に書かれていない**
4. **「動かしてみないとわからない」を含む論証**
5. **複雑度が著しく高く、6ヶ月以内に PoC 到達できないもの**

## メタ — このテンプレート自体について

- このテンプレートは **生きた文書**。プロポーザル評価を通じて改訂してよい
- 採点項目の不在で見落としが発生したら、新項目を追加して既存プロポーザルを再採点する
- 「点数を取るためのプロポーザル」が出てきた場合 (Goodhart の法則)、項目を見直す
