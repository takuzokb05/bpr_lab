## 2. うまくいったこと

### サブエージェント並列実装
- F4/F5/F6+F7 を3並列で同時実装 → 大幅な時間短縮
- 依存グラフを事前に分析して並列化可能ポイントを特定したのが効いた

### 中間レビュー（2回実施）
- code-review + error-handling-audit を2並列で実施
- 1回目: Critical 3 + High 7 を検出・全件修正
- 2回目（再レビュー）: C0/H0 を確認
- 「修正→再レビュー」の2段階が品質担保に有効だった

### 安全設計の徹底
- KillSwitch 6種 + ドローダウン5段階 + SL/TP必須設計
- カオステスト6シナリオ全合格で安全性を実証
- 「安全チェックでの例外 = 取引停止に倒す」という原則を全コードで貫徹

### BrokerClient抽象化レイヤー
- F2 で抽象インターフェースを先に定義したことで、ブローカー変更に対応可能な設計になった
- 実際に OANDA → MT5 への切替が必要になったが、抽象レイヤーのおかげで risk_manager/strategy は変更不要

## 3. 課題・想定外だったこと

### OANDA API アクセス制限
- **想定**: デモ口座で自由にAPI利用可能
- **現実**: OANDA証券は Gold会員+プロコース+25万円残高 が必要。OANDA Global は日本居住者不可
- **影響**: Phase 2 のブローカーを変更する必要あり
- **教訓**: ブローカーのAPI利用条件は Phase A（調査）時点で実際にアカウント作成して確認すべきだった

### 戦略のパフォーマンス
- **想定**: シンプルなMA+RSI戦略でもある程度の収益性（SR > 1.0）
- **現実**: 5ペア x 3TF = 15パターンで SR > 1.0 達成ゼロ。DD < 20% は全パス
- **解釈**: これは想定内。Phase 1 の目的はインフラ構築であり、戦略の収益性は Phase 2 以降の課題。リスク管理が正しく機能している証拠として DD の安全性が全パターンで確認できた

### バックテスト結果のキー名
- `_extract_metrics()` の返却キー名（`equity_final`, `in_sample`, `out_of_sample`）がコード外からすぐに分からず、実行時エラーで判明
- **教訓**: 公開メソッドの返却値の型は docstring かテストで明示的に文書化すべき
