## 致命的な脆弱性

### 1. OANDA一極集中 -- 単一障害点（Single Point of Failure）の設計

**該当箇所**: セクション2.1, 3.1, 7.1

**主張**: 「OANDA REST API v20が個人開発者に最適」「OANDA API v20を採用」

**反論**:

doc 04の全アーキテクチャはOANDAに完全に依存している。データ取得、注文執行、ペーパートレード、リアル取引の全てが単一ブローカーに紐づく。doc 02自身が「API接続断 > 30秒で全ポジション決済」というキルスイッチを設計しているが、これは**API接続断が致命的であることを認識しながら、その単一障害点を解消していない**ことを意味する。

具体的なリスク:
- OANDAが日本向けAPI条件を変更した場合（Gold会員の条件引き上げ等）、システム全体が機能停止する
- OANDAのスプレッドが競争力を失った場合、戦略の前提が崩れるが移行コストが膨大
- doc 02のセクション6.3では「バックアップブローカーへのフェイルオーバー」を推奨しているにもかかわらず、doc 04のアーキテクチャにはフェイルオーバー設計が存在しない
- Interactive Brokersはスプレッドが平均0.1 pipsとOANDAの1.0〜1.2 pipsに対して10分の1であり、取引コストの観点からもOANDA一択は最適解ではない可能性がある（[Benzinga比較](https://www.benzinga.com/money/oanda-vs-interactive-brokers)）

**暗黙の前提**: 「OANDAは将来にわたって安定的にAPIサービスを提供し続ける」

**前提が崩れるシナリオ**: OANDAの日本法人が事業縮小・撤退する、API利用条件が大幅に変更される、あるいは競合他社がより優れたAPI環境を提供し始める場合。実際、OANDAは2026年に貴金属取引のレバレッジ・注文サイズを突然変更した実績がある（[Finance Magnates](https://www.financemagnates.com/forex/brokers/record-silver-price-and-volatility-force-oanda-japan-to-slash-leverage-order-sizes/)）。

**確信度**: 高

**改善提案**: アーキテクチャにブローカー抽象化レイヤーを設け、`oanda_client.py`を`broker_client.py`のインターフェース実装として設計する。Phase 2の段階でInteractive Brokersの接続テストを実施し、フェイルオーバー経路を確保すべきである。

### 2. ペーパートレード「最低3か月」の根拠不在

**該当箇所**: セクション1, 6.2, 8（Phase 2）

**主張**: 「ペーパートレード（最低3か月）」「Phase 1で3か月のペーパートレード完了」が次フェーズへの前提条件

**反論**:

「3か月」という数字の根拠がdoc 01〜03のいずれにも示されていない。これは「なんとなく妥当そうな期間」を設定しただけの可能性が高い。

バックテストの統計的妥当性に関する研究では、**最低200〜500トレード**かつ**複数の市場レジーム（上昇・下降・レンジ）をカバー**することが信頼性の基準とされている（[BacktestBase](https://www.backtestbase.com/education/how-many-trades-for-backtest)）。移動平均クロスオーバー戦略（MA20/50）のシグナル頻度を考えると、3か月で得られるトレード数は極めて限定的である。日足ベースでMAクロスが発生する頻度は、1通貨ペアあたり年に数回〜十数回程度であり、3か月では5回未満のトレードしか発生しない可能性がある。

さらに問題なのは、3か月という期間が**1つの市場レジームしかカバーしない**可能性が高いことだ。たとえば2026年Q1がトレンド相場だった場合、レンジ相場やボラティリティ急上昇局面での戦略の挙動は全く検証されていないことになる。

加えて、ペーパートレードからリアル取引への移行に関する研究では、「ペーパー環境で長時間過ごすことが必ずしも最善の方法ではない」との知見もある（[TradersPost](https://blog.traderspost.io/article/paper-trading-vs-live-trading-key-differences-and-what-to-expect)）。75%のトレーダーが60日以内にライブに移行しており、ペーパーでの成功体験が実取引の心理的現実と乖離するリスクもある。

**暗黙の前提**: 「3か月あれば十分な数のトレードと市場環境の変化を観察できる」

**前提が崩れるシナリオ**: 3か月間が一方的なトレンド相場で、レンジ相場での挙動が未検証のままリアル移行し、相場環境が変わった途端に連続損失を出す。

**確信度**: 高

**改善提案**: 期間ではなく**トレード数**と**市場レジームの多様性**を基準にすべきである。具体的には「最低100トレード」かつ「トレンド相場とレンジ相場の両方を含む」を完了基準とする。または、4時間足・1時間足でのシグナル頻度を上げることで検証速度を加速する設計を検討すべきである。

### 3. 成功指標の甘さ -- シャープレシオ0.5は「成功」と呼べるのか

**該当箇所**: セクション10.1, 10.2

**主張**: 「シャープレシオ > 0.5」を合格基準、「月次損益マイナスでなければ合格」をPhase 3基準とする

**反論**:

シャープレシオ0.5は、機関投資家の基準（2.0以上）はもちろん、リテールトレーダーの一般的な基準（0.75以上）すら下回っている（[QuantStart](https://www.quantstart.com/articles/Sharpe-Ratio-for-Algorithmic-Trading-Performance-Measurement/)）。doc 04自身が「利益を出すことより、資金を守ることを最優先に設計する」と宣言しているが、シャープレシオ0.5は「資金を守れているかどうかも怪しい」水準である。

さらに問題なのは、Phase 3の成功基準「月次損益マイナスでなければ合格」である。この基準には以下の致命的な欠陥がある:

1. **インフレ・機会コストの無視**: 月次+0.01%でも「合格」だが、インフレ率を下回るリターンは実質的な資産減少
2. **運用コストの無視**: 月1,000〜8,000円のインフラコストが発生しているため、トレーディング収益がプラスでも総合収支はマイナスになりうる
3. **時間コストの無視**: 開発・運用に費やす自分の時間の機会コストが全く考慮されていない
4. **ベンチマークの不在**: 同期間のインデックス投資（S&P500、TOPIX等）と比較していない。単に「マイナスでない」ことは、パッシブ投資に対する優位性を何も示さない

doc 03が指摘する「個人トレーダーの80%が損失」という現実を踏まえると、「負けなければ合格」は**相対的には悪くない**が、システム開発にかかる膨大な時間と労力に対する成果としては不十分である。

**暗黙の前提**: 「まず負けないことが大事で、利益は後からついてくる」

**前提が崩れるシナリオ**: 6か月以上の開発・検証期間を経て、月次+0.1%（年率1.2%）の微小利益しか出ない場合、インデックスETFへの積立投資のほうが遥かに効率的だったと判明する。

**確信度**: 高

**改善提案**: 成功指標にベンチマーク比較を追加すべきである。具体的には「シャープレシオ > 1.0」「年率リターン > 同期間のリスクフリーレート + 3%」「総合収支（トレード損益 - インフラコスト - 時間の機会コスト）がプラス」といった基準を設定すべきである。

### 4. リスク管理の「過信」 -- リスク管理があれば安全という幻想

**該当箇所**: セクション4全体, 設計思想全体

**主張**: 「リスク管理を最優先アーキテクチャとして設計」「リスク管理レイヤーは戦略レイヤーの上位に位置する」

**反論**:

doc 04のリスク管理設計は教科書的には正しい。しかし、**リスク管理機構の存在自体が安全を保証するわけではない**。doc 02が詳細に記述したスイスフランショック（2015年）の教訓は、「ストップロス注文があっても、流動性がなければ約定しない」ことだった。5,382 pipsのスプレッド拡大の前では、どんな精緻なリスク管理ロジックも無力である。

doc 04は以下のシナリオに対する設計が不十分である:

1. **APIレベルのストップロスがギャップで約定しないケース**: 「サーバーサイドSL」はスイスフランショック級のイベントでは意味をなさない。流動性プロバイダーが全てクォートを撤回した場合、サーバー側のSLも約定しない
2. **キルスイッチ自体の障害**: キルスイッチのコードにバグがある場合の対処。Knight Capital事件はまさにこのパターンである
3. **20%ハードリミットの前提**: 「最大ドローダウン20%で全停止」と設計しているが、ギャップダウンで一瞬で30%を超える可能性がある。その場合、20%での停止は間に合わない
4. **リスクパラメータの相互作用**: 1%ルール、ドローダウン制御、日次損失上限、連続負け制限が同時に作用する場合の挙動が未定義。これらが矛盾する状況（例: ドローダウン10%で「ポジションサイズ半減」だが、1%ルールでは通常サイズが許可される）での優先順位が不明確

最も危険なのは、精緻なリスク管理設計が**「自分は十分に守られている」という過信**を生むことだ。doc 02のセクション8.Aが指摘する「人間がシステムを止める権限を持っている」リスクは、リスク管理への過信によってむしろ増幅される。つまり、「こんなに堅牢なリスク管理があるのだから、少しレバレッジを上げても大丈夫だろう」という心理が働くリスクがある。

**確信度**: 高

**改善提案**: リスク管理の「限界」を明示的にドキュメント化すべきである。「このシステムで防げないリスク一覧」を作成し、ブラックスワンイベント時の行動マニュアル（全ポジションの手動決済手順、ブローカーへの電話番号等）を準備すべきである。さらに、キルスイッチ自体のテスト（カオスエンジニアリング的アプローチ）をPhase 2の完了基準に追加すべきである。

## 注意すべき弱点

### 5. 「シンプルな戦略から始める」は合理的か -- MAクロスオーバーの逆説

**該当箇所**: セクション5.1, 5.2

**主張**: 「最もシンプルな戦略から始めて、リスク管理の動作を検証する」「この戦略自体で利益を出すことが目的ではない」

**反論**:

MAクロスオーバー（20/50）は確かにシンプルだが、学術研究では**統計的に有意なリターンを生まない**ことが繰り返し示されている。Lo and MacKinlay（1988）の研究ではSMAクロスオーバー戦略がFX市場で低パフォーマンスだったことが報告されており、2024年の研究でも移動平均クロスオーバーがパッシブ投資を上回るという統計的に有意なエビデンスは見つかっていない（[SSRN](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=2578302)、[ejournal](https://ejournal.lincolnrpl.org/index.php/ajmt/article/download/103/138/412)）。

「リスク管理の動作検証が主目的」と言うが、これには矛盾がある:
- **負ける戦略でリスク管理の「有効性」を検証できるのか?** リスク管理が正しく動作して損失を制限しても、元の戦略がマイナス期待値なら、リスク管理は「損失を遅らせている」だけであり、「資金を守っている」のではない
- ドローダウン制御やキルスイッチの発動テストが目的なら、**意図的に負ける戦略を使ってストレステストを行う**ほうが効率的
- Phase 1〜2で4〜5か月を費やしてMAクロスが「使えない」という当然の結論に達するのは、時間の浪費では?

**代替案**: 最初からRSIフィルターを含むやや高度な戦略で開始し、リスク管理のテストは別途ユニットテストとシミュレーションで行う。あるいは、松井証券の1通貨単位取引（約100円）でリアル環境のテストを早期に開始し、ペーパートレード期間を短縮する。

**確信度**: 中

### 6. コスト見積もりの楽観性

**該当箇所**: セクション7.3

**主張**: 月1,000〜8,000円、初期投資は口座25万円のみ

**反論**:

この見積もりは以下のコストを過小評価または無視している:

| 見落とされたコスト | 推定額 | 備考 |
|---|---|---|
| 開発者の時間コスト | 計算不能（数百時間） | Phase 1-3で少なくとも6か月の開発・運用 |
| VPSの24時間稼働 | 月2,000〜5,000円 | 「さくらVPS」月1,000円は最低スペック。監視・通知を含むと不足 |
| LINE Notify / Discord通知 | 月0〜1,000円 | 無料枠超過の可能性 |
| バックテストデータの品質向上 | 月0〜数千円 | OANDAの無料データはティックデータではない |
| 緊急時の手動介入コスト | 不明 | 深夜・休日のアラート対応に伴う生活コスト |
| 税務申告の手間 | 年1回 | FX利益は申告分離課税。確定申告の工数 |
| スプレッドコスト | 変動 | doc内のコスト見積もりにスプレッドが含まれていない |

特に重要なのは**スプレッドコスト**である。OANDAのUSD/JPYスプレッドは約1.0〜1.2 pips（[BrokerChooser](https://brokerchooser.com/compare/interactive-brokers-vs-oanda)）であり、Interactive Brokersの0.1 pipsと比較して10倍高い。取引回数が増えるほどこの差は累積し、戦略の期待値を大きく毀損する。

さらに、doc 03が指摘した「小規模な口座（100万円以下）では、インフラコストだけでリターンを上回る可能性がある」という警告が、doc 04のコスト見積もりには反映されていない。25万円の口座で月1%（2,500円）のリターンを目指す場合、インフラコスト月3,000〜5,000円で即座に赤字である。

**確信度**: 高

**改善提案**: コスト見積もりに「損益分岐点分析」を追加すべきである。「月次何%のリターンが必要か」を口座サイズ別に明示し、最低必要口座サイズを算出すべきである。

### 7. Backtesting.py の選択 -- Phase 4への移行困難

**該当箇所**: セクション6.1, 7.1

**主張**: Phase 1でBacktesting.pyを採用、Phase 4でVectorBTを検討

**反論**:

Backtesting.pyは「軽量、学習コスト低い」と評価されているが、以下の重大な制約がある:
- **マルチアセット非対応**: 複数通貨ペアの同時バックテストができない（[GitHub Issue #1120](https://github.com/kernc/backtesting.py/issues/1120)）。Phase 4の「複数通貨ペア対応」と根本的に矛盾する
- **ポジションサイジングの制約**: buy/sellメソッドが「全資金を使用」する設計であり、口座の1%リスクルールとの整合性が取りにくい（[GitHub Issue #10](https://github.com/kernc/backtesting.py/issues/10)）
- **OHLCVデータのみ対応**: センチメントスコアや追加のファンダメンタルデータを組み込みにくい

Phase 1でBacktesting.pyに習熟し、Phase 4でVectorBTに移行するとなると、**バックテストコードの全面書き直し**が必要になる。これは「シンプルから始める」の代償として、将来の技術的負債を積み上げていることに他ならない。

**代替案**: 最初からBacktraderまたはVectorBTを採用する。学習コストは確かに高いが、Phase 4への移行コストを考えると、トータルの工数は削減される可能性がある。

**確信度**: 中

### 8. AIを「使わない」判断の部分的な不合理性

**該当箇所**: セクション5.3, 9

**主張**: 「AIによる価格予測は使わない」「リスク管理をAIに任せない」

**反論**:

「価格予測にAIを使わない」は、doc 03の調査結果に基づく合理的な判断である。しかし、「リスク管理をAIに任せない」は検討が不十分な可能性がある。

doc 04のリスク管理は全てルールベースだが、ルールベースには以下の限界がある:
- **固定閾値の硬直性**: ATR > 通常の3倍でキルスイッチ発動、という固定ルールは、市場レジームの変化に適応しない。2024-2025年のボラティリティレジーム変化（doc 03 セクション7.3）を考えると、「通常」の定義自体が変わりうる
- **相関の動的変化**: 最大相関通貨ペア同時保有数を2に固定しているが、通貨間相関は市場ストレス時に急上昇することが知られている。危機時にはEUR/USD、GBP/USD、AUD/USDが全て同方向に動き、実質的に1つのポジションと化す
- **異常検知の可能性**: MLベースの異常検知（Isolation Forest等）は、人間やルールベースでは見落とす市場の構造変化を検出できる可能性がある

「ブラックボックスにしない」という原則は理解できるが、ルールベースのリスク管理を**主**とし、ML異常検知を**副**（アラート生成のみで自動行動はしない）とするハイブリッドは検討に値する。

**確信度**: 中

### 9. SQLiteの選択 -- 取引ログの長期運用に耐えうるか

**該当箇所**: セクション7.1

**主張**: 「データ保存: SQLite。取引ログ・バックテスト結果。軽量で十分」

**反論**:

SQLiteは同時書き込みに弱く、外部クライアントからの接続ができない。Phase 2以降で以下の問題が顕在化する可能性がある:
- リアルタイムの価格データ書き込みと、バックテスト結果の読み取りが同時に発生した場合のロック競合
- 取引ログの分析をJupyterから行いつつ、取引システムが稼働中の場合のファイルロック
- 数年分のティックデータ蓄積時のクエリパフォーマンス低下

時系列データの保存にはTimescaleDB（PostgreSQL拡張）のほうが適しており、将来の移行コストを考慮すると、最初からPostgreSQLを採用しておくほうが安全である（[db-engines比較](https://db-engines.com/en/system/SQLite%3BTimescaleDB)）。

ただし、Phase 1の規模ではSQLiteで十分であるため、これは「将来を見据えた選択か、現在の効率を重視した選択か」のトレードオフである。

**確信度**: 低

| 5 | 月1,000〜8,000円で運用できる | セクション7.3 | スプレッドコスト、時間コスト、VPSスペック不足を過小評価 |
| 6 | Phase間の移行がスムーズに行える | セクション8全体 | Backtesting.py→VectorBTの全面書き直し、戦略の高度化による複雑性爆発 |
| 7 | 口座25万円で意味のある検証ができる | セクション7.3 | インフラコストがリターンを上回り、損益分岐に達しない |

## 見落とされた代替案

### 代替案1: 既存サービスの活用を先に試す

doc 01で調査された松井証券の自動売買（1通貨単位、手数料無料）や、トライオートFXのビルダー機能を**先に試し、自動売買の感覚を掴んでから自作に移行する**という選択肢が検討されていない。松井証券なら100円から自動売買を始められ、リスク管理の「体感」を得るのに最適である。これにより、Phase 1の1〜2か月を短縮し、自作システムの設計にフィードバックを反映できる。

### 代替案2: TradingView + Webhook + OANDA のハイブリッド

doc 01のパターンCで紹介されたTradingView + Webhookアプローチは、Pine Scriptで戦略を記述し、PythonはWebhook受信と注文実行のみを担当する。これにより:
- TradingViewの豊富なバックテスト環境を活用でき、Backtesting.pyの制約を回避
- 戦略開発とリスク管理/注文実行を明確に分離
- TradingViewコミュニティの既存戦略を活用可能

この選択肢がdoc 04で評価・棄却された形跡がない。

### 代替案3: コピートレード/フォローとの併用

自作システムの開発と並行して、実績のあるシグナルプロバイダーのコピートレードを小額で運用し、「負けないトレーダーの挙動」を研究するアプローチ。これはdoc 03の「意思決定支援」思想と整合する。

### 代替案4: インデックス投資との比較実験

同じ25万円を低コストのインデックスETF（eMAXIS Slim全世界株式等）に投資し、自作FXシステムと6か月間のパフォーマンスを比較する「ベンチマーク実験」を並行実施する。これにより、「自作システムに時間をかける価値があるか」を定量的に判断できる。

## 堅牢な主張

### 1. 「やらないことリスト」の質が高い

**該当箇所**: セクション9

doc 01〜03の調査結果に基づき、明確な根拠とともに「やらない」判断を列挙している点は非常に優れている。特に以下は強固な判断である:
- マーチンゲール系の資金管理を排除（doc 02: 破産リスク）
- 高レバレッジ（15倍以上）を排除（ドローダウンの非対称性）
- バックテストのみでリアル移行を排除（ギャップの大きさ）
- 完全放置運用を排除（大損事例の多さ）

「やること」よりも「やらないこと」を明確にする設計姿勢は、特にリスクの高い領域において極めて重要である。

### 2. リスク管理の多層防御構造

**該当箇所**: セクション4

1トレードリスク制限 → ドローダウン5段階制御 → 日次/週次/月次損失上限 → キルスイッチ6種類という多層防御は、単一の防御線に依存しない堅牢な設計である。各レイヤーが独立して機能するため、1つのレイヤーが失敗しても次のレイヤーが作動する。この「深層防御（Defense in Depth）」の思想は、doc 02の調査結果を忠実に反映している。

（ただし、上記「致命的な脆弱性4」で指摘した通り、この多層防御にも限界がある点は認識すべきである。）

### 3. AI活用における「ハイブリッド運用」の判断

**該当箇所**: セクション5.3

doc 03の調査結果（予測精度と収益性の非相関、レジーム変化による信頼性喪失、取引コストによる収益毀損）を正確に反映し、AIを「予測」ではなく「フィルター」として活用する判断は合理的である。特に「リスク管理をAIに任せない」という判断は、Knight Capital事件（45分で4.4億ドル損失）の教訓を踏まえた健全な判断である。

## 総括

doc 04は全体として、調査に基づく慎重な設計文書であり、多くの個人開発者が陥る「利益最大化への過度な楽観」を回避している点は評価に値する。しかし、**慎重さ自体が生む盲点**が複数存在する。

最も深刻なのは以下の3点である:

1. **OANDA一極集中**: 慎重に選んだつもりが、単一障害点を構造的に抱えている
2. **検証期間と成功基準の甘さ**: 慎重に検証するつもりが、統計的に無意味な検証期間と、意味のない成功指標を設定している
3. **リスク管理への過信**: 慎重にリスク管理を設計したつもりが、「防げないリスク」の明示を怠っている

これらは「設計が間違っている」のではなく、「設計の前提が明示・検証されていない」という問題である。前提を可視化し、前提が崩れた場合のプランBを用意することで、本設計書の質は大幅に向上するだろう。
