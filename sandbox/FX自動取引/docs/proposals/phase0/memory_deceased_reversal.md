# プロポーザル: 亡き者反転戦略 (Deceased Reversal)

## 1. 戦略仮説 (1段落)

亡き者 (MTFPullback 系統) の本番取引データが 6ペア合計 PF=0.81 で構造的に負けていた事実 (`fx_trading_prod_snapshot.db`、64件、-1,180 JPY、勝率23.4%) を反転シグナルとして使う戦略。**「亡き者がエントリーした方向の逆を取る」** = 亡き者の SELL シグナル発生時に LONG、BUY 時に SHORT。エッジ源は (a) 既存戦略の系統的負け = 反転で系統的勝ち、(b) 「平均回帰のシグナル発見」(market wisdom: 退場したルーザーの反対側に立つ)。

## 2. 想定エッジ源 [G1-1]

- **構造的優位**: 亡き者 MTFPullback の負け方が **両方向で対称負け** (GBP_JPY long 27% / short 38% 勝率) → 「順張り pullback」自体に系統エッジが逆方向にある可能性
- **行動経済根拠**: Adverse selection — 「pullback を狙う遅効性参加者」が市場で逆指値刈りされるパターン (Hasbrouck 1991 microstructure adverse selection)
- **データ根拠**: 64件全体で 23.4% 勝率 → 反転すれば 76.6% 勝率の理論期待 (実際には対称ではないので逆転で 50-65% 勝率が現実的)
- **「過去データで効いた」だけではない**: 「亡き者がなぜ負けたか」の構造分析 (pullback タイミングの遅延 = mean reversion で先行く相手に置いて行かれる) が反転エッジの存在を示唆

## 3. シグナル定義 (擬似コード)

```python
# MTFPullback シグナルを"逆方向"で発火させる
def signal_reversal(bar_close, history):
    # 既存 src/strategy/mtf_pullback.py を再利用 (亡き者コード継承、PREMISE.md的に許容)
    raw_signal = mtf_pullback_signal(bar_close, history)

    if raw_signal is None:
        return None

    # 方向を反転
    reversed_direction = "short" if raw_signal.direction == "long" else "long"

    # SL/TP も対称に
    atr = ATR(history, 14)
    if reversed_direction == "long":
        sl = bar_close - atr * 1.5
        tp = bar_close + atr * 2.5
    else:
        sl = bar_close + atr * 1.5
        tp = bar_close - atr * 2.5

    return Signal(direction=reversed_direction, entry=bar_close, sl=sl, tp=tp)
```

**注**: 亡き者コード `src/strategy/mtf_pullback.py` は撤退ブランチに保存されている → 再利用容易

## 4. データ要件 [G1-2]

- **必要データ**: M15/H1 OHLC (既存)
- **取得元**: MT5、無料
- **コスト**: ゼロ
- **新規実装**: 反転ラッパ関数のみ (50 行以内)

## 5. リスクモデル [G1-5]

- **SL**: ATR × 1.5 (亡き者と同等)
- **TP**: ATR × 2.5 (亡き者の RR=2.0 を控えめに RR=1.67 に)
- **時間損切り**: 8時間 (亡き者中央値の 234 分の 2倍を許容)
- **ポジションサイジング**: 口座 0.5% / 取引 (反転戦略の検証段階としてはアグレッシブにせず)
- **想定 MaxDD**: 亡き者と対称想定 → -10〜-15%
- **テールリスク**: 「亡き者が稀に勝った日 (例 2026-04-22 +1,389 JPY)」の反対側で大負けする日が想定 → 月最大DD 上限 -5%

## 6. 自己改善メカニズム [G0-B] — 必須

### ドリフト検出
- **反転 PF 維持監視**: 直近20トレードで反転 PF<1.2 が3連続で warning
- **亡き者勝率変化**: 亡き者システム (=旧ロジック) の最近30件勝率を**並走計算**して仮想モニタ。亡き者勝率が30%超に上昇したら「市場が亡き者に有利化」 → 反転戦略停止
- **相関監視**: 反転戦略 PnL と亡き者仮想 PnL の相関係数。-0.7 を下回ったら反転エッジ消失と判定

### 自動再最適化
- **月次反転判定値再算出**: 過去6か月の亡き者仮想PnLを再算出し、PF<0.95 維持なら反転戦略継続。PF>1.0 に上振れしたら停止
- **パラメータ感度**: 反転は元戦略のパラメータを継承するので、自前のチューニングは不要 (元戦略のパラメータ変動を逆位相で吸収)

### フォールバック
- **累計-3% で半量化、-5% で完全停止**
- **過去 6か月の亡き者仮想モニタを `data/deceased_monitor.json` に記録**
- **亡き者 PF>1.0 復活時**: 「市場が亡き者の戦略を再評価し始めた」シグナル → 反転戦略は退場

### 擬似コード (1段落)
> 毎M15バーで MTFPullback シグナルを計算し、その逆方向に発注。並行して「亡き者仮想 PnL」(MTFPullbackをそのまま使った場合のPnL) を記録。亡き者仮想PnL が 30 trades rolling で PF<0.95 を維持する間は反転継続。PF>1.0 に転じたら自動停止。月次で過去6ヶ月の亡き者仮想を再計算してエッジ寿命を確認。

**月換算 +13,485 JPY、年率 +13.5% @ 100万円 (lot 0.034)** → **PF > 0.95 を大きく超える**

### 重要な留保
本BTは「亡き者の実トレード単位」での反転で、新規シグナル発生頻度は実取引と同等を仮定。**反転戦略を独立に走らせた場合、エッジ消失/拡大の可能性**あり → これは G0-A の論証として保守的なベンチマークと位置付け

## 8. WFA / OOS [G1-7]

- **OOS 検証**: 亡き者の取引は 2026-04-21〜05-01 の連続 11日 → OOS分割は意味なし
- **代替**: 亡き者の過去年データ (MTFPullback シミュレーション) を 2024-2026 で再シミュレートし、反転 PF を年次で測定 → Phase 2 で実行

## 11. リスク・既知の弱点

1. **「亡き者が勝つように変わる」リスク**: 市場レジーム変化で MTFPullback が再び有効になると、反転は系統的負けに転じる → 月次モニタで吸収
2. **EUR_USD 例外管理**: 全ペア機械反転は EUR_USD で大損 → ペア別enabled flag の運用が必要 (複雑度up)
3. **小サンプル**: 6ペア中4ペアは N=2〜3 の極小サンプル → BT結果は偶然の可能性大 → Phase 2 で過去2年シミュレートが必須
4. **既存コードへの依存**: `mtf_pullback.py` を継承するため、亡き者コードの保守が必要 (PREMISE.md「コードベース継承」と整合)
5. **「逆張りで負ける日が大きく出る」尾risk**: 亡き者が +1,389 JPY を出した 2026-04-22 のような日では -1,389 → 月最大DD-5%歯止め必須
6. **哲学的批判**: 「亡き者をパラサイト化しているだけで自前のエッジではない」→ 構造分析でこの批判をかわす論証が必要

## 撤退条件 (事前明記)

1. 30日で trades < 30 件 (亡き者シグナル頻度の低下)
2. 直近20トレードで反転 PF < 1.0 が確定
3. 累計 PnL < -2% (-20,000 @ 100万)
4. 亡き者仮想 PnL が PF>1.0 に転じる (エッジ消失)
5. 機会費用 (米国債4%) を 2ヶ月連続下回り
