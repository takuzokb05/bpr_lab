# プロポーザル: セッション別戦略割当 (Tokyo/London/NY 時間別)

## 1. 戦略仮説 (1段落)

亡き者は `src/session_filter.py` を実装したが、本probe で確認した通り**全ペアで平均保有時間が 200-400分**と長く、セッションを跨ぐ取引が多発していた。FX 市場はセッションごとに参加者・流動性・行動パターンが大きく異なる (Tokyo: クロス円中心、薄商い、明確レンジ / London: 最大流動性、トレンド形成 / NY: 米指標反応、午後ダレ)。本提案は **セッション別に異なる戦略を割り当て、セッション終了時に強制クローズ**することで「跨ぎリスク」を排除する。Tokyo 時間は平均回帰、London 時間はモメンタム、NY 時間は経済指標前後を回避する保守運用、という3戦略構成。**「FXの構造的非対称をそのまま戦略に翻訳」**するシンプルなアプローチ。

## 2. 想定エッジ源 [G1-1]

**構造的優位**:
- Tokyo 時間 (00:00-08:00 UTC) は流動性が低く、レンジ取引が支配 → 平均回帰戦略が有利 (Ito & Hashimoto 2006, "Intra-day Seasonality")
- London 時間 (07:00-15:00 UTC) は世界最大の出来高、トレンド形成期 → モメンタム戦略が有利
- NY 時間 (12:00-20:00 UTC) は経済指標 (NFP, FOMC, CPI 等) の発表が集中、ニュース駆動。**ニュース時間 ±30分は取引停止**で安全運用
- セッションを跨いだ保有は「片方のセッションで儲かっても、もう片方で全部吐き出す」リスクが構造的に高い

**実取引データの裏付け** ([[feedback-deceased-world-data-inheritance]] 遵守):
- 亡き者 prod DB の平均保有時間: EUR_USD 393分、GBP_JPY 258分、USD_JPY 377分 → **全ペアでセッション跨ぎが常態化**
- セッション境界で「逆方向の動き」に飲まれた loss trade が一定数あるはず (Phase 1 で検証宿題)
- session_filter.py が実装されていたのに、ai_decision/ai_regime が全件 NULL = **セッションフィルタが本番パイプラインに繋がっていなかった可能性**

## 3. シグナル定義 (擬似コード)

```python
def generate_signal(data_m15, data_h1, utc_dt):
    hour = utc_dt.hour

    # セッション判定
    if 0 <= hour < 7:
        session = "TOKYO"
    elif 7 <= hour < 12:
        session = "LONDON_AM"  # London のみ
    elif 12 <= hour < 16:
        session = "LONDON_NY_OVERLAP"  # 最大流動性
    elif 16 <= hour < 20:
        session = "NY_PM"
    else:
        session = "QUIET"  # 20-24 UTC、取引停止

    # ニュース時間チェック (経済カレンダー API 必要)
    if is_near_high_impact_news(utc_dt, window_minutes=30):
        return HOLD, "ニュース回避"

    # セッション別戦略
    if session == "TOKYO":
        return BollingerReversal.generate_signal(data_m15)  # レンジ平均回帰
    elif session == "LONDON_NY_OVERLAP":
        return MTFPullback.generate_signal(data_m15, data_h1)  # トレンド押し目
    elif session == "LONDON_AM":
        return ATR_Breakout_Conservative.generate_signal(data_m15)  # 控えめブレイク
    elif session == "NY_PM":
        return BollingerReversal.generate_signal(data_m15)  # 午後ダレ平均回帰
    else:
        return HOLD

# セッション終了で強制クローズ
def on_bar(current_session, position):
    if position and position.entry_session != current_session:
        # セッション跨ぎ → 強制クローズ
        close_position(position, reason="session_boundary")
```

## 6. 自己改善メカニズム [G0-B] — 必須

**ドリフト検出 (毎週)**:
- 直近30日のセッション別 PF/勝率を集計
- 過去365日対比で各セッションの PF が ±0.3 変動でアラート
- セッション境界クローズの平均 PnL (cut short された利益/限定された損失) を監視

**パラメータ自動再最適化 (月次)**:
- 各セッションに割り当てる戦略を入れ替え可能なグリッド
- 例: Tokyo に BR か MTF か Conservative_Breakout か
- ニュース回避時間幅 (±15/±30/±60分) のグリッド
- 直近 12ヶ月 IS / 3ヶ月 OOS で評価

**フォールバック**:
- 新割当で 30日 PF<0.9 → 旧割当リバート
- ニュース API ダウン → 全セッション保守運用に切替
- 連続2回再最適化失敗 → 戦略停止

**トリガー条件**:
- 毎週月曜 06:00 JST ドリフトチェック
- 月初1日 全パラメータ再最適化
- 累計 -3,000 JPY / 90日 trades<15 → 撤退

## 11. リスク・既知の弱点

1. **セッション分割でサンプル過少**: 各セッションで year-30 trade だと評価不能
2. **境界クローズの機会損失**: 利益が乗っている trade を強制決済する形になる
3. **ニュース API 依存**: ForexFactory が止まると全体が止まる
4. **セッション定義の硬直**: 夏時間/冬時間の切替で UTC オフセットが変動、判定エラーリスク
5. **戦略×セッションの組合せ最適化**: 試行数 81 で過剰最適化リスク
6. **24時間連続データ前提**: 週末 (金 22 UTC〜日 22 UTC) の処理が複雑
