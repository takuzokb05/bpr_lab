## 1. エグゼクティブサマリー

doc 01〜03の調査を総合すると、FX自動取引システムの設計方針は以下に収束する:

- **プラットフォーム**: OANDA REST API v20 + Python（doc 01: API比較より。日本国内で最もAPI環境が整備されている）
- **リスク管理**: 1-2%ルール + ATRベース損切り + 5段階ドローダウン制御 + キルスイッチ（doc 02: リスク管理の業界標準）
- **AI活用**: 完全自動ではなくハイブリッド運用。センチメント分析とテクニカル指標最適化に限定（doc 03: AIの限界を踏まえた現実的アプローチ）
- **開発プロセス**: バックテスト → ペーパートレード（最低100トレード + 複数市場レジーム） → 最小ロットでリアル取引 → 段階的拡大

**最重要原則**: 利益を出すことより、資金を守ることを最優先に設計する。

**設計の核心**: リスク管理レイヤーは戦略レイヤーの**上位**に位置する。どんなに優秀なシグナルが出ても、リスク管理が「NO」と言えば取引しない。

**ブローカー抽象化**: データ・実行レイヤーにはブローカー抽象化レイヤーを設ける。初期実装はOANDA REST API v20だが、`BrokerClient` インターフェースを定義し、将来のブローカー追加（Interactive Brokers等）やフェイルオーバーに対応できる設計とする。OANDA一極集中による単一障害点リスク（API条件変更、マージン率変更、ポジション制限変更等の事業条件変更を含む）を軽減するため、Phase 2でバックアップブローカーの接続テストを実施する。

**BrokerClient最小メソッドセット**（Phase 1で定義。過度な汎用化を避ける）:

| メソッド | 責務 | 備考 |
|---------|------|------|
| `get_prices(instrument, count, granularity)` | 価格データ取得 | OHLCV形式 |
| `market_order(instrument, units, stop_loss, take_profit)` | 成行注文 | SL/TP必須 |
| `limit_order(instrument, units, price, stop_loss, take_profit)` | 指値注文 | SL/TP必須 |
| `get_positions()` | 保有ポジション照合 | ローカル状態との整合性確認用 |
| `close_position(trade_id)` | ポジション決済 | キルスイッチから呼び出し |
| `get_account_summary()` | 口座残高・証拠金取得 | リスク計算に使用 |

> **注意**: OANDAはRESTful API（ステートレス）、IBはTWS API（ステートフル・ソケットベース）であり、抽象化の実装コストは高い。Phase 1ではOANDA実装のみとし、IB実装はPhase 2の接続テスト完了後に判断する。

## 4. リスク管理の設計（最優先）

doc 02の調査結果を実装レベルに落とし込む。

### 4.1 リスクパラメータ

```python
# config.py に定義するリスク関連定数

# === 1トレードあたりのリスク ===
MAX_RISK_PER_TRADE = 0.01      # 口座資金の1%
MAX_RISK_PER_TRADE_HARD = 0.02 # 絶対上限2%

# === レバレッジ ===
MAX_LEVERAGE = 10              # 自主制限（法的上限25倍の40%）

# === ドローダウン制御（5段階） ===
DRAWDOWN_LEVELS = {
    0.05: "WARNING",      # 5%: 警告通知
    0.10: "REDUCE",       # 10%: ポジションサイズ半減
    0.15: "MINIMUM",      # 15%: 最小ロットのみ
    0.20: "STOP",         # 20%: 全停止（ハードリミット）
    0.25: "EMERGENCY",    # 25%: 全ポジション強制決済
}

# === 日次・月次損失上限 ===
MAX_DAILY_LOSS = 0.02          # 日次: 口座の2%
MAX_WEEKLY_LOSS = 0.05         # 週次: 口座の5%
MAX_MONTHLY_LOSS = 0.10        # 月次: 口座の10%

# === 連続負け制限 ===
MAX_CONSECUTIVE_LOSSES = 5     # 5連敗で一時停止

# === ポジション制限 ===
MAX_OPEN_POSITIONS = 3         # 最大同時ポジション数
MAX_CORRELATION_EXPOSURE = 2   # 相関通貨ペアの最大同時保有数
```

### 4.2 キルスイッチの設計

```
キルスイッチ発動条件:
├── 損失ベース: 日次損失 > 2% → 当日取引停止
├── 連続負け: 5連敗 → 24時間停止
├── ボラティリティ: ATR > 通常の3倍 → 新規エントリー停止
├── スプレッド: スプレッド > 通常の5倍 → 新規エントリー停止
├── API接続: 接続断 > 30秒 → 全ポジション決済
└── 手動: 管理者が即時停止可能
```

### 4.3 損切り戦略

doc 02の推奨に基づき、**ATRトレーリングストップ**を標準採用:

```
損切り幅 = ATR(14) × 2.0
トレーリング: 利益方向に価格が動いたら損切りラインを追従
```

## 5. 戦略設計

### 5.1 最初の戦略: RSIフィルター付き移動平均クロスオーバー

複雑なAI戦略から始めるのではなく、**シンプルだが統計的に正の期待値を持ちうる戦略で始める**。

```
戦略ロジック:
- 短期MA(20) が 長期MA(50) を上抜け + RSI(14) < 70 → 買いシグナル
- 短期MA(20) が 長期MA(50) を下抜け + RSI(14) > 30 → 売りシグナル
- ATR(14) × 2.0 のトレーリングストップ
- リスクリワード比 1:2 以上のみエントリー
- 4時間足をメインタイムフレームとする（日足は別検証トラックで後段追加）
```

RSIフィルターを追加する理由: 素のMAクロスオーバーは偽シグナル率が57-76%と高い（出典: [SSRN](https://papers.ssrn.com/sol3/Delivery.cfm/5186655.pdf?abstractid=5186655&mirid=1)）。RSIフィルターで買われすぎ・売られすぎ局面での逆方向エントリーを除外し、シグナルの質を向上させる。

**リスク管理の動作検証はカオステスト+ユニットテストを主手段とする**（セクション6.3参照）。ペーパートレードはシステム全体の統合テストと実環境でのスリッページ・スプレッド計測が主目的。

### 5.2 段階的な戦略高度化

| Phase | 戦略 | 追加要素 | 前提条件 |
|-------|------|---------|---------|
| 1 | RSIフィルター付きMAクロスオーバー（4時間足） | RSI(14)で偽シグナル低減 | カオステスト+ユニットテスト合格、バックテストWFE>50% |
| 2 | 複数テクニカル指標 | MACD、ボリンジャーバンド追加 | Phase移行チェックリスト（セクション6.4）全項目合格 |
| 3 | センチメントフィルター | ニュースセンチメントで取引可否を判定 | Phase 2の実績が安定 |
| 4 | パラメータ最適化 | MLによる指標パラメータの自動調整 | 十分なデータ蓄積 |

### 5.3 AI活用の方針（doc 03を反映）

doc 03の結論「完全自動ではなくハイブリッド運用」を設計に反映:

| AI活用領域 | 方式 | 理由（doc 03の根拠） |
|-----------|------|---------------------|
| 価格予測 | **使わない** | 予測精度と収益性が非相関。過学習リスクが高い |
| センチメント分析 | **フィルターとして使う** | FinBERT 84.33%精度。短期トレンド予測を約11%改善 |
| パラメータ最適化 | **補助的に使う** | ウォークフォワード検証を必須とし過学習を抑制 |
| 異常検知 | **Phase 2で検討** | ルールベースの固定閾値が市場レジーム変化に適応しない問題を補完。ただし安全性の最終判断はルールベースを維持 |
| リスク管理の最終判断 | **ルールベース（ML非使用）** | 安全性に関わる部分はブラックボックスにしない |

## 6. バックテストと検証

### 6.1 バックテスト環境

| ライブラリ | 特徴 | 採用判断 |
|-----------|------|---------|
| **Backtesting.py** | 軽量、Pythonネイティブ、学習コスト低い | **Phase 1で採用** |
| VectorBT | 高速（ベクトル化）、大量パラメータ最適化向き | Phase 4で検討 |
| Backtrader | 高機能だが学習コスト高い | 不採用 |

> **注意**: Backtesting.pyはマルチアセット非対応のため、Phase 4（複数通貨ペア対応）ではVectorBTへの移行が必要になる。移行コスト（バックテストコードの書き直し）を認識した上で、Phase 1の学習コスト削減を優先する判断。

### 6.2 検証プロセス

```
1. イン・サンプルテスト（過去3年データの70%）
   └→ 戦略の基本性能を確認

2. アウト・オブ・サンプルテスト（残り30%）
   └→ 未知データでの性能を確認

3. ウォークフォワード分析
   └→ WFE（ウォークフォワード効率）50%以上を合格基準とする
   └→ doc 03: 「イン・サンプルで正のシャープレシオの戦略の
       最大78%がアウト・オブ・サンプルで負に転落」

4. ペーパートレード（最低100トレード、推奨200トレード、かつ市場レジーム条件を満たす）
   └→ OANDA のデモ口座で実行
   └→ スリッページ・スプレッドの実測値を記録
   └→ 4時間足をメインタイムフレームとし、十分なシグナル頻度を確保
   └→ 市場レジーム判定基準（定量）:
      - トレンド相場: ADX(14) ≥ 25 の期間が全検証期間の30%以上
      - レンジ相場: ADX(14) < 25 の期間が全検証期間の30%以上
      - 両レジームの存在をシステムで自動判定し、手動判断に依存しない

5. 最小ロットでのリアル取引（1か月）
   └→ 1通貨 or 最小ロットで実取引
   └→ バックテストとの乖離を定量測定

6. 段階的ロット拡大
   └→ Phase移行チェックリスト（セクション6.4）の全項目合格が前提
   └→ 3か月連続月次プラス かつ 最大ドローダウン基準内で次のロットへ
```

### 6.3 リスク管理の検証手法（カオステスト + ユニットテスト）

リスク管理の動作検証はペーパートレードの副産物に頼らず、**能動的なテストを主手段とする**。

#### ユニットテスト（Phase 1完了時に全合格必須）

| テスト | 検証内容 | 合格基準 |
|--------|---------|---------|
| ポジションサイジング | 口座サイズ変動時の計算精度 | 全ケースで1-2%ルール遵守 |
| ドローダウン制御 | 5段階の閾値で正しいアクション発動 | 全5段階のアクションが正確に発動 |
| 損失上限チェック | 日次/週次/月次の損失上限到達時の停止 | 上限±0.1%以内で発動 |
| 連続負けカウンター | 5連敗で一時停止 | 正確にカウント・停止 |
| レバレッジ制限 | 10倍上限の遵守 | 超過する注文を拒否 |

#### カオステスト（Phase 1完了後、Phase 2開始前に実施）

| シナリオ | 注入する異常 | 合格基準 |
|---------|------------|---------|
| API接続断 | OANDAへの接続を30秒間遮断 | 既存ポジションにサーバーサイドSL設定済みであること。再接続後にポジション照合が成功すること |
| スプレッド急拡大 | スプレッドを通常の10倍に設定 | 新規エントリーが自動停止すること |
| 急激なドローダウン | 口座残高を人為的に15%減少させた状態を注入 | ポジションサイズ25%制限が発動すること |
| 連続損失注入 | 5回連続の損失トレードをシミュレート | 24時間取引停止が発動すること |
| ギャップ注入 | 価格データに5%のギャップを挿入 | サーバーサイドSLが約定すること（ギャップ分のスリッページは許容） |
| 同時多発障害 | API接続断 + スプレッド急拡大を同時注入 | (1) 既存ポジションにサーバーサイドSLが設定済みであること（API未接続時のフォールバック）。(2) API接続復旧後、最初の接続確認で全ポジション決済を試行すること |

> カオステストの結果は全て記録し、合格/不合格を判定する。不合格が1件でもあればPhase 2に進まない。

## 9. このシステムで防げないリスク

リスク管理の多層防御を設計しているが、以下のリスクは**本システムでは防げない**。過信を避けるために明記する。

| リスク | なぜ防げないか | 対処 |
|--------|--------------|------|
| スイスフランショック級の流動性消失 | 流動性がゼロになるとサーバーサイドSLも約定しない | 投入資金は総金融資産の10%以下、かつ6か月分の生活費を除いた余剰資金に限定する |
| ブローカー自体の破綻・出金停止 | システム外の事象 | 日本の信託保全制度に頼る。1ブローカーへの集中は総金融資産の5%以下 |
| キルスイッチ自体のバグ | 安全機構のバグはテストで発見しにくい | Phase 1→2移行前にカオステスト6シナリオ全合格を必須とする（セクション6.3参照） |
| ギャップによる20%ハードリミット突破 | 月曜窓開け等で一瞬で20%超の損失 | 金曜21:00（NY時間）までにポジションを50%以下に縮小するルールを自動実行 |
| ブローカーの事業条件変更 | マージン率変更、ポジション制限変更、レバレッジ制限等はAPI仕様変更とは別次元の問題でコードでは対処不能（2025-2026年にOANDAで銀のマージン率引き上げ、建玉数量変更の実績あり） | ブローカー抽象化レイヤーで影響を局所化 + OANDAの公式ニュース・メール通知を監視し、条件変更時の影響シミュレーションを実施 |
| 開発者のバーンアウト | 深夜アラート対応、長期の微小リターン | 撤退基準を事前に定義（セクション11.2参照）。アジアセッション早朝（5:00-7:00）のアラートは非通知（キルスイッチ発動時を除く）。週末のシステム確認は30分以内に制限 |

> **心構え**: リスク管理は「損失をゼロにする」仕組みではなく「壊滅的損失を回避する」仕組みである。最悪ケースでは口座資金の一部を失う覚悟が必要。

## 11. 成功指標

### 11.0 シャープレシオの計算方法（全Phase共通）

本プロジェクトにおけるシャープレシオは以下の定義で統一する:

```
SR（年率） = (年率リターン - リスクフリーレート) / 年率リターンの標準偏差
```

- **リスクフリーレート**: 日本国債10年物利回り（計算時点の最新値を使用）
- **年率換算**: 日次リターンを√252で年率換算（4時間足の場合は√(252×6)で換算）
- **測定期間**: 最低3か月分のデータ。3か月未満のSRは参考値とし判定に使用しない

### 11.1 Phase 1-2（バックテスト・ペーパー段階）

| 指標 | 基準 | 根拠 |
|------|------|------|
| 最大ドローダウン | < 20% | doc 02: ハードリミット |
| ウォークフォワード効率 | > 50% | doc 03: 信頼性の基準 |
| シャープレシオ | > 1.0（年率換算、上記定義） | リテール基準0.75以上。インフラコストを考慮し1.0に設定 |
| カオステスト | 6シナリオ全合格 | セクション6.3: リスク管理の能動的検証 |
| システム連続稼働 | > 30日間無障害 | 安定性の最低基準 |
| ペーパートレード数 | ≥ 100（推奨200） | 統計的有意性の最低基準 |
| 市場レジームカバー | ADX≥25が30%以上 AND ADX<25が30%以上 | 定量基準。主観判断に依存しない |

### 11.2 Phase 3（リアル取引段階）

| 指標 | 基準 | 根拠 |
|------|------|------|
| 月次総合収支 | インフラコスト込みでプラス | コスト負けは不合格 |
| ベンチマーク比較 | リスクフリーレート+3%（年率）を上回る | FXとインデックス投資はアセットクラスが異なるため、リスクフリーレート+αを基準とする |
| ペーパーとの乖離 | SR乖離が±20%以内 | doc 02: ギャップの許容範囲 |
| 最大ドローダウン | < 15% | リアルではより厳しく |
| 投入資金上限 | 総金融資産の10%以下 かつ 6か月分生活費を除いた余剰資金 | 口座全損を前提とした資金管理 |

> **撤退基準**: Phase 3開始後6か月間でリスクフリーレート+3%を下回り続けた場合、システムの根本的な見直しまたは撤退を判断する。撤退判断はSR < 0.5（6か月移動平均）を定量基準とし、「もう少し様子を見よう」の誘惑に乗らない。
