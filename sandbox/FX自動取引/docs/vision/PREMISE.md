# 世界観の前提 — 亡き者と継承

> このドキュメントは `STORY.md`（物語）と `OPERATING_MODEL.md`（運用モデル）の**間**に置く。
> 運用モデルを読む前に、**いま私たちがどの世界に立っているか**を確認するための一枚。

---

## なぜこのドキュメントが必要か

`OPERATING_MODEL.md v1` を反論屋に叩かれたとき、もっとも重い指摘は次のものだった:

> 「既存実装と未実装の混在により、未実装の機能を既存機能のように描くトリックになっている。
>  読み手は『これらは動いている』と誤読する」

これは、**書き手（私）と読み手（次の私／AI／未来の自分）が同じ世界観を共有していなかった**から起きた。

そこで、運用モデルを読む前提として、**この世界はゼロベースの再構築フェーズである**と明示する。

---

## いま私たちが立っている世界

### この世界では、以下は **「亡き者」** として扱う

過去の運用は終わった。**遺産は引き継がず、構築をやり直す**。

| 亡き者にしたもの | 理由 |
|---|---|
| 既存戦略（MTFPullback / BollingerReversal / MA Crossover / 派生） | バックテスト前提（PF 2.05 等）が再現未検証 |
| 既存パラメータ（RSI 35/65 / ATR 2.0 / ADX 25 / 等） | 元データ・期間・グリッド最適化の根拠が脆弱 |
| 既存ペア構成（EUR_USD / USD_JPY） | ペア選定の論理的根拠が欠落 |
| 既存数値（PF 2.05 / 1.97、勝率28%、累計-8129 JPY 等） | カーブフィット疑い・短期サンプル誤差 |
| 既存仮説（「強トレンドに弱い」「USDJPY と Pullback の相性」等） | 検証されていない一次仮説 |

これらは**コードベースには残っていても、新しい運用モデルの前提条件にはしない**。

### この世界では、以下を **継承する**

過去の運用から得た**ノウハウ・教訓**は、新しい世界の土台として使う。

| 継承するもの | 何を学んだか |
|---|---|
| バックテスト vs 本番の乖離は普遍的 | 数値は実証されるまで仮値。再現性検証ステージを必須に |
| カーブフィット罠 | グリッドサーチ最良値は信用しない。Walk-forward / IS-OOS で確かめる |
| ペア別の振る舞い違い | 「全ペア一律」は危険。ペア別最適化を最初から前提に |
| 撤退判断は数値で支える | 物語的判断は意思決定を遅らせる |
| 戦略は生命体である | エコシステム発想を採用（→ STORY） |
| 庭師×賢者の対話モデル | ヒト×AI協業の作法を採用（→ STORY / 運用モデル） |
| 学習ループの必要性 | 失敗を遺伝子として残す（→ STORY / 運用モデル） |
| スリッページ実測の重要性 | 「想定価格」を粗い計測で誤計上する罠を学んだ |
| 物語の自己永続化バイアス | 物語が間違っても捨てる条項を最初から組み込む |
| 反論屋・レビュワーの並列起動 | 設計段階で多視点を必須化する（メタなノウハウ） |

これらは **「過去から学んだ抗体」** として、新しい運用モデルの構造に組み込む。

## いま、何をしていないか

意図的に**しない**ことを明示する。誤読を防ぐため。

- **数字を使わない**: 閾値・期間・PF・パラメータをロジック構築段階では書かない。数字を入れると「具体的に見える」が、ロジックが固まる前に数字を入れると**ロジックが数字に縛られる**。
- **既存コードに新機能を生やさない**: 運用モデルが描く仕組みのうち、未実装のもの（系統樹・図鑑・墓地・賢者の世代交代等）を**いきなり実装しない**。ロジックが固まり、数字が降りてから実装計画を立てる。
- **既存運用の判断（USD_JPY 処遇 / GBP_JPY 撤退PR 等）を運用モデルで決めようとしない**: それらは**亡き者の世界の問題**で、新しい運用モデルが答える問題ではない。

## いま、何をしているか

ロジックを作る過程である。具体的には:

1. **ビジョン (STORY.md)** — 何のために何を作るかの北極星
2. **世界観の前提 (本ドキュメント)** — 亡き者と継承の整理
3. **運用モデル (OPERATING_MODEL.md)** — どう動くシステムか、数字なしでロジックを描く ← **いまここ**
4. **SPEC** — ロジックが固まったら数字を降ろす（将来）
5. **実装** — SPEC が固まったらコードに落とす（将来）

## 読み手への注意

運用モデルを読むときは、必ず先に本ドキュメントを読むこと。

- 既存ファイル名（`STATUS.md` / `regime_detector.py` / `trades.db` 等）が運用モデルに登場しても、それは**「亡き者」を指している**か、**「将来再構築の参考」**として書かれているかのいずれか。**「いま動いているもの」**として読むのは誤り。
- 数字が出てこないのは**未確定**だからではなく、**意図的に保留**しているから。
