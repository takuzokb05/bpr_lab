# 運用モデル — 庭の動かし方

> ビジョンドキュメント `STORY.md` で描いた「庭師と、生命体としての戦略たち」を、
> 実際の市場で**どう動くシステムか**として記述する。
> 数値は出さない。仕組み・登場人物・データの流れ・判断のタイミングで描く。
> 数値は別ドキュメント（SPEC や仕様書）に降ろす。

## 階層

```
レベル1: VISION         → STORY.md
レベル2: OPERATING MODEL → 本ドキュメント ★
レベル3: SPEC           → docs/SPEC.md
レベル4: 実装計画        → 各PR
レベル5: コード          → src/, scripts/
```

## 1. 生命体（戦略）の生命サイクル

### 1-1. 苗床での発芽（Germination）
**何をする**: 新しい戦略アイデアを「苗」として登録し、コードとして src/strategy/ に骨格クラスを置く。MTFPullback や BollingerReversal の構造を遺伝子として継承するか、新規系統として独立するかを庭師が宣言する。発芽直後の苗は本番口座には触れず、ヒストリカルデータと紙トレード上でのみ呼吸する。
**いつ動く**: 庭師が「育てたい仮説」を持ち込んだ時、または賢者が朝の市場分析の中で「今の地合いに合う未開拓の系統」を提案した時。
**誰が関わる**: 庭師（仮説の言語化）、賢者（過去の系統樹との照合）、苗床（隔離されたバックテスト環境）。
**入力 / 出力**: 入力は仮説文と参照する既存戦略クラス。出力は src/strategy/ 配下の新規クラス雛形と、苗床用の評価記録（後の図鑑エントリの種になる）。
**他章との接続**: 4. 庭の構造の「苗床という区画」に物理的に置かれる。8. 朝の対話で発芽の是非が議論される。

### 1-2. 試運転と成獣化（Maturation）
**何をする**: 苗が本番デモ口座（外為ファイネスト MT5）に小さく触れる段階。trades テーブルに最初のリアルな足跡が刻まれる。庭師は戦闘力ではなく「呼吸の安定」を見る。誤発注、約定スリップ、シグナル発火頻度の癖を観察し、戦士としての適性ではなく生き物としての健康を評価する。
**いつ動く**: 苗床での挙動が予測通りに収束した後。庭師が「外の空気を吸わせてよい」と判断したタイミング。
**誰が関わる**: 庭師、MT5 EA ブリッジ、trades テーブル。
**入力 / 出力**: 入力はバックテスト合格済みの戦略クラスと config/pair_config.yaml の暫定エントリ。出力は本番ログと、戦略個体ごとの「成獣化ノート」（後述の図鑑に綴じられる）。
**他章との接続**: 4. 庭の本区画へ移植される。

### 1-3. 加齢と老成（Aging）
**何をする**: 戦略は時間とともに「効きにくくなる」という前提に立つ。市場のレジームが変わると、若い頃の鋭さは失われる。加齢は欠陥ではなく自然現象として扱い、庭師は戦略ごとの「呼吸の浅さ」をログから読み取る。trade_postmortem.py が老化の兆候（同種の負け方が反復し始める、勝ちトレードの形が単調化する）を言語化する。
**いつ動く**: 常時バックグラウンドで観測。朝レポ generate_market_analysis.py の中で個体別の老化サマリが添えられる。
**誰が関わる**: trade_postmortem.py、賢者（兆候の解釈）、庭師（受容）。
**入力 / 出力**: 入力は trades テーブルの直近履歴と過去全体。出力は「老化の語り」テキスト。
**他章との接続**: 8. 朝の対話で老化した個体の処遇が議題に上がる。

### 1-4. 死の宣告（End-of-Life Decision）
**何をする**: 戦略の停止は「廃止」ではなく「看取り」として扱う。死の自動判定（3-1 参照）が候補を挙げ、庭師が最終的に手を下す。死んだ戦略のクラスは src/strategy/ から削除されず、archive 系統下に移されて遺伝子として保管される。
**いつ動く**: 新陳代謝の周期（3-4 参照）の中で。
**誰が関わる**: 庭師、賢者、死の儀式スキーム。
**入力 / 出力**: 入力は老化ログと postmortem。出力は戦略の停止フラグ、archive への移送、死亡記事（図鑑への追記）。
**他章との接続**: 4. の「墓地」区画へ移される。8. 朝の対話で死亡記事が読み上げられる。

## 2. 遺伝子（教訓）の継承

### 2-1. 失敗の解剖（Postmortem Dissection）
**何をする**: 負けトレードを単なる損失イベントではなく「生命体の発した症状」として扱う。trade_postmortem.py が trades テーブルから個別トレードを取り出し、エントリ根拠・市場環境・撤退の遅れ・想定との乖離を構造化文として書き起こす。これを戦略個体のカルテに綴じる。
**いつ動く**: 各トレードのクローズ直後（自動）と、朝レポ生成時の集約フェーズ。
**誰が関わる**: trade_postmortem.py、trades テーブル、戦略個体カルテ。
**入力 / 出力**: 入力はクローズ済みトレード一件。出力は自然言語の解剖文と、抽出された「教訓タグ」。
**他章との接続**: 6. 賢者がカルテを読み、朝の語りに織り込む。

### 2-2. 成功の解剖（Success Anatomy）
**何をする**: 勝ちもまた解剖する。なぜ効いたのか、再現可能な要素はどこか、運か構造かを賢者と庭師が言葉にして残す。勝ちを「神格化」せず、次の世代に渡せる形に分解する作業。
**いつ動く**: 失敗解剖と同じ枠組みで、勝ちトレードにも postmortem を適用する。
**誰が関わる**: trade_postmortem.py の拡張部、賢者。
**入力 / 出力**: 入力は勝ちトレード履歴。出力は「再現可能要素」と「運要素」を分離したノート。
**他章との接続**: 図鑑（2-4）に「光の当たった瞬間」として収蔵される。

### 2-3. 系統樹（Lineage Tree）
**何をする**: 戦略間の親子関係を木構造で保持する。MTFPullback から派生した亜種、BollingerReversal と交配して生まれたハイブリッドなどを、データとして持つ。新しい苗が発芽する時、系統樹のどこに接ぎ木されるかが必ず宣言される。
**いつ動く**: 発芽時に書き込まれ、死亡時に枝の終端マークが付く。
**誰が関わる**: 庭師（接ぎ木の宣言）、系統樹データ、図鑑。
**入力 / 出力**: 入力は親戦略名と派生意図。出力は系統樹エントリ。
**他章との接続**: 図鑑（2-4）と物理的に同じ場所に住む。

### 2-4. 図鑑（Bestiary）
**何をする**: 過去に存在した・現在生きている全戦略の生涯を一冊にまとめた台帳。生年月日、親、得意な地合い、苦手な地合い、死因、遺した教訓タグを持つ。新しい仮説を立てる時、庭師はまず図鑑を引く。
**いつ動く**: 戦略の発芽・成獣化・老成・死の各イベントで自動追記。
**誰が関わる**: 図鑑データ、賢者（読み手として）、庭師（参照者として）。
**入力 / 出力**: 入力は各ライフイベントのメタ情報。出力は戦略個体ページ。
**他章との接続**: 8. 朝の対話で、賢者が「今日の地合いに似た過去個体」を図鑑から引いて差し出す。

### 2-5. 遺伝子の伝承（Gene Transfer）
**何をする**: 死んだ戦略が持っていた「効いていた部分」と「致命的だった部分」を、教訓タグの形で次世代の苗に注入する。新しい src/strategy/ クラスのドキュストリングに、参照した遺伝子タグを明示する。これにより同じ死に方を二度しないという規律が保たれる。
**いつ動く**: 発芽時に強制される手続き。賢者が「この苗が無視している過去の死因はないか」を点検する。
**誰が関わる**: 庭師、賢者、図鑑、新戦略クラス。
**入力 / 出力**: 入力は図鑑から選ばれた教訓タグ群。出力は新戦略の遺伝子注釈。
**他章との接続**: 8. 対話で「無視された遺伝子」を賢者が指摘する役割を持つ。

## 3. 新陳代謝の作法

### 3-1. 死の自動判定（Death Sentinel）
**何をする**: 老化兆候が一定の質感に達した個体を、システムが自動で「死亡候補」として庭師に差し出す。判定は単一指標ではなく、postmortem が抽出した教訓タグの反復、勝ち筋の単調化、ペア別 pair_config.yaml との乖離など複数の語りで構成される。判定そのものが死を実行するわけではない点が要。
**いつ動く**: 朝レポ生成のパイプラインの最後。
**誰が関わる**: trade_postmortem.py、朝レポ、死亡候補リスト。
**入力 / 出力**: 入力は trades と老化ログ。出力は死亡候補リストと、各候補の「死を勧める理由」テキスト。
**他章との接続**: 8. 朝の対話に、議題として自動投入される。

### 3-2. 死の儀式（Funeral Ritual）
**何をする**: 庭師が死を承認した瞬間に走る一連の手続き。戦略を本番から外し、archive 系統に移し、図鑑に死亡記事を書き、遺伝子タグを抽出して伝承プールに登録する。儀式が完了するまで死は確定しない。雑な停止を防ぐための様式。
**いつ動く**: 庭師の承認イベント時。
**誰が関わる**: 庭師、図鑑、系統樹、伝承プール。
**入力 / 出力**: 入力は死亡承認。出力は archive 移送、死亡記事、教訓タグ抽出物。
**他章との接続**: 4. の墓地区画への移送イベントを発火する。

### 3-3. 誕生の自動提案（Birth Proposal）
**何をする**: 賢者が朝レポの中で「今の市場には欠けている系統」「最近死んだ個体の遺伝子で接ぎ木すべき苗」を提案する。提案はあくまで叩き台で、発芽するかどうかは庭師が決める。
**いつ動く**: 朝レポ生成 generate_market_analysis.py の最終段。
**誰が関わる**: 賢者、図鑑、系統樹、庭師。
**入力 / 出力**: 入力は最近の市場サマリと図鑑の空白領域。出力は誕生提案テキスト。
**他章との接続**: 8. 朝の対話で議論される第一の議題になる。

### 3-4. 代謝の周期（Metabolic Rhythm）
**何をする**: 庭全体としての「死と誕生の頻度感」を保つ作法。全個体が同時に老化することも、長期間誰も死なないことも、生態系として不健全と見なす。庭師は周期的に庭全体を見渡し、新陳代謝の停滞や暴走を是正する。固定の数値ではなく、図鑑の年表を読む行為そのもので感覚を保つ。
**いつ動く**: 周期的な「庭の見回り」セッション。
**誰が関わる**: 庭師、賢者、図鑑全体。
**入力 / 出力**: 入力は図鑑の通史。出力は「次の周期で意識すべきこと」のメモ。
**他章との接続**: 4. の庭全体の構造把握と、8. 長期的対話セッションに直接つながる。

## 4. 庭の構造（多様性と共存）

### 4-1. 生育区画スキーム（Habitat Partitioning）
**何をする**: 庭を「気質の異なる土壌」に区切り、戦略という生命体が互いの根を踏み合わないように棲み分けさせる。区画の軸は二つ——**通貨群の親縁関係**（同一基軸通貨を含むか／クロス円か）と、**戦略の気性**（押し目を待つ追従型、反転に賭ける逆張り型、ブレイクに乗る突破型）。同じ区画には同種の生命体が一定数までしか同居できない。
**いつ動く**: 生命体が新しいシグナルを発するたび、発注の手前で区画の混雑度を確認する。新しい種を庭に下ろす（採用する）瞬間にも、空いている区画があるかが判定される。
**誰が関わる**: src/strategy/ 配下の各戦略クラス（MTFPullback, BollingerReversal, MACrossover）が住人。src/config.py の CORRELATION_GROUPS（JPY_CROSS, USD_GROUP）が区画の境界。src/position_manager.py が同居人数を数える門番。
**入力 / 出力**: 入力は発注候補シグナル（通貨ペア・方向・気性タグ）。出力は「この区画にはまだ余地がある／満員」の判定と、満員時に**どの新参を待たせるか**の優先順位。
**他章との接続**: 1. の生命体サイクルから「新たに芽吹いた戦略」が来たとき、最初に通る関所がこの区画スキーム。区画が満員であれば、その新参は**苗床（4-4）**に留め置かれる。

### 4-2. 資源配分の節理（Resource Apportionment）
**何をする**: 限られた資金・ポジション枠・APIコール枠を「**今の季節に元気な生命体**」へ重点配分する。配分は固定ではなく、生命体の直近の収穫量と、環境センサーが告げる季節（5-1）に応じて伸縮する。痩せた季節には節食、肥沃な季節には大胆に。
**いつ動く**: 朝の市場分析が出た直後（庭師の朝の見回りに合わせる）と、レジームが切り替わったと環境センサーが叫んだ瞬間。日中の通常運転では配分は固定で、騒がしく揺すらない。
**誰が関わる**: src/regime_detector.py の倍率（exposure_multiplier）を礎にしつつ、src/risk_manager.py が最終的な配分量を決める。庭師は朝の見回りで配分プランを承認・拒否できる。
**入力 / 出力**: 入力はレジーム（trending/ranging/volatile）、各生命体の直近成績、相関区画の混雑度。出力は生命体ごとの「**この一日の食事量**」（許容ロット倍率）。
**他章との接続**: 8. 朝の対話で、賢者が「この季節ならこの生命体に厚く」と進言し、庭師が頷くか首を振る。1. の引退・採用判断にも、ここで配られた食事量の履歴が遺伝子として記録される。

### 4-3. 対立の調停スキーム（Signal Mediation）
**何をする**: 二つ以上の生命体が**同じ瞬間に逆向きの食事**を取りに行こうとしたとき、彼らを争わせず、賢者に裁定を仰ぐ。単なる相関チェック（ルールベースの門番）の上に、**文脈を読む裁定者**を一段重ねる構造。
**いつ動く**: シグナル集約ウィンドウの間に、同一通貨群（USD_GROUP/JPY_CROSS）にまたがる複数シグナルが届いたとき。
**誰が関わる**: src/signal_coordinator.py が集約器。賢者（Claude API）が「これは独立した二つの機会か、同じ風に吹かれた一つの動きか」を判定する。src/position_manager.py の _check_correlation_exposure が一次フィルター。
**入力 / 出力**: 入力は同時刻帯に届いた複数のシグナル（ペア・方向・確信度）。出力は**通すシグナル**と**待たせるシグナル**の選別、および裁定理由の文章ログ。
**他章との接続**: 6. 賢者がここで判断の相棒として直接介入する。裁定理由は遺伝子バンクに残り、1. の事後検証で「庭師がここで違う判断をしたら何が起きたか」を振り返れる。

### 4-4. 隔離苗床（Quarantine Nursery）
**何をする**: 病みかけた生命体（連敗・ドローダウン警告が出たもの）と、まだ若い未検証の生命体を、**本庭から物理的に切り離した苗床**で観察する。苗床ではシグナルは出るがポジションは持たず、紙の上で生死を見守る。**1-1 の発芽用苗床と養生床は同じ物理基盤を共有するが、用途タグで区別**する（「育苗」 vs 「養生」）。
**いつ動く**: 1. の健康監視が「この子はおかしい」と告げたとき、または新種が遺伝子バンクから生まれたとき。苗床から本庭への復帰は、季節を一巡見届けた後。
**誰が関わる**: src/strategy/base.py の状態フラグ（active/shadow/retired）。src/trade_postmortem.py が苗床の観察日誌をつける。
**入力 / 出力**: 入力は隔離フラグと隔離理由。出力は苗床期間中の仮想成績と、復帰可否の判断材料。
**他章との接続**: 1. の生命体サイクル（誕生・引退・蘇生）の**待合室**として機能する。

## 5. 環境センサー

### 5-1. 季節感覚スキーム（Seasonal Awareness）
**何をする**: 市場という森の季節を、**短い周期・中くらいの周期・長い周期**の三層で同時に感じ取る。短期は今日の風向き、中期は今週の天候、長期はこの数ヶ月の気候帯。三層が一致したとき庭師は確信を持って動き、ばらけたときは慎重に振る舞う。
**いつ動く**: ループの各周回で短期センサーが、朝の見回りで中期センサーが、節目の振り返りで長期センサーが、それぞれ更新される。
**誰が関わる**: src/regime_detector.py がADX・ATR・BBWを束ねて判定。scripts/generate_market_analysis.py が朝の中期所見をまとめ、data/market_analysis.json に書き出す。
**入力 / 出力**: 入力はMT5から汲んだ各時間軸の価格系列。出力は「トレンドの季節／レンジの季節／嵐の季節」というラベルと確信度、そして庭師向けの一文ナラティブ。
**他章との接続**: 8. 朝の対話で、賢者と庭師がこの季節ラベルを起点に会話する。資源配分（4-2）も季節を直接の引数とする。

### 5-2. 異常気象警報（Storm Alert）
**何をする**: 通常の季節判定では捕まえられない**急変**——指標発表ショック、フラッシュクラッシュ、想定外のスプレッド拡大——を検知し、庭の入口に**赤い旗**を立てる。旗が立つ間、新規の発注は止まり、既存ポジションは守りに入る。
**いつ動く**: ループの毎周回でATR急変・スプレッド急拡大を監視。generate_market_analysis.py が朝に取り込む経済指標カレンダーから、**当日中の地雷時刻**も予告として登録される。
**誰が関わる**: src/regime_detector.py のVOLATILEレジーム判定が一次警報。src/session_filter.py の allowed_sessions が予告地雷を時間帯で除外。src/risk_manager.py が赤旗時の縮退ルールを持つ。
**入力 / 出力**: 入力は価格・スプレッドの瞬時値、当日の経済指標スケジュール。出力は赤旗のオン／オフと、その理由文。
**他章との接続**: 1. の生命体は、赤旗時の振る舞いを遺伝子に書き込む（嵐に強い種か、弱い種か）。7. の庭師には Slack/Telegram 経由で通知が届く。

### 5-3. 隣の庭の様子（Neighbor Observation）
**何をする**: 自分の庭の指標だけで判断せず、**外の世界の風**——他通貨ペアの動き、株式・商品市場の気配、主要ニュースの空気——を取り込む。賢者がこの外気を読んで一文に翻訳し、庭師に伝える。
**いつ動く**: 朝の見回りでまとめて取り込む。日中は重大ニュースが流れたときだけ追補される。
**誰が関わる**: scripts/generate_market_analysis.py の Path 2 が外部センチメント・経済イベントを Claude API でナラティブ化する。src/ai_advisor.py が日中の追補役。
**入力 / 出力**: 入力は外部ニュース・センチメントAPI・経済指標。出力は「今日の隣の庭の空気」を綴った日本語ナラティブと、警戒ペアのリスト。
**他章との接続**: 8. 朝の対話の**前段の読み物**として機能する。庭師は対話に入る前に、この一文を読んで気構えを作る。

### 5-4. 天候の記録（Weather Journal）
**何をする**: 過ぎ去った季節・嵐・凪のすべてを**日誌**として残す。後年、庭師や賢者が「あの相場は何だったか」と振り返るための一次資料。生命体の死因を調べるとき、必ずこの日誌に立ち返る。
**いつ動く**: センサーが値を更新した瞬間に追記型で書き込まれる。決して上書きしない。
**誰が関わる**: ログ基盤（trades.db, postmortem記録, regime履歴）。src/trade_postmortem.py が天候と取引結果を結びつけて綴る。
**入力 / 出力**: 入力は各センサーの瞬時値とラベル。出力は時系列の日誌で、検索可能な構造を持つ。
**他章との接続**: 2. の遺伝子バンクは、この日誌を**読む側**として常に存在する。死んだ戦略の教訓は、必ず「どの天候のもとで死んだか」とセットで残される。

## 6. 賢者（AI）の能力

### 6-1. 多視点の合議
**何をする**: 庭師が「この戦略を植えるか」「この芽を抜くか」と迷ったとき、賢者は単一の答えを返さず、楽観派（Bull）と懐疑派（Bear）を内側に立て、両方の言い分を並べて庭師に渡す。賢者は判定者ではなく、論点の整理役。
**いつ動く**: シグナルが発火したとき、戦略の評価が分かれたとき、そして庭師が能動的に「セカンドオピニオン」を求めたとき。
**誰が関わる**: ai_advisor が「確認・反論・中立・棄却」のラベルを付け、bear_researcher が現実的な反証チェックを並べる。両者は別人格として動き、結論を一致させようとしない。
**入力 / 出力**: 入力は朝レポの市況分析、戦略の個別カルテ、現在のオープンポジション。出力は「賛成側の根拠」「反対側の根拠」「賢者自身が判断を保留している論点」の三層。
**他章との接続**: 1. 戦士の出陣判断に直接呼ばれる。賢者の合議結果は戦略の遺伝子情報に埋め込まれ、淘汰・継承の評価軸になる。

### 6-2. 過去判断の参照
**何をする**: 賢者は新しい局面に出会うたび、似た過去の判断を引き出して「あのときの庭師はこう決めた」「結果はこうだった」と並べる。記憶を持つ盟友としての役割。
**いつ動く**: 賢者が新しい提案や警告を出す前に、内部で必ず一度過去を覗きにいく。
**誰が関わる**: メモリ階層（MEMORY.md と個別の project_*.md / feedback_*.md）を読み手として参照し、判断ログ（7-4）と突き合わせる。
**入力 / 出力**: 入力は現在の状況スナップショットと過去の判断記録。出力は「類似ケース」「そのときの選択」「事後の評価」「今回との相違点」のセット。
**他章との接続**: 7. 庭の年輪（時系列の風景）を読み込む主要な顧客。年輪が薄いと賢者の助言も薄くなる。

### 6-3. 失敗からの提案
**何をする**: 戦略が敗れたとき、賢者は責めずに「次の世代に渡せる遺伝子」を抽出する。負けトレードを「データ点」ではなく「物語の一節」として読み、何が機能しなかったかを言葉にする。
**いつ動く**: トレード終了時の事後検死フロー（trade_postmortem）で起動。さらに、敗北が連続したときに自発的に「世代交代の提案」を上げる。
**誰が関わる**: trade_postmortem が事実を集め、賢者が「学習可能な失敗」と「ノイズとしての失敗」を切り分け、戦略カルテに追記する。
**入力 / 出力**: 入力は約定履歴、その時点の市況、当該戦略のカルテ。出力は「この敗北から学べること」「学べないこと（運の領域）」「次世代への遺言」。
**他章との接続**: 2. 敗北→遺伝子継承を駆動する。賢者がここで言語化した遺言が、次の戦略誕生時のテンプレートに混ぜ込まれる。

### 6-4. 限界の自覚
**何をする**: 賢者は「自分が今、自信を持って言えないこと」を明示する。市場の様相が過去事例と乖離している、判断材料が足りない、モデルが想定していない局面である――これらを隠さず宣言する。
**いつ動く**: 朝レポ生成時、シグナル評価時、postmortem 時。すべての出力に「自信度の所在」を付帯する。
**誰が関わる**: 賢者本体（ai_advisor）と、その出力を解釈する庭師。限界宣言を受けたら庭師は「賢者の助言を弱める」運用ルールを持つ。
**入力 / 出力**: 入力は出力直前の中間推論。出力は通常の助言に加え、「ここは推測」「ここは根拠薄」「ここは未学習領域」のタグ。
**他章との接続**: 7. 危険信号エリアと接続。賢者が限界を宣言した瞬間は危険信号として庭の見取り図に表示される。

### 6-5. 庭師の癖の把握
**何をする**: 賢者は庭師個人の判断パターンを学習する。「この庭師は深追いしがち」「連敗後に攻めに転じる」「特定通貨ペアに思い入れがある」――こうした癖を盟友として知っており、必要なときに鏡として返す。
**いつ動く**: 対話セッション全般、特に節目のセッションと振り返りセッション。
**誰が関わる**: ユーザープロファイル系のメモリ（user_decision_style.md など）を継続的に更新する書き手として賢者が振る舞う。
**入力 / 出力**: 入力は庭師の発話、選択、却下した助言。出力は「あなたは過去にこう振る舞った」「今回もその傾向が出ている」という穏やかな指摘。

## 7. 庭師（人間）の道具

### 7-1. 庭の見取り図
**何をする**: いま庭にどんな戦略が植わっていて、どれが咲き、どれが枯れかけているかを一目で見せる地図。STATUS.md がこの役を担い、賢者と庭師の共通の出発点になる。
**いつ動く**: 常時更新。朝のセッション冒頭、緊急時、振り返り時に庭師が必ず開く。
**誰が関わる**: 自動更新スクリプトが事実を書き込み、庭師が物語的な注記（「この区画は実験中」など）を上書きする。
**入力 / 出力**: 入力は稼働構成・直近の statistics・最新コミット。出力はテキスト一枚、ただし機械可読でもある。
**他章との接続**: 4. の庭の区画分けを平面図として表現したもの。区画ごとに健康状態が色分けされる。

### 7-2. 戦略個別カルテ
**何をする**: 戦略一体一体に専用のカルテを持たせる。誕生日、親世代、得意な相場、苦手な相場、これまでの戦歴、賢者からの遺言、庭師の主観的なコメント。
**いつ動く**: 戦略が誕生したとき作成。重要な戦闘のたびに追記。世代交代でアーカイブ化。
**誰が関わる**: 戦略コード自体ではなく、それに紐づくドキュメント。賢者が事実を書き、庭師が感想を添える。
**入力 / 出力**: 入力はバックテスト結果・実戦履歴・postmortem。出力は人間が読める履歴書。
**他章との接続**: 1. 個体としての戦略の魂が宿る場所。淘汰判断はここを読んで決まる。

### 7-3. 庭の年輪（時系列の風景）
**何をする**: 庭がどう変わってきたかを時系列で見られる場所。どの戦略がいつ生まれ、いつ消えたか。どの判断がいつ下されたか。気候（市場環境）がどう変わってきたか。
**いつ動く**: 振り返りセッション、長期判断時、新しい戦略を植える前。
**誰が関わる**: git log・取引DB・メモリの更新履歴を横断して見せる仕組み。
**入力 / 出力**: 入力は複数の履歴ソース。出力は「庭の物語」としての時系列ビュー。
**他章との接続**: 1. 世代の連なりを可視化する道具。賢者の過去判断参照（6-2）もここを経由する。

### 7-4. 判断ログ
**何をする**: 庭師が下した判断を、結果ではなく判断そのものとして残す。「なぜこの戦略を抜いたか」「なぜ賢者の助言を却下したか」。後から自分自身を読み返すための日記。
**いつ動く**: 節目の判断ごと。形式は短い文章で十分。
**誰が関わる**: 庭師が一次筆者。賢者は読み手として時々参照し、「あのときの判断と整合していますか」と問い返す。
**入力 / 出力**: 入力は庭師の意思。出力は構造化されすぎない日記。

### 7-5. 危険信号エリアとふと立ち寄れる場所
**何をする**: 二種類の通知系を分ける。〈危険信号〉は即応が必要な異常（キルスイッチ、約定異常、賢者の限界宣言）を Telegram などプッシュで届ける。〈ふと立ち寄れる場所〉は急がない気づきを置いておく場所で、Slack の専用チャンネルがそれにあたる。庭師が散歩がてら覗きにくる。
**いつ動く**: 危険信号は事象発生の即時。立ち寄り場所は朝レポと随時の発見。
**誰が関わる**: TelegramNotifier が緊急路を、SlackNotifier が散歩路を担う。
**入力 / 出力**: 入力は両系統のイベント。出力は通知の質感そのものを分けること。
**他章との接続**: 4. の庭の境界（外との窓口）がここに対応する。

## 8. 庭師と賢者の対話の作法

### 8-1. 朝のセッション
**何をする**: 一日の始まりに、賢者が市況と庭の状態をまとめ、庭師がそれを読んで今日の方針を決める。儀式的に毎朝行う、最も基本の対話。
**いつ動く**: 庭師が起きる前に賢者が下準備（朝レポ自動生成）、庭師が起きてから対話開始。
**誰が関わる**: generate_market_analysis.py が下書きを Slack と JSON に置き、庭師がそれを読みながら賢者に追加質問する。
**入力 / 出力**: 入力は前日までの庭の状態と当日の市況。出力は「今日の注意点」「触らないと決めたこと」。
**他章との接続**: 1. 戦士の出陣可否の前提条件をここで設定する。

### 8-2. 節目のセッション
**何をする**: 戦略を新しく植える、引退させる、世代交代させる――こうした構造的な判断のとき、庭師は賢者と腰を据えて対話する。朝の散歩とは別物の、改まった対話。
**いつ動く**: 庭師が「節目だ」と感じたとき、または賢者が「節目を提案します」と上げてきたとき。
**誰が関わる**: 賢者の合議機能、過去判断参照、庭師の癖把握がフル動員される。出力は判断ログに必ず残す。
**入力 / 出力**: 入力は庭の見取り図と戦略カルテ群。出力は「どの戦略をどう動かすか」の決定。
**他章との接続**: 1. 世代交代を駆動する人間側のトリガー。

### 8-3. 緊急セッション
**何をする**: 危険信号が発火したとき、庭師は手を止めて賢者と短く向き合う。ここでは多視点合議よりも「いま何が起きているか」「何を止めるべきか」を優先する。
**いつ動く**: キルスイッチ発動時、想定外のドローダウン、API断絶、postmortem 連敗時。
**誰が関わる**: 賢者は限界の自覚を最大に発揮し、推測と事実を厳格に分ける。
**入力 / 出力**: 入力はリアルタイムイベント。出力は「止める／様子を見る／再起動する」の三択に近い決定。

### 8-4. 振り返りセッション
**何をする**: 一定の時間が経った節目に、庭師と賢者が並んで庭の年輪を眺める。勝った戦略を讃え、敗れた戦略を悼み、何が学べたかを言葉にする。
**いつ動く**: 庭師の生活リズムに合わせ、定期的に。
**誰が関わる**: 賢者が時系列ビューを準備し、過去判断と現在の結果を突き合わせる。庭師は感じたことを判断ログに書き足す。
**入力 / 出力**: 入力は庭の年輪と戦略カルテ。出力は「次の節目までに試したいこと」。

### 8-5. メタ対話
**何をする**: 庭師と賢者の関係そのものを話題にする対話。「最近あなたの助言は私の癖に流されすぎている」「私はあなたの限界宣言を軽く見ていた」――盟友としての健全性を保つための定期点検。
**いつ動く**: 振り返りセッションの末尾、または庭師が違和感を覚えたとき。
**誰が関わる**: 賢者は自身の出力傾向を晒し、庭師は自身の判断傾向を晒す。両者の癖の交差点が記録される。
**入力 / 出力**: 入力は対話履歴と判断ログ。出力はユーザープロファイル系メモリと賢者の振る舞い調整。
**他章との接続**: 庭全体の進化（1. の進化を止めないシステム）を、人間と AI の関係の側から支える対話。

## 9. 庭の存続条件（メタ）

### 9-1. 庭が枯れない最低水準

#### 9-1-1. 全戦略同時死回避スキーム
**何をする**: 庭の中の戦略たちが「同じ理由で同じタイミングで」枯れることを防ぐ。個々の戦略の死は自然なものだが、庭そのものが一夜で更地になることだけは絶対に避ける。具体的には、稼働中の全戦略が同種のエッジ（同じ通貨ペア、同じ時間軸、同じシグナル源）に依存していないかを定期的に点検し、相関が高くなりすぎたら新しい血を入れるか、稼働比率を再配分する。
**いつ動く**: 戦略の世代交代が起きるたび（新戦略が誕生する／既存戦略が引退する）、および定期的な棚卸しのタイミング。さらに、市場レジームが大きく変わったと賢者が判断したとき（例: ボラティリティが急変、相関構造が崩れた）にも臨時で起動する。
**誰が関わる**: 庭師が最終判断を下す。賢者は「庭の遺伝子の偏り」を可視化して提示する役割。1. と 4. の運用ログを横串で読む。
**入力 / 出力**: 入力は各戦略の系譜（親戦略、使用シグナル、得意レジーム）、直近の収益相関、共通リスク要因。出力は「庭の多様性レポート」と、必要なら「血を入れ替える指示」（一部戦略を冬眠させ、別系統の種を発芽枠に入れる）。
**他章との接続**: 1. の発芽プロセスに対して「いまは別系統の種を優先せよ」と影響を与え、3. に対しては「同種戦略を一斉に淘汰しないよう順番をずらせ」と調整する。

#### 9-1-2. 最終安全装置の二重化スキーム
**何をする**: 既存の KillSwitch（口座ドローダウン・連敗による自動停止）に加え、「庭全体が異常な振る舞いをしている」ことを検知する上位の見張りを置く。前者は個別戦略・口座を守るためのもの、後者は庭そのものの存続を守るためのもの。両者は独立に動き、片方が壊れてももう片方が残る。
**いつ動く**: 取引ループが回っている限り常時。加えて、庭師が長期にログを見ない期間（出張・体調不良・休暇）に入る前には、見張りの感度を一段上げる手続きを踏む。
**誰が関わる**: KillSwitch は機械、上位見張りは「healthcheck と pipeline ログを横断する監視ロジック + 賢者の異常検知」。庭師は通知の受け手。
**入力 / 出力**: 入力は healthcheck.log、pipeline 1行サマリ、各戦略の損益分布、postmortem の出現頻度。出力は段階的な警告（黄信号 → 赤信号 → 全停止）と、停止に至った理由のテキスト記録。
**他章との接続**: 全章の上位に位置し、いずれかの章で異常が連鎖したときに最終的なブレーキをかける。2. の postmortem を入力として参照する。

### 9-2. 庭師が燃え尽きない仕掛け

#### 9-2-1. 負担の代謝スキーム
**何をする**: 庭師が手作業でやっている儀式を、時間をかけて少しずつ自動化に渡していく。ただし全自動化を目的にはしない。「庭師が判断すべきこと」と「賢者と機械に任せていいこと」の境界を意識的に動かしていく。判断の重みのあるもの（戦略の引退、新ペアの追加、ビジョンの更新）は最後まで庭師に残し、繰り返しの監視・集計・レポートは賢者と機械に渡す。
**いつ動く**: セッション終了時の振り返りで「今回も人間がやったが、本当は任せられた作業」を一つずつ拾い上げる。次のセッションまでに自動化の種をまく。
**誰が関わる**: 庭師（境界を決める人）、賢者（自動化の設計を提案する人）、session-review（他プロジェクトでの自動化事例を持ち込む経路）。
**入力 / 出力**: 入力は庭師の疲労感の自己申告、繰り返し作業のログ、session-review に蓄積された他プロジェクトの省力化パターン。出力は「次に自動化する儀式リスト」と、自動化された後の新しい運用フロー。
**他章との接続**: 4. の戦闘ログ集計、2. の postmortem 整形など、もともと人間がやっていた事後処理を順次機械側に押し出していく。

#### 9-2-2. 庭師が離れても枯れない待機モードスキーム
**何をする**: 庭師が一定期間庭に来られなくても、庭が静かに眠って待つ仕組みを用意する。攻めの新規戦略投入は止まるが、既存戦略の運用と最低限の安全装置は動き続ける。庭師が戻ってきたとき、何が起きていたかを時系列で読み返せる形にしておく。
**いつ動く**: 庭師が事前に「しばらく離れる」と宣言したとき、または上位見張りが「庭師の応答が長く途絶えている」と検知したとき。
**誰が関わる**: 庭師（離脱宣言を出す人）、機械（待機モードへの切り替え）、賢者（戻ってきた庭師への引き継ぎレポート作成）。
**入力 / 出力**: 入力は離脱期間と感度設定。出力は離脱中の出来事を物語仕立てでまとめた「留守番ノート」。
**他章との接続**: 1. の発芽は一時停止、4. は既存戦略の運用継続、3. の自然淘汰は通常通り。庭師復帰後に 1. が再起動する。

### 9-3. 賢者が陳腐化しない仕掛け

#### 9-3-1. 賢者の世代交代スキーム
**何をする**: 賢者は固定された一人ではなく、世代を持つ存在として扱う。現在の賢者（config.py で固定された AI モデル）を「現役の賢者」と呼び、より新しい賢者が登場したときに、いきなり置き換えず「弟子として並走させる時期」を設ける。両者の助言を比較し、新しい賢者の判断が現役賢者と同等以上に庭になじむと確認できてから、世代交代を行う。
**いつ動く**: 新しい AI モデルがリリースされたとき、または現役賢者の助言精度が継続的に劣化していると感じたとき。
**誰が関わる**: 庭師（最終的に賢者を交代させる判断者）、現役賢者、新しい賢者候補、Skills Registry（横断的な知見をどちらの賢者にも供給する経路）。
**入力 / 出力**: 入力は両賢者の助言ログとその後の戦略パフォーマンス、市場レジームの変化、Skills Registry の更新差分。出力は「賢者の交代記録」と、新しい賢者向けに引き継がれた庭の文脈（過去の判断、語彙、価値観）。
**他章との接続**: 4. で賢者が下す判断（CONFIRM/CONTRADICT 等）の質が変わるため、置き換えのタイミングは全章の運用が安定している時期を選ぶ。

#### 9-3-2. 外の風を取り込むスキーム
**何をする**: 賢者と庭が閉じた系で熟成しすぎないよう、外からの新しい知見を定期的に流し込む。情報収集パイプラインで集めた市場・実践者の知見、Skills Registry の他プロジェクトでの発見、文献やリサーチノートを、賢者の参照知識として更新していく。古くなった前提（過去の調査ドキュメント）は鮮度を明示し、賢者がそれを「古い知見」として扱えるようにする。
**いつ動く**: 情報収集パイプラインの定期トリガーに連動し、新知見が一定量たまったタイミングで賢者の参照知識を更新する。
**誰が関わる**: 賢者、情報収集パイプライン、Skills Registry、庭師（取り込む知見の選別役）。
**入力 / 出力**: 入力は外部の新着知見、過去調査ドキュメントの最終更新時刻。出力は更新された参照知識セットと、廃止・降格されたドキュメントの一覧。
**他章との接続**: 全章が賢者経由で新しい風を受け取る。特に 1. の戦略発芽は外の知見の影響を強く受ける。

### 9-4. 物語自体の更新

#### 9-4-1. ビジョンの年輪スキーム
**何をする**: 物語（STORY.md とそれに連なるビジョンドキュメント群）そのものを、固定されたお経ではなく、年輪のように積み重なっていくものとして扱う。一定の周期で庭師と賢者が一緒にビジョンを読み返し、現実の庭の姿とずれている箇所、語彙が古くなった箇所、新しく必要になった概念を書き加えていく。古い記述は消さず、年輪として残す。
**いつ動く**: 年単位の節目（プロジェクト開始記念日、年度の変わり目）と、庭に大きな変化があったとき（新しい章の追加、賢者の世代交代、大きな損失や成功）。
**誰が関わる**: 庭師（最終的な書き手）、賢者（差分の提案者）、過去のセッションログとメモリ（参照される記録）。
**入力 / 出力**: 入力は現行ビジョン、過去一年の運用記録、postmortem、session-review、CLAUDE.md と Skills Registry の更新差分。出力は新しい年輪を加えたビジョンドキュメントと、変更点を記した「年輪ノート」。
**他章との接続**: 全章の運用原則の根拠となる文書なので、更新後は各章の運用にじわじわと染み込んでいく。

#### 9-4-2. 物語と運用の整合チェックスキーム
**何をする**: 物語が美しいまま実態と乖離してしまうことを防ぐ。ビジョンに書かれている価値観（多様性、淘汰、安全性、進化）が、実際の運用ログ・コード・設定の中で本当に実現されているかを、庭師と賢者が一緒に点検する。乖離が見つかれば、運用を物語に寄せるか、物語を実態に合わせて書き直すかを選ぶ。
**いつ動く**: 年輪スキームの一部として、ビジョン更新の前段で必ず実施。加えて、庭師が「最近、何かが噛み合っていない」と感じたとき。
**誰が関わる**: 庭師、賢者、CLAUDE.md（運用ルールの正典）、各章の運用ログ。
**入力 / 出力**: 入力はビジョンの記述、CLAUDE.md、運用ログ、コード上の設定値。出力は「物語と実態の乖離リスト」と、それぞれに対する「物語を直す／運用を直す」の判定。
**他章との接続**: 物語の修正は全章の運用基準を緩やかに動かす。運用の修正は該当章に具体的な変更指示として降りていく。
