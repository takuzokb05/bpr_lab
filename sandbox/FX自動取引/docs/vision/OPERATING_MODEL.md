# 運用モデル v2.1 — 庭の動かし方

> 本ドキュメントは `STORY.md`（物語）と `PREMISE.md`（世界観の前提）を読了した上で読まれることを前提とする。
> `PREMISE.md` の宣言通り、過去のパラメータ・戦略・数値は**亡き者**として扱い、ノウハウだけを継承する。
>
> **PREMISE 整合の一行注**: コードベース・実装インフラは残す。**戦略選定・パラメータ・数値・仮説**を捨てる。
> 各 `[骨格実装済・パラメータ未確定]` タグは「コード骨格は再利用するが、それが動いていた閾値・係数は SPEC v2 で再導出する」を意味する。
>
> 数字を入れたくなったら、SPEC へ進む合図である。本ドキュメントには絶対に書かない。

## 1. 戦略のライフサイクル（3スキーム）

### 1-1. 発芽（Germination） [新規構築] [ノウハウ継承]
**何をする**: 新しい戦略アイデアを「苗」として `src/strategy/` 配下に骨格クラスで登録する。発芽直後の苗は本番口座に触れず、ヒストリカルデータと紙トレード上でのみ呼吸する。発芽宣言時に庭師は「親戦略名」と「過去のどの死から学んだか」を一行で書き残す。
**動くタイミング**: 庭師が仮説を持ち込んだ時、または賢者が朝の対話で系統の空白を指摘した時。
**登場人物**: 庭師（仮説の言語化）、賢者（過去の死因の照合）、苗床（隔離されたバックテスト環境）。
**他章との接続**: 3-1（区画分け、空き判定）/ 5-2（戦略カルテに親フィールドを書き込む）。
**SPEC で降ろす数字**: 苗床での観察期間長、本庭への昇格に必要な最小取引数、紙トレード成績の合格ライン、walk-forward / IS-OOS の最低分割数（カーブフィット罠への抗体）。

### 1-2. 成獣化（Maturation） [新規構築] [ノウハウ継承]
**何をする**: 苗床で予測通りに収束した苗を、本番デモ口座（MT5）に小さく触れさせる。庭師は戦闘力ではなく「呼吸の安定」——誤発注、約定スリップ、シグナル発火頻度の癖——を観察する。**バックテスト vs 本番乖離の罠**（PREMISE 継承）に対応するため、紙トレードと実トレードのスリップ実測差分が一定範囲に収まるかを成獣化のゲートとする。**ゲート不合格時は苗床へ差し戻し（死産経路）**、改善見込みなしと判断されれば 1-3 死へ移送される。
**動くタイミング**: 苗床での観察期間が満了し、庭師が「外気を吸わせてよい」と承認した時。
**登場人物**: 庭師、MT5 EA ブリッジ、`trades` テーブル、賢者（乖離の解釈）。
**他章との接続**: 2-1（季節判定で許容ロット倍率が決まる）/ 3-2（資源配分の対象に加わる）/ 1-3（成獣化失敗時の死産経路）。
**SPEC で降ろす数字**: スリップ乖離の許容上限、成獣化に要する実トレード件数、最大ロット倍率、苗床差し戻しの最大回数、死産判定の連続不合格回数。

### 1-3. 死と遺伝子継承（Funeral & Gene Transfer） [新規構築] [ノウハウ継承]
**何をする**: 死の自動判定（2-4）が候補を挙げ、庭師が承認した瞬間に一連の儀式が走る。戦略を本番から外し、`archive` 系統に移し、戦略カルテに死亡記事と「教訓タグ」を追記する。教訓タグは次の発芽（1-1）時に賢者が「この苗が無視している過去の死因はないか」を点検する素材となる。**v1 系統樹・図鑑・遺伝子伝承の三スキームをここに統合**。**「死産」（成獣化失敗）も「老衰死」（長期運用後の劣化）と同じ儀式を経る**。
**動くタイミング**: 庭師の承認イベント時。
**登場人物**: 庭師、賢者、戦略カルテ、`archive` ディレクトリ。
**他章との接続**: 1-1（次世代発芽時に教訓タグを参照）/ 5-1（postmortem に死亡理由が綴じられる）。
**SPEC で降ろす数字**: 死亡候補から実死までの猶予期間、`archive` への自動移送タイマー、教訓タグの最大保持数。

## 2. 監視と安全装置（4スキーム）

### 2-1. 季節判定（Seasonal Awareness） [骨格実装済・パラメータ未確定]
**骨格の所在**: `src/regime_detector.py`、`scripts/generate_market_analysis.py`、`data/market_analysis.json`
**閾値継承**: ❌（既存の ADX/ATR/BBW 閾値は SPEC v2 で再導出）
**何をする**: 市場という森の季節を**短期・中期・長期**の三層で同時に感じ取る。短期は今日の風向き、中期は今週の天候、長期はこの数ヶ月の気候帯。三層が一致した時に庭師は確信を持って動き、ばらけた時は慎重に振る舞う。
**動くタイミング**: 取引ループの各周回で短期センサーが、朝レポ生成で中期センサーが、振り返りで長期センサーが更新される。
**登場人物**: `regime_detector.py`、`generate_market_analysis.py`、賢者（ナラティブ化）。
**他章との接続**: 3-2（資源配分が季節を引数とする）/ 4-1（朝の対話の起点）。
**SPEC で降ろす数字**: 三層の各観察期間長、レジーム判定の各指標閾値、三層一致判定のスコアリング重み。

### 2-2. 異常気象警報（Storm Alert） [骨格実装済・パラメータ未確定]
**骨格の所在**: `src/session_filter.py` (allowed_sessions)、`src/risk_manager.py` (縮退ルール)、`src/regime_detector.py` (VOLATILE 判定)、Telegram 通知
**閾値継承**: ❌（スプレッド・ATR 急変閾値は SPEC v2 で再導出）
**何をする**: 通常の季節判定では捕まえられない**急変**——指標発表ショック、フラッシュクラッシュ、想定外のスプレッド拡大——を検知し、庭の入口に**赤い旗**を立てる。旗が立つ間、新規発注は止まり、既存ポジションは守りに入る。
**動くタイミング**: ループの毎周回でスプレッド・ATR 急変を監視。朝レポ取り込み時に当日地雷時刻を予告登録。
**登場人物**: `regime_detector.py` の VOLATILE 判定、`session_filter.py` の allowed_sessions、`risk_manager.py` の縮退ルール、`telegram_notifier.py`。
**他章との接続**: 2-3（最終安全装置の入力）/ 4-2（緊急対話の発火源）。
**SPEC で降ろす数字**: スプレッド急拡大の検知閾値、ATR 急変の検知ウィンドウと倍率、赤旗継続の最低時間、地雷時刻前後の除外幅。

### 2-3. 最終安全装置（Final KillSwitch） [骨格実装済・パラメータ未確定]
**骨格の所在**: `src/risk_manager.py` の KillSwitch、`healthcheck.log`、`pipeline` ログ
**閾値継承**: ❌（DD・連敗閾値は SPEC v2 で再導出）
**何をする**: 個別戦略・口座を守るための既存 KillSwitch（口座ドローダウン・連敗による自動停止）に加え、「庭全体が異常な振る舞いをしている」ことを検知する**上位の見張り**を独立に置く。前者は機械、後者は `healthcheck.log` と `pipeline` ログを横断する監視ロジック + 賢者の異常検知。両者は独立に動き、片方が壊れてももう片方が残る。
**動くタイミング**: 取引ループが回っている限り常時。庭師が長期離脱する前は感度を一段上げる。
**登場人物**: 既存 KillSwitch（機械）、上位見張り（横断監視、新規構築）、賢者（異常解釈）、庭師（通知の受け手）。
**他章との接続**: 2-2（赤旗が連続した時に格上げ）/ 5-1（停止理由は postmortem に必ず残す）。
**SPEC で降ろす数字**: 口座ドローダウン停止閾値、連敗停止閾値、上位見張りの黄信号・赤信号・全停止の段階閾値、長期離脱時の感度倍率。

### 2-4. 老化観察と死の自動判定（Aging & Death Sentinel） [新規構築]
**何をする**: 戦略は時間とともに効きにくくなるという前提に立ち、`trades` テーブルから「同種の負け方の反復」「勝ちトレードの形の単調化」「直近成績と過去全体の乖離」を観察し続ける（旧 1-3 老化を統合）。観察結果が一定の質感に達した個体を、システムが自動で「死亡候補」として庭師に差し出す。判定は単一指標ではなく、postmortem が抽出した教訓タグの反復、勝ち筋の単調化、相関区画（3-1）との不適合など複数の語りで構成される。**判定そのものが死を実行するわけではない**点が要——実死の引き金は庭師の承認（1-3）。
**動くタイミング**: 常時バックグラウンドで観測。朝レポ生成のパイプライン最終段で死亡候補リストを出力。
**登場人物**: `trade_postmortem.py`、朝レポ、死亡候補リスト、賢者（理由の言語化）。
**他章との接続**: 1-2（成獣化失敗の死産経路を受ける）/ 1-3（出力先）/ 4-1（朝の対話で議題化）。
**SPEC で降ろす数字**: 老化判定の連敗閾値、勝ち単調化の検出ウィンドウ、過去成績との乖離許容幅、死亡候補に上げる連敗閾値、PF 下限、教訓タグ反復回数の閾値、観察期間長。

## 3. 配分と調停（3スキーム）

### 3-1. 区画分け（Habitat Partitioning） [骨格実装済・パラメータ未確定]
**骨格の所在**: `src/config.py` の `CORRELATION_GROUPS`、`src/position_manager.py` の `_check_correlation_exposure`
**閾値継承**: ❌（区画境界・最大同居数は SPEC v2 で再定義）
**何をする**: 庭を「気質の異なる土壌」に区切り、戦略が互いの根を踏み合わないように棲み分けさせる。区画の軸は二つ——**通貨群の親縁関係**（同一基軸通貨か、クロス円か）と**戦略の気性**（追従型・逆張り型・突破型）。同じ区画には同種の生命体が一定数までしか同居できない。**マクロショック時の全戦略同時死を防ぐため、入口（区画）と出口（2-3 最終安全装置の上位見張り）の両側でガードする**。
**動くタイミング**: 生命体が新しいシグナルを発する時、発注の手前で区画の混雑度を確認する。新種を庭に下ろす時にも空き区画判定が走る。
**登場人物**: `src/strategy/` の各クラス、`config.py` の `CORRELATION_GROUPS`、`position_manager.py`（門番）。
**他章との接続**: 1-1（発芽時に空き区画があるかを判定）/ 2-3（マクロショック時の同時死回避を出口側で補完）/ 3-3（調停の前提）。
**SPEC で降ろす数字**: 各区画の最大同居数、気性タグの分類定義（カテゴリ判定式の型を含む）、満員時の優先順位ルール。

### 3-2. 資源配分（Resource Apportionment） [骨格実装済・パラメータ未確定]
**骨格の所在**: `src/regime_detector.py` の `exposure_multiplier`、`src/risk_manager.py`
**閾値継承**: ❌（レジーム別倍率は SPEC v2 で再導出）
**何をする**: 限られた資金・ポジション枠・APIコール枠を「**今の季節に元気な生命体**」へ重点配分する。配分は固定ではなく、生命体の直近収穫量と、季節判定（2-1）に応じて伸縮する。痩せた季節には節食、肥沃な季節には大胆に。日中の通常運転では配分は固定で、騒がしく揺すらない。
**動くタイミング**: 朝の市場分析直後と、レジームが切り替わったと環境センサーが叫んだ瞬間。
**登場人物**: `regime_detector.py` の `exposure_multiplier`、`risk_manager.py`、庭師（朝の見回りで承認・拒否）。
**他章との接続**: 2-1（入力）/ 4-1（朝の対話で進言される）。
**SPEC で降ろす数字**: 各レジームの倍率、直近成績による補正係数、生命体ごとの基準ロット、配分更新の最小間隔。

### 3-3. シグナル調停（Signal Mediation） [骨格実装済・パラメータ未確定]
**骨格の所在**: `src/signal_coordinator.py`、`src/position_manager.py` の `_check_correlation_exposure`
**閾値継承**: ❌（相関閾値・集約ウィンドウは SPEC v2 で再導出）
**何をする**: 二つ以上の生命体が**同じ瞬間に逆向きの食事**を取りに行こうとした時、彼らを争わせず、賢者に裁定を仰ぐ。単なる相関チェック（ルールベースの門番）の上に、**文脈を読む裁定者**を一段重ねる構造。
**動くタイミング**: シグナル集約ウィンドウの間に、同一通貨群にまたがる複数シグナルが届いた時。
**登場人物**: `signal_coordinator.py`（集約器）、`position_manager._check_correlation_exposure`（一次フィルター）、賢者（X-1 の多視点合議で最終裁定）。
**他章との接続**: 3-1（区画満員時にここへ落ちる）/ 4-1（裁定理由は朝の対話で振り返られる）/ X-1（合議の常駐）。
**SPEC で降ろす数字**: シグナル集約ウィンドウ長、相関閾値、賢者裁定を呼び出すまでの一次フィルター段数、裁定タイムアウト。

## 4. 賢者と庭師の対話（2スキーム）

### 4-1. 朝のセッション（節目モード兼用）（Morning Session, with Retrospective Mode） [骨格実装済・パラメータ未確定]
**骨格の所在**: `scripts/generate_market_analysis.py`、Slack 通知
**閾値継承**: ❌（朝レポ生成時刻・通知優先度は SPEC v2 で再導出）
**何をする**: 一日の始まりに、賢者が市況と庭の状態をまとめ、庭師がそれを読んで今日の方針を決める。儀式的に毎朝行う、最も基本の対話。死亡候補リスト（2-4）、誕生提案、季節ラベル（2-1）、配分プラン（3-2）がすべてここで一度に議題化される。**節目（戦略の発芽・引退・世代交代、大きな損失・成功、定期的な振り返り）には「節目モード」で起動**し、通常モードより深く掘る——勝った戦略を讃え、敗れた戦略を悼み、次の節目までに試したいことを判断ログ（5-2 庭師欄）に書き残す。
**動くタイミング**: 通常モードは毎朝。節目モードは庭師が「節目だ」と感じた時、賢者が「節目を提案します」と上げてきた時、または定期周期。
**登場人物**: `generate_market_analysis.py`（下書き生成）、Slack（散歩路）、賢者、庭師、X-1（多視点合議の常駐）。
**他章との接続**: 全章のハブ。1-3 / 2-4 / 3-2 / 3-3 がここに集約される。5-2 へ判断ログ書き出し。
**SPEC で降ろす数字**: 朝レポ生成のスケジュール時刻、Slack 通知の優先度しきい値、対話タイムアウト、節目モードの定期周期、節目モードの最小トレード件数。

### 4-2. 緊急セッション（Emergency Session） [新規構築] [ノウハウ継承]
**何をする**: 危険信号（2-2 赤旗、2-3 KillSwitch）が発火した時、庭師は手を止めて賢者と短く向き合う。ここでは多視点合議よりも「いま何が起きているか」「何を止めるべきか」を優先し、「止める／様子を見る／再起動する」の三択に近い決定を下す。Telegram で即時プッシュ。
**動くタイミング**: KillSwitch 発動、想定外のドローダウン、API 断絶、postmortem 連敗時。
**登場人物**: 賢者（限界の自覚を最大化、推測と事実を厳格に分ける）、庭師、`telegram_notifier.py`。
**他章との接続**: 2-2 / 2-3 から発火。5-2（判断ログに必ず残す）。
**SPEC で降ろす数字**: 緊急判定の発火条件、Telegram 通知の優先度、応答タイムアウト後の自動アクション。

## X. 横断機能

### X-1. 多視点合議と限界の自覚（Bull/Bear & Self-Awareness） [骨格実装済・パラメータ未確定]
**骨格の所在**: `src/ai_advisor.py`（CONFIRM/CONTRADICT/NEUTRAL/REJECT）、`src/bear_researcher.py`、`src/conviction_scorer.py`
**閾値継承**: ❌（スコア閾値・重み付け係数は SPEC v2 で再導出）
**位置づけ**: 章番号外。**全対話・全シグナル評価に常駐するモード**。「セッション」ではなく「常時稼働するレンズ」として扱うため章から分離。
**何をする**: 庭師が「この戦略を植えるか」「この芽を抜くか」と迷った時、賢者は単一の答えを返さず、楽観派（Bull）と懐疑派（Bear）を内側に立て、両方の言い分を並べて庭師に渡す。さらに賢者はすべての出力に「自信度の所在」を付帯する——市場の様相が過去事例と乖離している、判断材料が足りない、未学習領域である——これらを隠さず宣言する。**設計段階の意思決定でも、賢者を複数視点で並列起動することを必須とする**（メタなノウハウ継承）。
**動くタイミング**: シグナル発火時、戦略評価が分かれた時、庭師がセカンドオピニオンを求めた時、すべての出力時、そして**運用モデル / SPEC / 実装計画の設計レビュー時**。
**登場人物**: `ai_advisor.py`、`bear_researcher.py`、`conviction_scorer.py`、設計レビュー時の反論屋エージェント（karen / pragmatist 等）。
**他章との接続**: 4-1 / 4-2 すべての対話に常駐。3-3 のシグナル裁定にも呼ばれる。**設計レビュー時は本ドキュメント自身に対しても呼ばれる**。
**SPEC で降ろす数字**: Bull/Bear のスコア閾値、限界宣言を出す自信度の下限、合議結果の重み付け係数、設計レビュー時の最小反論屋数。

## 5. 学習と更新（2スキーム）

### 5-1. Postmortem（事後検死） [骨格実装済・パラメータ未確定]
**骨格の所在**: `src/trade_postmortem.py`、`trades` テーブル
**閾値継承**: ⚠️（PR #25 で max_tokens 拡大は継承可、教訓タグ抽出ロジックは SPEC v2 で再設計）
**何をする**: 負けトレードも勝ちトレードも、単なる損益イベントではなく「生命体の発した症状」として扱う。`trade_postmortem.py` が `trades` テーブルから個別トレードを取り出し、エントリ根拠・市場環境・撤退の遅れ・想定との乖離を構造化文として書き起こす。勝ちは「再現可能要素」と「運要素」を分離する。
**動くタイミング**: 各トレードのクローズ直後（自動）と、朝レポ生成時の集約フェーズ。
**登場人物**: `trade_postmortem.py`、`trades` テーブル、戦略カルテ（5-2）、賢者（教訓タグの抽出）。
**他章との接続**: 2-4（老化判定の入力）/ 1-3（死亡時の遺伝子抽出元）/ 5-2（カルテへの追記）。
**SPEC で降ろす数字**: postmortem 起動の最小トレード単位、集約ウィンドウ、教訓タグの抽出閾値、運/構造の分離スコア閾値。

### 5-2. 戦略カルテ + 判断ログ（Strategy Chart with Decision Journal） [新規構築] [ノウハウ継承]
**何をする**: 戦略一体ごとに専用カルテを持たせる。**機械欄**（誕生日、親戦略名、得意/苦手な相場、戦歴、教訓タグ群、賢者からの遺言）と**庭師欄**（庭師が下した判断とその理由——なぜこの戦略を抜いたか、なぜ賢者の助言を却下したか、ふと感じた違和感）を**同じカルテに同居**させる。**v1 系統樹・図鑑・遺伝子伝承・庭師の癖把握・メタ対話をこのスキームに圧縮**。庭師欄の自由記述から「物語破棄オプション」発火条件 (c) が自動カウントされる。
**動くタイミング**: 戦略の発芽（1-1）、postmortem（5-1）の追記、節目（4-1）、緊急セッション（4-2）後、庭師が違和感を覚えた時。
**登場人物**: 戦略コードに紐づくドキュメント、賢者（機械欄の書き手）、庭師（庭師欄の書き手）、`user_decision_style.md`（庭師の癖の蓄積先、本スキームの副産物として更新）。
**他章との接続**: 1-1 / 1-3 で読み書きされる。4-1 / 4-2 から書き込み。X-1 が参照する。**物語破棄オプション条項の自動発火カウンタとも接続**。
**SPEC で降ろす数字**: カルテ自動追記の頻度、保持期間、教訓タグの最大数、archive 移送のトリガー、`user_decision_style.md` 更新の周期、「噛み合わない」記載の検出キーワード集。

## 物語破棄オプション条項（v2.1 改訂版）

本運用モデルが現実と乖離した時、**この物語を捨てる権利**を庭師は留保する。

### 発火条件

**(a) 庭師の能動的破棄宣言**: 5-4 の年輪点検（巻末「物語の年輪」を参照）で「物語と運用の乖離」が修復不能と判定された時。

**(b) 物語の語彙では説明できない死因の連続**: 連続する postmortem が「庭師・賢者・生命体・遺伝子・季節・嵐」の語彙では記述できない死因を示した時。具体的には postmortem の構造化文に物語語彙が一語も含まれない事象が連続発生した時。

**(c) 自動発火**: **戦略カルテ庭師欄（5-2）に「噛み合わない」「違和感」「腑に落ちない」のいずれかのキーワードが3回連続で記載された時、システムが自動的に庭師に確認を求める通知を発火する**（v2 の主観条件「三度感じた」を客観カウントに変換）。庭師は確認時に物語破棄か継続かを選択する。

### 期限規定（v2.1 で追加）

**自動再評価義務**: SPEC v2 確定から **6ヶ月後** に、本運用モデルは自動的に再評価対象となる。庭師と賢者は再評価セッション（4-1 節目モード）を必ず開く。

**自動アーカイブ**: 再評価セッションが期限から **+30日以内に開かれなかった場合**、本ドキュメント・PREMISE.md・STORY.md は自動的に `_archive/YYYY-MM-DD/` 配下へ移送され、新しい世界観の起草が必須となる。**書いた本人にも消す権利があるのと同様、書いた本人にも忘れる責任がある**。

### 手続き

破棄が発火した時:
1. `STORY.md` / `PREMISE.md` / 本ドキュメントを `_archive/YYYY-MM-DD/` 配下に保管
2. **コードベース・データ・既存運用は維持**（コードと物語は別レイヤー）
3. 新しい世界観を起草（白紙から、または既存の年輪を踏まえて）
4. 移送と起草の経緯を git log と判断ログ（5-2 庭師欄）に残す

### 抗体としての意味

物語が美しいまま実態と乖離する**自己永続化バイアス**（PREMISE 参照）への抗体として、本条項は**書いた瞬間から有効**である。

**条項自体の永続化禁止**: 本条項を「より厳しく」する改訂は受け付けない（緩和方向のみ可）。ただし v2 の「凍結禁止」だけでは弱いため、**期限規定 (上記)** を併用することで、条項自体も時間経過で自動的に再評価対象になる。条項が物語の存続装置になることを構造的に防ぐ。

## 物語の年輪（廃止 → 巻末条項に集約）

v2 の 5-4「ビジョンの年輪と物語破棄」スキームは独立スキームとして廃止し、本巻末「物語破棄オプション条項」に統合した。

「**物語の年輪**」概念自体は残す: STORY.md / PREMISE.md / OPERATING_MODEL.md は固定された経典ではなく、年単位の節目（プロジェクト開始記念日、年度の変わり目、賢者世代交代、大損失・成功）に庭師と賢者が一緒に読み返し、書き加える。古い記述は消さず年輪として残す。**ただし条項の発火条件 (a)(b)(c) のいずれかが発火した場合は、年輪ではなく破棄が選択される**。
