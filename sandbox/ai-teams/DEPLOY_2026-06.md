# AI Teams v3 デプロイ手順（個人利用・出先アクセス / 2026-06）

VPS（常駐）にバックエンドとフロントを置き、Cloudflare Tunnel + Access で安全に公開する個人向け構成。

## なぜ VPS か（Vercel ではない理由）

バックエンドは **常駐プロセス**が必須でサーバーレス（Vercel）に載らない:
- SSE の長時間ストリーム（討論は数分、floor-open は無期限待機）
- リクエストを越えて生きるバックグラウンドスレッド（producer 完走 / paused 待機）
- インメモリのセッション状態（`SESSIONS` 辞書・スレッド・条件変数）＝**単一ワーカー前提**

→ FastAPI バックエンドは VPS。フロント（Next.js）は同一 VPS 同居が最も簡単（Vercel も可）。

## 構成図

```
[ブラウザ(出先)] --HTTPS--> [Cloudflare Tunnel + Access(ログイン)]
                                   │
                              [Caddy (VPS)]
                                   ├ /api/*  → strip /api → uvicorn :8000 (FastAPI・SSE素通し)
                                   └ /*      → next start :3000 (フロント)
```

- `/api/*` を **Caddy が直接バックエンドへ**回すのが肝。Next.js の rewrite を SSE に使うと**バッファされてライブ配信が壊れる**（実測済み）。Caddy/cloudflared は SSE を素通しする。
- 同一ホスト名なので CORS は実質発火せず、Cloudflare Access のログインセッション（Cookie）が frontend/backend 双方をカバーする。

## 必要なもの
- VPS（Windows Server。FX 運用と同居可）／ Python・Node 導入済み／ ドメイン（Cloudflare 管理）

## 1. 環境変数

`sandbox/ai-teams/.env`（gitignore 済み）:
```
ANTHROPIC_API_KEY=sk-ant-...           # 実LLM/Web検索用
AI_TEAMS_MODEL=claude-opus-4-8          # 任意（既定 opus-4-8）
AI_TEAMS_API_TOKEN=<長いランダム文字列> # 認証トークン（多層防御の控え）
AI_TEAMS_ALLOWED_ORIGINS=https://aiteams.example.com  # 本番フロントのオリジン
```

`web/.env.local`（フロントのビルド時に埋め込まれる）:
```
NEXT_PUBLIC_API_BASE=/api               # 同一ホスト相対。Caddy が /api→backend に回す
NEXT_PUBLIC_API_TOKEN=<同じトークン>     # 全 fetch が Bearer で送る
```
> 注: `NEXT_PUBLIC_*` はクライアント JS に埋め込まれる＝**真の秘密にはならない**。実際のアクセス制御は **Cloudflare Access のログイン**が担う。トークンは「Access を迂回して backend トンネルへ直当てされた場合」の控え（多層防御）。

## 2. バックエンド起動（単一ワーカー）

```
cd sandbox/ai-teams
uvicorn api.main:app --host 127.0.0.1 --port 8000
```
- **必ず単一ワーカー**（`--workers` を付けない）。インメモリ状態をプロセス間で共有しないため。
- Windows では **Task Scheduler** で常駐起動する（ssh 越しの `Start-Process` は silent death する既知の罠。FX 運用と同じく `Start-ScheduledTask` 経由）。
- `127.0.0.1` バインドで外部直アクセスを塞ぎ、公開は Caddy/トンネル経由に限定。

## 3. フロント起動

```
cd sandbox/ai-teams/web
npm run build
npm run start   # :3000（単一ホストで同居）
```
これも Task Scheduler で常駐。

## 4. Caddy（リバースプロキシ・SSE 素通し）

`Caddyfile`:
```
aiteams.example.com {
    handle_path /api/* {
        reverse_proxy 127.0.0.1:8000 {
            flush_interval -1   # SSE をバッファせず即フラッシュ
        }
    }
    handle {
        reverse_proxy 127.0.0.1:3000
    }
}
```
- `handle_path /api/*` が `/api` を剥がして backend に渡す（フロントの `NEXT_PUBLIC_API_BASE=/api` と整合）。
- `flush_interval -1` で SSE をバッファしない（ライブ配信維持）。

## 5. Cloudflare Tunnel + Access（HTTPS＋ログイン）

```
cloudflared tunnel login
cloudflared tunnel create aiteams
# config.yml: aiteams.example.com → http://127.0.0.1:443（Caddy） もしくは Caddy を介さず
#   ingress で /api と / を分けてもよい（Caddy を使うなら Caddy 一本に流す）
cloudflared tunnel route dns aiteams aiteams.example.com
cloudflared tunnel run aiteams
```
- これで**ポート開放なし・自動 HTTPS**。Windows でも `cloudflared` を Task Scheduler 常駐。
- **Cloudflare Access**: Zero Trust ダッシュボードで `aiteams.example.com` に Application を作り、自分のメール/Google だけ許可するポリシーを設定。→ **これが実質の認証（ログインゲート）**。

## セキュリティ・コストの要点（個人利用でも必須）

1. **無認証で公開しない**: 公開した瞬間、誰でも `/sessions` を叩いて**あなたの Anthropic キーで課金**できる。Cloudflare Access（ログイン）＋ `AI_TEAMS_API_TOKEN`（控え）の二段で塞ぐ。
2. **実LLM/Web検索は従量課金**: 出先で気軽に回すとコストが乗る。Mock トグル（既定）で無料確認、実 LLM は意識して使う。
3. **単一ワーカー固定**（状態共有のため）。
4. **VPS の .env を git に乗せない**（gitignore 済みを確認）。チャットにトークン/キーを貼らない。
