# AI Teams 討論 vs 単体LLM 比較検証（2026-06）

実LLM（Opus 4.8）で、AI Teams の多人格討論と、Opus 4.8 単体回答を同一議題で比較した記録。

## 検証設定

- **議題**: 「高齢化が進む地方自治体で、住民の行政手続き（住民票・各種申請・相談）をデジタル化・オンライン化で効率化すべきか、対面の窓口サービスを維持・充実させるべきか。限られた職員と予算でどちらに軸足を置くべきか。」（行政の意思決定）
- **討論**: AI Teams（API・Opus 4.8）。パネル = 司会 / 論理 / アイデア / 共感 / **スティーブ・ジョブズ（演）** / **ソクラテス（演）** / 議長。17ターン・約465秒。
- **単体**: Opus 4.8 に「あなた1人の最善の助言」を依頼。約39秒。
- **コスト最適化（CLAUDE.md 準拠）**: 討論＝テスト対象のアプリ本体なので API 実走（コスト承認済み）。単体＝Claude Code サブエージェント（現セッションが Opus 4.8）で取得し**API二重課金を回避**。

## 結論

**標準的な実務判断なら単体 Opus 4.8 で十分。だが価値観が絡む高難度の意思決定では討論が単体を明確に超えた**——特に**ソクラテスとジョブズが、単体回答が完全に見落とした次元を掘り当てた**。

## 単体 Opus 4.8（強いベースライン）

クリーンで即実装可能な管理職向け助言：
- 結論「対面を軸足に、デジタルはバックヤード効率化で相談時間を生む補完」。
- 「対面 vs オンライン」を誤った二択と退け、**フロント（住民接点）/バック（内部処理）の分離**で再構成。
- リスク（移行期の二重運用コスト増・中途半端デジタルの逆効果・"窓口維持"が現状温存の言い訳・単独内製は割に合わない）。
- アクション（手続き2軸棚卸し・書かない窓口・KPIは"削減"でなく"再配分"・デジタル弱者の代替導線）。
- **39秒・速い・実行可能**。

## 討論が「上乗せ」した価値（単体に無かったもの）

| 上乗せ | 具体（討論からの抜粋） |
|---|---|
| **倫理次元の発掘（ソクラテス）** | 「『やれている気がする』ことと『やれている』ことは同じか」「効率化された後の住民は、まだ"市民"か、それとも"顧客"か」「**見守りと監視は、どこで分かれるのか**」「この変更で、住民が『**放っておいてくれ**』と言う自由は増えるか減るか」「行政が本人より先に"あなたは困っている"と判定する権限を持つ」。単体は効率/公平は論じたが、**監視性・自律性の倫理を一切触れなかった**。 |
| **KPIの反転＋操作不能化** | 単体は「KPIは再配分」止まり。討論は「**届いていない人（＝闇）の数を恥の指標として毎年晒す**」へ反転し、論理担当が「**外部データ突合でしか出すな**（課税×減免申請／介護認定×利用実績／最終接触日×年齢世帯——誰が計算しても同じ＝操作不能）」と具体設計まで。 |
| **対抗的な潰し→再構築** | ジョブズ「顔の要否で線引き」→共感/アイデアが「定型に紛れた非定型（離婚届を取りに来た震える手）を見逃す」と論破→ジョブズ「線で割り切れると思ったのが誤りだった」と自己修正。KPIも逆選別批判で複数回潰され鍛え直された。 |
| **未解決の保存** | 議事録が「観測KPI化の是非／誰が困窮を定義するか／把握の主体そのもの／解けなかった非対称性」を**丸めず保留**。単体は綺麗に決着＝決着しない論点を隠した。 |
| **枠の破壊（アイデア）** | 「**把握の主体を行政→住民同士へ**（隣人・配達・民生委員の気づきを行政が受け取る回路）。データ突合は人の網が切れた地域だけに向ける」——単体未到達の再構成。 |
| **本質の絞り込み（ジョブズ）** | 「役所への用事をゼロにする」「機械にできることは裏で終わらせ、人間にしかできないことに人を100%」「測るべきは"観測した数"でなく"**届いていない人の数**"だ」。 |

## 討論のコスト

- **約12倍遅い（約8分 vs 39秒）＋実APIコスト**（17 Opus ターン）。
- **即実行性は単体が上**。討論は深いが「算数（人手）が成立しないかも」「保留」が多く、意思決定者が自分で統合する負荷が残る。

## 正直な要点（なぜ差が出たか）

討論の上乗せの一部は「**ペルソナが構造的に角度を強制する**」効果（Red Team・ソクラテスの倫理・ジョブズの本質主義）。**単体でも「監視性も論じよ／反対側を steelman せよ」と指示すれば近づける**。ただし討論は**反復的な潰し合いで結論をさらに先へ**進める（KPIが4〜5回鍛え直された）点が単発プロンプトでは再現しにくい。

## 使い分け

- **ルーチン判断** → 単体 Opus（速い・サブスク内で安い）。
- **後戻りできない／価値観が割れる／盲点が危険な判断** → 討論（コスト相応の深さ。倫理次元・操作不能なKPI設計・未解決の明示が効く）。

## ゲストペルソナの実挙動（jobs / socrates・初の実ログ）

- **ソクラテスがMVP**。診断で懸念された「問いだけで議論停滞」は**杞憂**——各ターン末に「次の人が名指せ」と**前進を促し**、討論を深化させた。→ 保留中だった socrates の前進担保ガードは**ほぼ不要**（現状維持でよい）。
- **ジョブズも良好**。論理担当の「絞れ」と役割衝突せず（jobs=本質/体験、logic=数字/算数で分化）、自己修正も機能。保留ガードの必要性は軽微。

## この検証が示した最大の弱点と、次機能の示唆

3討論すべてに共通する弱点は **「事実・数字を持たないまま抽象で進む」** こと（論理担当が繰り返し「私はここを数字で持っていない、要検証」「件数分布の数字を誰か持っていますか」と言い続けた）。ここから次機能が示唆される：

1. **討論前の事実接地（ファイル読込／Web検索）**: ユーザーは「普通のAIを使って"なんだかなー"となってから」この討論ツールに来る想定。手元の資料・データを読み込ませ、必要なら Web で事実を引いて、**仮説でなく実状況の上で討論**させる。本検証の「数字がない」壁を直接解消する最有力候補。
2. **討論前の主訴確認（曖昧なら質問）**: 生の議題が曖昧なとき、開始前に2〜3問の確認（本当の意思決定は何か／制約／既に試したこと／"良い"の定義）を挟み、**主訴からの逸脱を防ぐ**。司会プロンプトの「具体に降ろす問い」を、討論の外（intake）に出して固定する発想。

いずれも「討論の手前に **準備フェーズ（intake＋接地）** を足す」という同じ方向。詳細は別途検討。
