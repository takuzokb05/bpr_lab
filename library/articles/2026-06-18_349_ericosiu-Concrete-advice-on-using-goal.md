## 投稿内容

Most people who've tried /goal in Claude Code are using it wrong.

They're running it on vague tasks with no clear finish line. The agent stalls, they check in, it stalls again. Nothing ships unless someone's watching.

The fix is defining a real "definition of done." Not "work on the execution panel" but "build it, verify it, run a dry run, confirm the output." That specificity is what separates an agent that loops until it quits from one that actually delivers.

I run Hermes in Slack with Codex as the frontier model. When I drop a list of open threads and say "slash goal them," it knows what done looks like because I specced each one up front. No babysitting.

The part nobody talks about: /goal doesn't come with an overnight operator loop built in. 

I built /goal overnight as a custom preset. One high-priority mission, no destructive commands, work verified at the end, morning packet dropped with everything it completed.

I wake up to a finished task instead of a stalled thread.

If you're running a business and you've got half-built projects sitting across multiple agents, the bottleneck isn't effort or compute. It's that you haven't told the agent what done actually looks like. 

Fix that, and you can genuinely let it run while you sleep.
