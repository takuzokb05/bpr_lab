## 要約
CLAUDE.mdに「何でも書こう」という発想が逆効果になる問題を、良い例・悪い例の対比で解説した保存版記事。核心は「削除してもClaudeがミスをしないなら書くな」という判断基準。WHAT（技術スタック・依存関係）、WHY（アーキテクチャ決定の理由）、HOW（作業ルール）の3層構造で記述する。悪い例として「コードは読みやすく書いてください」「エラーハンドリングを適切に行ってください」など自明すぎる指示を挙げる。良い例として「`npm run lint`ではなく`npx eslint --cache`を使うこと（CIと同一環境のため）」など具体的な制約・理由付きの指示を挙げる。200行超えると重要指示を読み飛ばすため、`/init`コマンドで自動生成したベースに少しずつ追記する育て方を推奨。ドメイン知識など頻繁に参照しないものはSkillsへ分離するのが正しい設計。
