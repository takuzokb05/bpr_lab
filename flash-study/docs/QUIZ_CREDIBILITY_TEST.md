# クイズ信憑性テスト レポート（genQuiz 実プロンプト）

> 対象: `prototype/index.html` の `genQuiz()`（L475-487）が出力した実 4 択クイズ
> 評価軸: `quiz-evaluator` agent 定義（grounding / answer-key / distractor / difficulty・coverage / format）
> 関連: 本実装は `docs/UIUX_REDESIGN_SPEC.md` §4.4 / Step 4「クイズ信頼ガード」（L257, L284-287, L399）
> 作成日: 2026-06-01

---

## 0. テストの位置づけ（前提の正直メモ）

flash-study の最大リスクは仕様書（spec §7・CLAUDE.md「Product principle」）が明言するとおり
**「もっともらしいが微妙に誤った 4 択を AI が量産すること」**。本テストはその実害度を、本実装着手前に実プロンプトで測るもの。

- 評価は **常に SOURCE（`deck.source` 原文）基準**。評価者の外部知識で正誤を判定しない（hallucination をあぶり出すため）。
- 今回の集計対象は **24 問**。サンプルは「横浜市第9期介護保険事業計画」系の行政・数値中心テキスト（`prototype/#_A1_第9期介護保険_.json` がその代表デッキ、14 問）。

## 1. 信憑性サマリ

| 指標 | 件数 | 率 |
|---|---|---|
| 総問題数 | 24 | 100% |
| **信用可（credible）** | **24** | **100.0%** |
| 要改善（flagged / WEAK 相当） | 0 | 0.0% |
| **major（致命的・正解誤りや幻覚）** | **0** | **0.0%** |

**信憑性率 = 100.0%（24/24）／ major 0 件。**

`flagged: []`（指摘ゼロ）。つまり今回のサンプル範囲では、

- 正解キー（`a`）が原文と一致しないもの = **0**
- 原文に無い事実を要求する幻覚 = **0**
- 誤選択肢が原文上も正解になり得るもの = **0**
- 解説（`e`）と正解・原文の不整合 = **0**

### ただし、この 100% を額面どおり受け取らない（過信防止）

| 注意点 | 内容 |
|---|---|
| **ドメイン偏り** | サンプルが「行政の確定数値・所管・法定期限」という **事実想起型**に偏る。これは原文照合で正誤が一意に決まりやすく、AI が最も得意な領域。法令解釈・条件分岐・因果関係を問う問題では崩れ方が変わる。 |
| **小 N** | 24 問は「重大欠陥が高頻度で出ていないこと」は示すが、**低頻度（数% オーダー）の誤りの不在は保証しない**。本番では生成回数が桁違いに増え、低頻度バグが必ず顕在化する。 |
| **生成側の自己防衛が効いている** | 後段の `normalizeQuiz`（L417-433）が「正解本文での再特定」「特定不能なら問題ごと drop」「空・重複選択肢除去」を実施済み。**つまり一部の崩れは集計に上がる前に黙って落とされている可能性がある**（＝表示問題数の信憑性は高いが、生成歩留まりは別途見るべき）。 |

> 結論の先取り: **「現状の生成は良好だが、本実装でガード（レビュー画面＋自己検証）を外してよい根拠にはならない」**（→ §5）。

## 2. 失敗パターンの類型化

`quiz-evaluator` が警戒する 4 類型に照らした結果。**今回サンプルでは 4 類型いずれも 0 件**。
ただし「なぜ 0 だったか」「どこが崩れる一歩手前か」を、実問題を引いて記録する（本実装ガードの設計根拠になる）。

### 類型A: 一意性崩れ（正解が複数取れてしまう）
- **検出: 0 件。** 数値・所管・人名を問う設問は、原文の表が単一の値を与えるため一意性が保たれた。
- **崩れ手前の例**: Q「介護保険課が直接所管する事業はどれか」（正解「給付適正化事業」）。原文 §1 は所管/非所管を明示分けしているため成立。だが**もし距離の近い「給付費」「保険運営費」を誤選択肢に並べたら、原文の表現次第で一意性が揺らぐ**（今回は遠い誤選択肢＝計画策定・施設整備・AIケアプランで回避できていた）。一意性は「設問の良さ」ではなく「誤選択肢の選び方」に依存している、という弱点が見える。

### 類型B: 誤選択肢が正解になり得る（distractor が事実上も正しい）
- **検出: 0 件。** 多くの設問で誤選択肢が「原文に登場する別の数値」を流用しており（例: 事業費総額の誤肢に 3,140.5/3,306/3,485 億円）、**原文照合では明確に別物**＝誤りと判定できた。これは良い distractor 設計。
- **注意**: この手法は「同種・近傍の数値を誤肢にする」ため、**1 箇所でも原文の値を取り違えると即 major（正解と誤肢が入れ替わる）**になる構造。今回は無事だったが、リスクの芽は数値想起型に内在する。

### 類型C: 幻覚（原文に無い事実を要求／作り出す）
- **検出: 0 件。** 設問・正解・解説のすべてが原文 §1-§9 のいずれかに対応づけられた。
  例: Q「合議体の委員数」→ §7「委員5人」／Q「法定の通知期限」→ §7「30日以内」／Q「+233% 急増の細事業」→ §5「次期介護保険システム運用事業 +233%」。
- **崩れ手前**: 「30日以内」を一般知識で「14日」「60日」と混同するのは典型的幻覚源だが、今回は**原文に 30日が明記**されていたため踏みとどまっている。原文に根拠が無い論点を出題し始めた瞬間に幻覚化する。

### 類型D: 解説不整合（`e` が正解・原文と食い違う）
- **検出: 0 件。** 解説はいずれも 40 字以内（プロンプト制約）かつ正解と整合（例: `"審査会資料点検事業が1.94億円で最大。"`）。原文の数値ともズレなし。

### 類型まとめ
| 類型 | 件数 | 主因（なぜ抑えられたか） |
|---|---|---|
| A 一意性崩れ | 0 | 原文が単一値を与える事実想起型＋誤肢が十分に遠い |
| B 誤肢が正解化 | 0 | 誤肢に「原文の別数値」を流用＝原文照合で明確に誤り |
| C 幻覚 | 0 | 全設問が原文の節に逆引き可能 |
| D 解説不整合 | 0 | 40字制約＋正解の言い換えに留めている |

**最頻パターン: 該当なし（全 0）。** ただし**最も再発しやすいと予測されるのは A と B（数値取り違え起点）** であり、本実装ガードはここを主標的に設計する（§3, §4）。

## 3. genQuiz プロンプト改善案（現行 → 改善）

### 現行プロンプト（`index.html` L477-483）
```
あなたは資格試験の問題作成者です。次の本文だけを根拠に4択問題を作ってください。
・問題数は本文量に応じて適切に（目安: 本文200〜300字あたり1問、重要論点を網羅）。
・選択肢は必ず4つ・正解1つ。・解説は40字以内で簡潔に。
出力はJSON配列のみ。前後の説明やマークダウンは禁止。
形式: [{"q":"問題文","o":["1","2","3","4"],"a":正解番号(0始まり),"e":"短い解説"}]
本文:
"""${text}"""
```

良い点: 「本文だけを根拠に」「JSON のみ」を既に含む。崩れていないのはこの 1 行（本文限定）が効いている。
弱い点: ①本文外参照の禁止が**一文だけで弱い** ②選択肢の**相互排他（誤肢が正解化しない）要求が無い** ③**根拠スニペットを残させていない**ため後段の自己検証・本文引用 UI（§4.4）が作れない ④数値取り違えへの注意喚起が無い。

### 改善版（具体文言・そのまま差し替え可）
```
あなたは資格試験の問題作成者です。【本文】の記述だけを根拠に4択問題を作ります。

# 厳守ルール（破ったら出題しない）
1. 本文外参照の禁止: 本文に書かれていない事実・一般常識・あなたの知識を一切使わない。
   本文に根拠が無い論点は出題しないこと（無理に問題数を埋めない）。
2. 一意性: 正解は本文上ただ1つに定まること。本文だけで答えが2つ以上取れる問題は作らない。
3. 選択肢の相互排他: 4つの選択肢は互いに両立しない（同時に正しくならない）ものにする。
   誤りの選択肢3つは「本文に照らすと明確に誤り」であること。
   本文上も正しい・部分的に正しい選択肢を誤りの選択肢に混ぜない。
4. 数値・固有名詞: 本文の値を一字一句そのまま使う。近い数値を誤りの選択肢に置くときも、
   正解の値は本文と完全一致させる（取り違え厳禁）。
5. 各問に "src"（本文根拠スニペット）を必須で付ける:
   その問題の正解の根拠になる本文の一節を、改変せず10〜60字そのまま抜き出す。
   抜き出せない＝本文に根拠が無い問題は作らない。
6. 解説 "e" は40字以内。正解が本文のどこに基づくかを簡潔に書く。

# 出力（JSON配列のみ。前後の説明・マークダウン・コードフェンス禁止）
[{"q":"問題文","o":["選択肢1","選択肢2","選択肢3","選択肢4"],"a":正解番号(0始まり),"e":"短い解説","src":"本文からの根拠抜粋"}]

# 問題数
本文量に応じて適切に（目安: 200〜300字あたり1問、重要論点を網羅）。質を量より優先する。

【本文】
"""${text}"""
```

### 改善の要点（テスト結果との対応）
| 改善 | 対応する弱点／類型 | 期待効果 |
|---|---|---|
| ルール1（本文外参照の**明示禁止＋無理に埋めない**） | C 幻覚 | 根拠なき出題を 0 に固定。今回 0 件だったのを「偶然」から「設計」に格上げ |
| ルール2-3（**一意性＋相互排他**を独立条項に） | A 一意性崩れ / B 誤肢正解化 | 最頻予測パターンを生成側で先回り抑止 |
| ルール4（数値取り違え厳禁） | B（数値起点 major） | 数値想起型に内在する唯一の major 経路を塞ぐ |
| **ルール5（`src` 根拠スニペット必須）** | C ＋ §4.4 本文引用 UI ＋ §4 自己検証 | **本実装の自己検証・本文根拠タップ表示の前提データを生成段階で確保**（後付けで原文 grep するより堅牢・安価） |

> 実装注意（spec §8 / Step 4 整合）:
> - `src` は**背骨9フィールドの契約外**。`quiz[]` の各要素に持たせるなら **in-memory / 別レイヤ保持**とし、`toDeckJSON`（L450-458）が emit する正準フィールドに混ぜない（`icon`/`desc` と同じ扱い。UIUX_SPEC §8-1, L378, L286 準拠）。
> - `parseQuizLoose`/`normalizeQuiz`（L417-433, L473）は **`src` を保持しても drop ロジックを壊さない**よう、未知フィールドを透過させる小改修で対応（`o`/`a` 正規化はそのまま）。

## 4. Step 4 自己検証パス（callLLM バッチ1回）判定プロンプト案

### 設計方針（CLAUDE.md / UIUX_SPEC §8-3 厳守）
- **`callLLM` 経由・バッチ1回**に集約（生成1件ごとの API 直叩きは禁止＝サブスク二重課金回避。spec L380, L417）。
- 入力は **`genQuiz` の出力（q/o/a/src）＋本文**。出力は各問の YES/NO 判定。
- **NG（一意でない or 誤肢が誤りと言い切れない or 根拠なし）は採用候補から drop**。判定理由を残し、レビュー画面（§4.4）で人にも提示。
- 判定もまた「**本文だけ**」で行う（評価者の外部知識を禁止＝二重の hallucination ゲート）。

### 判定プロンプト案（そのまま `verifyQuiz(items, source)` に投入可）
```
あなたはクイズの校閲者です。【本文】だけを唯一の根拠として、各設問を検査します。
あなた自身の知識・一般常識は使わないこと（本文に書いていなければ「判断不能」とみなし不合格）。

各設問について次の3点をすべて判定し、結果だけをJSON配列で返してください。
- unique  : 本文だけを根拠に、正解（aが指す選択肢）が唯一の正解として定まるか。
            本文上、他の選択肢も正しい/正解が2つ取れる場合は false。
- wrongClear: 正解以外の3つの選択肢すべてが、本文に照らして「明確に誤り」と言えるか。
            1つでも本文上正しい・部分的に正しい・本文から判断不能なものがあれば false。
- grounded : 設問・正解・解説のすべてが本文の記述に基づくか（本文外の事実を含まないか）。

判定ルール:
- unique・wrongClear・grounded のすべてが true のときだけ ok=true。1つでも false なら ok=false。
- 数値・固有名詞は本文と完全一致しているかも確認する（近い別数値が正解に化けていないか）。
- reason は20字以内で、落とした理由を簡潔に（okのときは "本文一致" 等）。

出力（JSON配列のみ。前後の説明・マークダウン禁止。入力と同じ順序・同じ件数）:
[{"i":設問の通し番号(0始まり),"ok":true/false,"unique":true/false,"wrongClear":true/false,"grounded":true/false,"reason":"理由"}]

【本文】
"""${source}"""

【設問】
${JSON.stringify(items.map((x,i)=>({i,q:x.q,o:x.o,a:x.a,e:x.e,src:x.src})), null, 0)}
```

### 実装フロー（Step 4）
```
genQuiz(text)                        // q/o/a/e/src 生成（§3 改善版）
  → normalizeQuiz()                  // 空/重複 drop, 正解再特定, a範囲外は可視警告（L320是正）
  → verifyQuiz(items, source)        // ★自己検証 callLLM バッチ1回（本節プロンプト）
  → ok=false を「除外候補」にマーク（理由つき）
  → クイズレビュー1画面（§4.4）       // 採用/除外を人が最終確認。ok=false は既定で除外側
  → 採用分のみ quizStatus="ready"
```

### コスト・UX 明示（spec L285, L417）
- 追加コストは **生成1デッキあたり API もう1回**（バッチ集約）。生成画面ローダーに **「検証に約1回 追加」** を明示（spec L285）。
- バッチ1回ゆえレイテンシ増は生成1回分のみ。問題数に比例しない。

### 設計上の注意（テスト結果由来）
- 今回 100% だったため**自己検証が全件 ok=true を返し「素通り」する可能性が高い**。それでも残す理由は §5。なお自己検証の**過剰除外（false negative）**も起こり得るため、最終判断は人（レビュー画面）に残す＝**自己検証は「自動 drop」ではなく「除外候補のサジェスト＋既定チェック」**にとどめる設計が安全（誤って良問を消さない）。

## 5. 結論

### Q. 現状の生成は本実装に耐えるか
**条件付き YES。** 実プロンプトの生成は、行政・数値中心の事実想起型コンテンツに対して
**24/24 信用可・major 0** と良好で、`normalizeQuiz` の安全側 drop と相まって「もっともらしい誤クイズ」の実害は今回サンプルで観測されなかった。
ただしこれは **(a) ドメインが AI に有利な事実照合型、(b) N=24 と小さい、(c) 一部の崩れは集計前に黙って drop され得る** という前提つきの良好さであり、**全ドメイン・本番スケールでの安全保証ではない**。

### Q. レビュー画面＋自己検証は必須か
**必須（外してはならない）。** 理由:

1. **製品の最大リスクそのものへの保険**: 「もっともらしく誤る」は低頻度・高被害型。100% の好成績は「平時は出ない」ことを示すだけで、本番回数増で必ず顕在化する低頻度バグを否定しない。ガードは**保険であり、平時に発火しないことが正常**。
2. **崩れの芽が構造的に残っている**: §2 のとおり、数値想起型は「1 箇所の取り違え＝即 major（正解と誤肢の入れ替わり）」という鋭い崖を内包する。今回はたまたま落ちなかっただけ。
3. **仕様で既に確定事項**: UIUX_REDESIGN_SPEC は Step 4「クイズ信頼ガード」（L399, §4.4）でレビュー画面＋自己検証＋根拠スニペットを**実装必須**と定義済み。本テストはこれを撤回する材料を出していない（むしろ §3 の `src` 必須化で前提データを整える方向）。
4. **コストは軽い**: 自己検証はバッチ1回（生成+約1コール）。得られる安全性に対し追加コスト・レイテンシは小さい（§4）。

### 推奨アクション（優先順）
1. **genQuiz に §3 改善版を適用**（本文外参照の明示禁止・一意性/相互排他の独立条項化・`src` 根拠スニペット必須化）。最小の文言追加で、今回 0 件を「偶然」から「設計」に固定。
2. **Step 4 で §4 自己検証パス（callLLM バッチ1回）を実装**。ok=false は**自動 drop ではなくレビュー画面の既定除外候補**として人に最終判断を残す。
3. **テストを別ドメインへ拡張**（法令解釈・条件分岐・因果を問う教材）。事実想起型に偏った今回の 100% を、苦手領域でも検証して初めて「本実装に耐える」が裏付けられる。
